Acceptance Criteria Strategy · v1 · method set + coverage map + worked samples

No single notation can verify the whole PRD well. This proposes a small set of five complementary acceptance-criteria methods, routes every PRD section to the right one, and shows one worked sample per method in the project's ubiquitous language.

Read with concept.html (the why) and prd.html (the what); this is the how we'll know it's true. It is a strategy plus samples, not the full AC catalogue — the samples fix the house style for each method so the bulk can be authored against it. There is no implementation code yet, so criteria are written as authored specs (human- and LLM-readable), shaped to bind to @stint/core, tt, and the GUI once they exist.

01 Why a set, not one method

The PRD has five kinds of claim

The requirements in prd.html are not uniform, and the temptation to pick one notation (all-Gherkin, or all-unit-tests) breaks on that variety. Sorting the PRD by what kind of thing is being asserted yields five distinct shapes, each with a notation that fits and several that don't:

  • Pure, deterministic logic — the one-open-entry invariant, derived elapsed, billable-duration arithmetic, rounding maths, split/merge, overlap detection, UTC/DST math. Truths over all inputs; best caught by properties and example tables, not prose scenarios.
  • Observable behavioural flows — start/stop/resume/backfill, "start while open stops the open one", overlap warns-not-blocks. Given/When/Then in domain language reads as both spec and test: Cucumber's home turf.
  • A machine contract — tt's exit codes, table output, and --json shape; CSV columns; the export schema. Verified by golden/snapshot comparison and schema validation, where the artefact is the criterion.
  • Subjective & emergent qualities — "feels at home on the desktop," monochrome, accent on the primary action only, empty states that instruct, flags visible in context. No assertion captures these; a rubric judged by an LLM (or human) over driven-UI screenshots can.
  • Physical / OS reality — real sleep/wake via powerMonitor, wall-clock-gap reconciliation on launch, notifications firing on a wall-clock schedule, and "makes no network connections." These resist CI; they need scripted-but-supervised checks on real hardware.

So the method set is chosen to cover those five shapes with minimal overlap — each method earns its place by being the best (often the only) fit for one shape, while a couple double as secondary nets for others.

02 The five methods

The toolkit, and what each is for

CodeMethodBest forWeak for / not
BDD Gherkin / Cucumber scenarios. Given/When/Then in ubiquitous language, stepped onto @stint/core and tt (cucumber-js, or plain vitest "scenario" helpers). User-observable flows and the §16 edge-case table; cross-surface behaviour; the living, readable spec a non-coder can check. Exhaustive input spaces (combinatorial blow-up), exact byte-for-byte output, anything visual or timing-physical.
PROP Property-based tests. Generated input sequences against invariants (fast-check, run under vitest), plus stress/concurrency variants. Invariants that must hold over all inputs: one-open-entry under any start/stop ordering, stored-truth immutability, rounding/elapsed algebra, split-then-merge round-trips. Pinning a specific expected value; explaining intent to a stakeholder; UI and OS effects.
GOLD Golden / snapshot + schema. Approval tests (vitest snapshots) over tt stdout/stderr and exit codes; JSON-Schema validation of --json; fixed-fixture CSV diffs. The CLI contract and export formats — the parts where the exact text, columns, and structure are the requirement. Behaviour and intent (a snapshot says what, never why); brittle if used for logic that example tables should own.
JUDGE LLM-judged + agentic UI exercising. A driver (Playwright driving the Electron app via Chromium) walks the renderer windows and captures screenshots + the accessibility tree; an LLM scores them against a written rubric. Deterministic GUI facts get plain Playwright assertions, and main-process state that lives outside the DOM — globalShortcut registration, tray-triggered actions — is exercised via electronApp.evaluate(). The renderer runs headless under xvfb-run; the tray icon itself has no host in headless CI, so its on-screen rendering goes to MANUAL. Subjective and emergent GUI qualities of the main window and popover, empty-state copy, "flag visible on the affected entry," look-and-feel — judged, not asserted. Anything requiring exactness or a hard pass/fail boundary; non-deterministic, so gated by a rubric + human spot-check, never the sole gate on money-affecting logic.
MANUAL Scripted manual / supervised checks. A numbered runbook on real hardware, using the OS's sleep controls (systemctl suspend / rtcwake, or the platform equivalent), simulated sleep, and a network monitor; an operator (human or agent) confirms each step and records evidence. Physical/OS reality: real sleep/wake + wall-clock-gap reconcile, notification wall-clock cadence across relaunch, the no-network guarantee, and the tray's on-screen count-up and global-hotkey toggle (no system-tray host under headless CI). Regression speed and scale (slow, can't run on every commit); reserved for what genuinely can't be faked in CI.
Cucumber, specificallyGherkin is adopted for the behavioural layer only — it is not the whole strategy. Using it where input spaces explode (rounding, time parsing) or where output must match byte-for-byte (CLI/JSON) would be a misuse; those go to PROP and GOLD. One feature can drive either @stint/core directly or the tt executable, which is how the "full parity" claim (§17 R8) gets tested without writing every scenario twice.
03 How to route a requirement

A decision rule for any new criterion

So the set stays disciplined as the catalogue grows, route each requirement by the first rule that matches:

  1. Does it touch money-affecting maths or a stored invariant (durations, rounding, exclusion, one-open-entry, immutability)? → PROP for the law, GOLD/example table for specific values. Never let JUDGE be the only gate here.
  2. Is the deliverable exact text or a data shape (tt output, exit code, --json, CSV columns)? → GOLD + schema.
  3. Is it a user-observable flow or decided edge case a stakeholder should be able to read? → BDD, stepped on @stint/core or tt.
  4. Is it visual, subjective, or emergent in the GUI? → JUDGE over agentic exercising, with plain Playwright for any deterministic sub-fact in the renderer, and electronApp.evaluate() for main-process facts (e.g. the global shortcut is registered and its handler toggles the open entry).
  5. Does it depend on real OS/hardware behaviour over wall-clock time? → MANUAL runbook, with as much scripted setup as possible.

Two requirements often split across methods: §17 R5 (sleep produces a flagged entry with second-accurate spans and a working subtract) becomes PROP for the excluded_seconds arithmetic, BDD for the subtract flow, JUDGE for the flag being visible in context, and MANUAL for the spans being real after an actual sleep. That decomposition is the point of having a set.

06 Sample — Gherkin / BDD

BDD behavioural flows in ubiquitous language

One .feature can be stepped onto @stint/core directly or onto the tt executable. Note the deliberate vocabulary: open entry, start, stop, overlap, warned-not-blocked — all straight from the glossary.

Feature: One open entry across surfaces
  # PRD §05 R1, §16 "start while running", §17 R2 — the keystone invariant,
  # observed as behaviour. Steps bind to @stint/core; @cli re-runs them through tt.

  Background:
    Given an empty database
    And a client "Client A" with project "API"

  Scenario: Starting while an entry is open closes the first one
    Given I start an entry "auth refactor" for "Client A" / "API" at 09:00
    When  I start an entry "code review" at 10:30
    Then  exactly one entry is open
    And   the entry "auth refactor" is closed with end 10:30
    And   the open entry is "code review"

  @cli
  Scenario: A timer started in tt is visible to status with no disagreement
    When  I run `tt start "auth refactor" --client "Client A" --project API`
    Then  the exit code is 0
    And   `tt status` reports an open entry "auth refactor" for "Client A / API"

  Scenario: Backfill that overlaps an existing entry is warned, not blocked
    # PRD §06 R4 — overlap is allowed if meant, and flagged downstream
    Given a closed entry from 09:00 to 11:00
    When  I backfill an entry from 10:00 to 10:30
    Then  the backfill succeeds
    And   a non-blocking overlap warning is surfaced
    And   both entries are flagged "overlapped" in a report covering the day

  Scenario: Resume copies attributes but is a new entry
    # PRD §05 R4 + glossary: resume is never a re-opening of the old row
    Given a closed entry "auth refactor" for "Client A" / "API", billable
    When  I resume
    Then  a new open entry exists with description "auth refactor", "Client A", "API", billable
    And   it has a different id from the original

The same Background + Scenario text, tagged @cli, is run a second time through the tt executable — directly exercising the §17 R8 parity claim without a second copy of the spec.

07 Sample — Property-based

PROP invariants over generated inputs

Where Gherkin pins one path, properties assert a law over thousands of generated ones. These are the criteria guarding the numbers that reach an invoice.

// PRD §03 invariant + §17 R2 — at most one open entry, under ANY interleaving
test.prop([fc.array(timerOp())])("at most one open entry after any op sequence", (ops) => {
    const core = openCore({ inMemory: true });
    ops.forEach((op) => core.apply(op));       // start/stop/resume/backfill, any order
    expect(core.openEntries().length).toBeLessThanOrEqual(1);
});

// PRD §03 + §17 R4 — stored truth is immutable under rounding & subtract
test.prop([entryArb(), roundIncArb(), secondsArb()])("rounding and subtract never mutate start/end", (e, inc, sub) => {
    const before = [e.startUtc, e.endUtc];
    report(e, { rounding: inc });              // display lens
    subtractSlept(e, sub);                     // excluded_seconds only
    expect([e.startUtc, e.endUtc]).toEqual(before);
});

// PRD §03 — billable duration is exactly raw minus exclusion, never negative
test.prop([entryArb()])("billable = (end ?? now) - start - excluded, clamped ≥ 0", (e) => {
    const raw = (e.endUtc ?? now()) - e.startUtc;
    expect(e.billableDuration).toBe(Math.max(0, raw - e.excludedSeconds));
});

// PRD §06 — split then merge is identity on the covered span (round-trip)
test.prop([closedEntryArb(), instantArb()])("split @t then merge restores the original span", (e, t) => {
    fc.pre(e.contains(t));                     // vacuously, t must be inside
    const [a, b] = core.split(e, t);
    const m = core.merge([a, b]);
    expect([m.startUtc, m.endUtc]).toEqual([e.startUtc, e.endUtc]);
});

// PRD §04/§16 — duration is DST-safe because it's UTC math
test.prop([entryArb(), timeZoneArb()])("duration is invariant across the wall-clock zone it's rendered in", (e, tz) => {
    expect(renderedIn(tz, e).rawDuration).toBe(e.rawDuration);
});

The rapid-start/stop concurrency of §17 R2 also gets a stress variant: many concurrent processes issue start/stop against one WAL database with the busy timeout set, asserting the open-entry count stays ≤ 1 throughout — the property above, but proven across processes rather than in one.

For tt output, the CSV, and --json, the artefact is the criterion. Golden files capture human output; a JSON Schema validates structure so scripts can rely on it; exit codes are asserted directly.

09 Sample — LLM-judged + agentic

JUDGE the GUI qualities that resist assertion

An agent drives the GUI through a scripted route (Playwright over the Electron app via Chromium), captures screenshots and the accessibility tree at each step, and an LLM scores the evidence against an explicit rubric. Deterministic facts in the renderer (a Stop button exists and is hittable) are plain Playwright assertions; facts that live in the main process rather than the DOM (the global shortcut is registered, its toggle handler ran) are exercised via electronApp.evaluate(). The renderer windows run headless under xvfb-run; the tray icon's own title text has no host there, so its live count-up is confirmed under MANUAL on a real desktop session.

Agentic route driver script

  1. Launch with an empty database. Capture the main window. → empty state
  2. Assert the global shortcut is registered, then invoke the timer-toggle it is bound to via electronApp.evaluate(). Open the tray popover window and capture it. → running state, count-up
  3. Wait 3s; re-capture the popover. → count-up advanced
  4. Backfill an overlapping entry via tt; refocus the window. Capture. → overlap flag in context
  5. Simulate a sleep span on the open entry; capture the affected entry row. → slept-through flag + subtract affordance

Judge rubric pass / fail per item

# Each item scored PASS/FAIL with a one-line justification citing the screenshot.
# Any FAIL fails the suite; a sample of PASSes is spot-checked by a human.

EMPTY_STATE     The empty main window instructs rather than decorates — it names a
                concrete next action ("press Ctrl+Alt+T" or "run tt start"). (PRD §12 R5)

TRAY_COUNTUP    The tray popover shows a single running timer counting up; between
                the two captures the displayed elapsed increased by ~3s, not reset.
                (The tray icon's own title count-up is checked under MANUAL.) (§12 R1)

ACCENT_DISCIPLINE  The system accent colour appears on the running state / primary
                action ONLY; the rest of the chrome is monochrome system grays. (§07, §15)

FLAG_IN_CONTEXT The overlapped entry and the slept-through entry each show their flag
                ON the affected row — not in a separate list — with the one-tap
                subtract present on the slept entry. (§12 R4, §10 R5)

DESKTOP_FEEL    The window reads as a quiet, minimal desktop app that respects the
                system (system typography, light/dark following the OS, monochrome
                with restrained accent) — not a busy or branded web page. (§15)
GuardrailJUDGE is non-deterministic, so it is never the sole gate on anything that changes a billable number. It owns presentation and discoverability; the arithmetic underneath those screens is already nailed by PROP and GOLD. Rubric items are written as crisp PASS/FAIL claims, not vibes, to keep the judgement repeatable.
10 Sample — Manual / supervised checklist

MANUAL the physical & OS-level guarantees

Real sleep/wake, notification cadence over wall-clock time, and the no-network promise can't be faked in CI. Setup is scripted as far as possible; an operator (human or agent with shell + GUI access) confirms each step and attaches evidence.

CHECK SLEEP-SPAN — second-accurate spans + working subtract   (§10a, §17 R5)
  1. tt start "deep work" --client "Client A"
  2. Record start. Sleep the machine:  systemctl suspend  (or the platform's sleep)
  3. Wake after ~120s. Note the suspend/resume times from the system log.
  ✓ The open entry is flagged slept-through.
  ✓ A SleepSpan (source=event) is attached; (resume − suspend) matches the
    powerMonitor delta within 1s.
  ✓ `tt sleep subtract <id>` moves those seconds into excluded_seconds;
    billable duration drops by exactly the span; raw duration is unchanged.
  ✓ Subtract is reversible — re-running restores the prior excluded_seconds.

CHECK MISSED-SLEEP RECONCILE — wall-clock gap on launch        (§10a, source=gap)
  1. With a timer open, fully quit the app, sleep the machine ~120s, wake, relaunch.
  ✓ On launch a SleepSpan (source=gap) is created from the wall-clock gap and the
    entry is flagged slept-through for review; the gap bounds the dead time.
  ✓ Because the gap can't tell true sleep from app-closed time, the span is
    a flagged suspicion only — never auto-subtracted (the operator decides).

CHECK CHECK-IN CADENCE + RELAUNCH                            (§10b, §17 R6)
  1. Start a timer. (Use a test cadence to avoid waiting 60m, but verify the
     real 60-then-30 defaults once in a long-form run.)
  ✓ First check-in fires at start + first-interval; then every interval.
  ✓ Ignoring one still fires the next on time (autonomous).
  ✓ Quit the app across a due check-in, relaunch:
    it fires ONCE on relaunch, then resumes cadence (no backlog).
  ✓ After a long sleep, the next check-in realigns from wake — no flood.

CHECK NO NETWORK                                             (§17 R9)
  1. Run the app + tt through a full session under a network monitor
     (lsof -i / ss / a packet monitor), exercising every feature.
  ✓ Zero outbound connections from the app or tt for the whole session.
  ✓ The app opens no listening or outbound sockets; no telemetry,
    update-check, or analytics code path exists.

The no-network check has a cheap GOLD backstop that does run in CI — scanning the build's dependency tree and source for any networking import or outbound-request code path (none permitted) — so a regression is caught fast even though the live-traffic confirmation stays manual.

Residual risk we accept

  • Sleep reconciliation is wall-clock-approximate. A sleep missed while the app was closed is reconstructed from the wall-clock gap (source=gap), which bounds the dead time but can't tell true sleep from app-closed time — so gap-sourced spans are flagged for review and never auto-applied. Verified on real hardware before release, not per commit.
  • Tray and global hotkey aren't exercised end-to-end in CI. Headless xvfb-run has no system-tray host and can't deliver an OS-global hotkey, so JUDGE covers the renderer windows and main-process state (electronApp.evaluate()) while the tray icon's live count-up and a real hotkey press are confirmed under MANUAL on a real desktop session.
  • Long-form cadence is sampled, not exhaustive. The real 60-then-30 schedule is verified in a long-running pass; day-to-day regression uses a compressed test cadence, so a defect that only appears at the true intervals could slip between full runs.
  • "Near-instantly" is a soft bound. File-watch propagation (§17 R1) is checked against a generous latency budget on the test machine; it is observed, not guaranteed under arbitrary system load.
  • JUDGE is advisory on look-and-feel. "Feels like a quiet desktop app" is scored by rubric and spot-checked by a human; it gates presentation but never a billable number.

Net: the money-affecting core (invariants, durations, rounding, exclusion, exports) reaches near-complete deterministic coverage; the irreducibly physical and the irreducibly subjective are covered by the right weaker instrument and named here as known residue rather than papered over.
