01 Overview & goals

What Stint is

Stint is a single-user cross-platform desktop application for tracking billable time, in the spirit of early Toggl: a live start/stop timer, with manual backfill, organized by client. It runs locally, stores all data in one SQLite file, and is fully offline. Its output is trustworthy time totals per client and date range, exported as CSV or JSON, which the user turns into invoices in a separate tool.

Primary goals

  • Accurate billable hours. The stored total is always the exact truth; nothing is silently rounded, added, or removed.
  • Equal CLI and GUI. Anything the tray app can do, tt can do, over the same database, with no privileged surface.
  • Low-friction tracking. Starting a timer takes one keystroke or one click; forgetting to start one is always recoverable by hand.
  • Quiet, minimal, cross-platform. A monochrome desktop app that respects system light/dark and stays out of the way.

Primary user

A freelancer, consultant, or contractor working solo at their own computer, tracking a handful of clients, comfortable in a terminal, who invoices from the exported numbers using other software.

03 Concepts & domain model

Entities and the core invariant

The conceptual keystone: a running timer is the single time entry whose end is null. "Running" is a row state, not a separate object. Elapsed time is always derived (now − start), never stored or incremented.

EntityKey fieldsNotes
Clientid, name, archivedTop of the hierarchy. No rate. Archivable, not deleted, once referenced.
Projectid, client_id, name, archivedOptional refinement. A project belongs to exactly one client; selecting a project implies its client.
Entryid, client_id?, project_id?, description?, start, end?, billable, excluded_secondsThe core record. end null ⇒ running. Times stored UTC. client_id nullable (clientless = internal time).
Tagid, name, archivedFlat, reusable, cross-cutting. Many-to-many with entries via a join.
SleepSpanid, entry_id, sleep_at, wake_at, sourceA sleep→wake cycle that occurred during a running entry. Multiple per entry. source = event | gap | unknown.
Settingkey, valueKey-value app config (rounding, week start, check-in cadence, etc.).

Invariants

  • At most one open entry (end IS NULL) exists at any time. Enforced transactionally in the shared core.
  • Billable duration of an entry = (end ?? now) − start − excluded_seconds. Stored truth (start, end) is never mutated by rounding or sleep subtraction.
  • A project's client is authoritative: an entry with a project takes that project's client.
04 Architecture

One core, one file, two thin shells

  1. Shared core. A TypeScript package (@stint/core) owns the schema, every state transition, and all invariants. The tray app and the tt CLI both import it; there is no duplicated logic and no privileged surface. implemented
  2. Single source of truth. One SQLite file in WAL mode (concurrent readers, single writer). All reads and writes go through @stint/core. implemented
  3. Derived elapsed. No process holds or increments a timer. Elapsed is computed now − start on demand; the tray's per-second update is display only. implemented
  4. Atomic transitions. Starting a timer is one transaction (BEGIN IMMEDIATE): close any open entry (end = now), insert a new open entry. This makes "start while running" safe regardless of which surface triggers it. The CLI uses a busy timeout so it cooperates with the running app. implemented
  5. Change propagation by file-watch. Each surface watches the SQLite file (and its -wal) via fs.watch and refreshes on change, so a write from one surfaces in the other near-instantly. (The tray's 1-second display tick is independent of this.) implemented
  6. UTC storage, local display. All timestamps stored UTC; rendered in the system locale and time zone. Durations are timezone-independent and DST-safe. implemented
05 Timer & entries

Tracking and backfill

  1. Start opens a new entry; any currently open entry is stopped first. Description, client, project, tags, and billable are all optional at start and editable later. Nothing is required to start a bare timer. partial core + tt start accept all attributes at start; the GUI Start control is parameterless — setting description/client/project/tags at start is TODO (§12 R5).
  2. Stop sets end = now on the open entry. implemented
  3. Status reports the open entry and its derived elapsed, or that nothing is running. implemented
  4. Resume starts a fresh entry copying the most recent entry's description, client, project, and billable flag. implemented
  5. Manual add creates a completed entry from explicit from/to times (backfill). Backfilled entries are first-class and indistinguishable from tracked ones except by their timestamps. partial core + tt add complete; the GUI "Add Entry Manually" form is TODO (§12 R7).
  6. The running entry is editable like any other (description, client, project, tags, even start); its end is set only by stopping it. partial core + tt edit complete; the GUI edit affordance is TODO (§12 R6).
  7. Default billable is true when a client is set, false when clientless; always overridable. implemented
  8. Switch new — a single action that stops the open entry and immediately starts a new one, optionally carrying forward or clearing client/project. Equivalent to stop-then-start as one atomic transition; surfaced as a dedicated control so the common "moved to a different task" case is one step. todo (GUI affordance §12 R5; tt's start already stops the open entry, so the CLI path exists — a tt switch alias is optional).
06 Editing

Split, merge, delete, overlap

  1. Edit any field of any entry. Delete any entry (CLI requires confirmation or --force). partial core + tt edit/tt rm complete; the GUI offers Delete only — inline field editing is TODO (§12 R6).
  2. Split cuts one entry into two at a chosen instant within its span; both inherit the original's client/project/tags/billable; the user adjusts from there. partial core + tt split complete; the GUI split affordance is TODO (§12 R6).
  3. Merge combines a contiguous selection into one entry spanning the earliest start to the latest end. Conflict rule: when merged entries disagree on client/project/billable, the GUI prompts which to keep; descriptions are concatenated and tags unioned. tt merge uses the first entry's attributes unless --client/--project override. partial core + tt merge complete; the GUI multi-select + conflict prompt is TODO (§12 R6).
  4. Overlap is warned, not blocked. Any add/edit that overlaps an existing entry surfaces a clear warning (GUI banner; tt note on stderr) but is allowed. Overlapped spans are flagged in reports so the same time cannot silently bill twice. partial detection + tt stderr note + report flagging complete; the inline GUI overlap banner is TODO (§12 R9).
07 Clients, projects, tags

Reference data

  1. Clients and projects can be created, renamed, archived, and listed from both GUI and CLI. A project must name a client. Referenced records are archived rather than hard-deleted to preserve history. partial core + tt client/tt project complete; a GUI management view (the Clients nav item) is TODO (§12 R10).
  2. Tags are flat, reusable labels, applied many-per-entry, used for filtering and grouping (e.g. all "meetings" across clients). Created on the fly when first applied. partial core + CLI complete; GUI tag display (chips on entries), editing, and management are TODO (§12 R6, R10).
08 Billable & non-billable

Tracking overhead without billing it

  1. An entry may have no client; clientless time defaults to non-billable internal time (admin, email, own tooling). implemented
  2. The billable flag overrides either way: a client entry can be marked non-billable (goodwill), and internal time can rarely be flagged billable. implemented
  3. Reports default to billable-only, with a toggle to show all or non-billable, so the user can see total worked vs. invoiceable. partial core + tt report --all|--non-billable complete; the GUI report builder (with this toggle) is TODO (§12 R8).

10 Running-timer hygiene

Two independent mechanisms

Sleep-span flagging suspend + resume

  1. While an entry is open, the app listens for the OS suspend and resume events (Electron powerMonitor) and timestamps each on receipt (source = event). implemented
  2. Each suspend→resume cycle during the open entry is recorded as a SleepSpan and attached to the entry, which is flagged "slept through." Multiple cycles per timer are all captured. implemented
  3. On launch, the app reconciles by comparing the wall-clock gap since its last recorded timestamp against the open entry: a gap large enough to imply the machine slept is recorded as a SleepSpan (source = gap), recovering a cycle missed while the app was not running. The wall-clock comparison captures that a sleep occurred and its approximate bounds. implemented
  4. The timer keeps running through sleep (now − start stays honest as wall-clock); the flag and spans let the user decide about the dead time. implemented
  5. Slept time handling: flagged, not auto-subtracted. A one-tap action (GUI) / flag (tt) adds the span(s) to excluded_seconds, trimming billable duration precisely and reversibly. implemented
  6. powerMonitor reports user-visible suspend/resume, so each sleep is recorded as one span. A sleep that occurs while the app is closed is recovered by the wall-clock-gap reconciliation in item 3 (see Appendix A). implemented

Check-ins notification

  1. The first check-in notification fires at start + 60 min, then every 30 min by default. implemented
  2. The schedule is autonomous: if the user ignores a notification, the next still fires on time. implemented
  3. The notification shows light context — the description if present (and client if set), otherwise "your timer." implemented
  4. It offers an inline Stop and a keep going action, plus a dropdown setting minutes until the next check-in (default 30). A custom pick applies to the next gap only, then reverts to the default. partial the notification with Stop and keep going actions fires; the inline per-notification interval dropdown is TODO.
  5. Cadence applies to every running timer, billable or not. implemented
  6. The schedule is persisted/derivable so it survives relaunch: a check-in that came due while the app was closed fires once on relaunch, then resumes cadence. After a long sleep, cadence realigns from wake — no backlog of missed notifications. implemented
Out of scopeOS-level user-idle detection (keyboard/mouse inactivity). It is fuzzy, OS-version-dependent, and a cross-application concern. Stint builds only on sleep/wake, which are clean and authoritative. See Appendix A.

13 Storage & schema

One file, overridable path

Default location: the OS's standard per-user app-data directory — the GUI resolves it via Electron's app.getPath('userData'), and tt resolves the documented per-OS equivalent (e.g. $XDG_DATA_HOME/~/.local/share on Linux, ~/Library/Application Support on macOS, %APPDATA% on Windows) — holding timetracker.sqlite (WAL mode). Both the app and tt resolve the same path, overridable by the TT_DB environment variable. Backup = copy the file; export = tt export. No network, ever.

-- schema sketch (illustrative)
CREATE TABLE client  (id INTEGER PK, name TEXT, archived INT DEFAULT 0);
CREATE TABLE project (id INTEGER PK, client_id INT REFERENCES client, name TEXT, archived INT DEFAULT 0);
CREATE TABLE entry (
  id INTEGER PK,
  client_id  INT REFERENCES client,        -- nullable
  project_id INT REFERENCES project,        -- nullable; implies client
  description TEXT,
  start_utc  TEXT NOT NULL,                 -- ISO-8601 UTC
  end_utc    TEXT,                          -- NULL = running
  billable   INT NOT NULL DEFAULT 1,
  excluded_seconds INT NOT NULL DEFAULT 0   -- subtracted sleep
);
CREATE TABLE tag (id INTEGER PK, name TEXT, archived INT DEFAULT 0);
CREATE TABLE entry_tag (entry_id INT, tag_id INT, PRIMARY KEY(entry_id, tag_id));
CREATE TABLE sleep_span (id INTEGER PK, entry_id INT REFERENCES entry,
  sleep_utc TEXT, wake_utc TEXT, source TEXT);
CREATE TABLE setting (key TEXT PRIMARY KEY, value TEXT);
CREATE TABLE app_state (key TEXT PRIMARY KEY, value TEXT);  -- heartbeat / last-seen & check-in schedule (gap reconciliation, §10)
-- invariant: at most one row with end_utc IS NULL, enforced in @stint/core transactions

15 Platform & tech

Cross-platform, shared-core

  • GUI: Electron — a Tray with a popover BrowserWindow for the running timer, and a standard BrowserWindow for lists and reports. The renderer is web tech (HTML/CSS, framework optional). Follows system light/dark; uses system typography; system accent applied only to the primary action and running state.
  • CLI: a Node executable, tt, built on commander with --json everywhere, importing the same @stint/core package.
  • Core: a TypeScript package, @stint/core, owning schema, transitions, invariants, reporting, and rounding. Both surfaces import it; no duplicated logic.
  • Persistence: SQLite via Node's built-in node:sqlite (Node 22.5+), WAL mode set by pragma, BEGIN IMMEDIATE writes, busy timeout for cross-process cooperation. No native build step. node:sqlite only exists on Node 22.5+ and runs in two distinct runtimes — the system Node behind tt and Electron's own bundled Node — so v1 requires system Node ≥ 22.5 and an Electron whose bundled Node is ≥ 22.5 with node:sqlite loadable in the packaged main process (it is a stability-experimental module; confirm against the pinned versions).
  • System integration: fs.watch for file-watch; Electron powerMonitor suspend/resume events (with wall-clock-gap reconciliation on launch) for sleep flagging, and powerMonitor.getSystemIdleTime() available for idle; Electron globalShortcut for the hotkey; Electron Notification for check-ins.

17 Acceptance criteria

  1. A timer started in tt appears in the tray near-instantly, and vice versa; tt status and the tray never disagree. implemented
  2. At most one entry is ever open, regardless of how rapidly start/stop is issued from either surface. implemented

  9. The app makes no network connections. implemented

A Appendix — Power/idle APIs

What the process can actually observe

Reference for §10, expressed against Electron's powerMonitor. Sleep/wake are event-backed (live suspend/resume), with a wall-clock-gap fallback on launch; idle is only pollable and fuzzy — which is why §10 builds on the former and excludes the latter. powerMonitor surfaces user-visible suspend/resume, so each sleep is one span and a missed sleep is reconstructed from the wall-clock gap.

SignalAvailable?How
Went to sleepYes, live eventpowerMonitor 'suspend' (stamp on receipt, source = event). If missed (app not running), inferred after the fact from a wall-clock gap, source = gap.
Woke from sleepYes, live eventpowerMonitor 'resume'. A wake missed while the app was closed is recovered on launch by comparing the wall-clock gap against the open entry.
Became idlePoll only, fuzzyNo event. Poll powerMonitor.getSystemIdleTime() (seconds since last input); infer start as now − idle. The OS "idle" definition is OS-provided and version-variable. Excluded from v1 per §10.
Stopped being idleInferredIdle time resets to 0 on input; powerMonitor 'unlock-screen' as a coarse proxy.
