01 Overview & goals

What Stint is

Stint is a single-user cross-platform desktop application for tracking billable time, in the spirit of early Toggl: a live start/stop timer, with manual backfill, organized by client. It runs locally, stores all data in one SQLite file, and is fully offline. Its output is trustworthy time totals per client and date range, exported as CSV or JSON, which the user turns into invoices in a separate tool.

Primary goals

  • Accurate billable hours. The stored total is always the exact truth; nothing is silently rounded, added, or removed.
  • Equal CLI and GUI. Anything the tray app can do, tt can do, over the same database, with no privileged surface.
  • Low-friction tracking. Starting a timer takes one keystroke or one click; forgetting to start one is always recoverable by hand.
  • Quiet, minimal, cross-platform. A monochrome desktop app that respects system light/dark and stays out of the way.

Primary user

A freelancer, consultant, or contractor working solo at their own computer, tracking a handful of clients, comfortable in a terminal, who invoices from the exported numbers using other software.

03 Concepts & domain model

Entities and the core invariant

The conceptual keystone: a running timer is the single time entry whose end is null. "Running" is a row state, not a separate object. Elapsed time is always derived (now − start), never stored or incremented.

EntityKey fieldsNotes
Clientid, name, archivedTop of the hierarchy. No rate. Archivable, not deleted, once referenced.
Projectid, client_id, name, archivedOptional refinement. A project belongs to exactly one client; selecting a project implies its client.
Entryid, client_id?, project_id?, description?, start, end?, billable, excluded_secondsThe core record. end null ⇒ running. Times stored UTC. client_id nullable (clientless = internal time).
Tagid, name, archivedFlat, reusable, cross-cutting. Many-to-many with entries via a join.
SleepSpanid, entry_id, sleep_at, wake_at, sourceA sleep→wake cycle that occurred during a running entry. Multiple per entry. source = event | gap | unknown.
Settingkey, valueKey-value app config (rounding, week start, check-in cadence, etc.).

Invariants

  • At most one open entry (end IS NULL) exists at any time. Enforced transactionally in the shared core.
  • Billable duration of an entry = (end ?? now) − start − excluded_seconds. Stored truth (start, end) is never mutated by rounding or sleep subtraction.
  • A project's client is authoritative: an entry with a project takes that project's client.
04 Architecture

One core, one file, two thin shells

  1. Shared core. A TypeScript package (@stint/core) owns the schema, every state transition, and all invariants. The tray app and the tt CLI both import it; there is no duplicated logic and no privileged surface.
  2. Single source of truth. One SQLite file in WAL mode (concurrent readers, single writer). All reads and writes go through @stint/core.
  3. Derived elapsed. No process holds or increments a timer. Elapsed is computed now − start on demand; the tray's per-second update is display only.
  4. Atomic transitions. Starting a timer is one transaction (BEGIN IMMEDIATE): close any open entry (end = now), insert a new open entry. This makes "start while running" safe regardless of which client triggers it. The CLI uses a busy timeout so it cooperates with the running app.
  5. Change propagation by file-watch. Each surface watches the SQLite file (and its -wal) via fs.watch and refreshes on change, so a write from one surfaces in the other near-instantly. (The tray's 1-second display tick is independent of this.)
  6. UTC storage, local display. All timestamps stored UTC; rendered in the system locale and time zone. Durations are timezone-independent and DST-safe.
05 Timer & entries

Tracking and backfill

  1. Start opens a new entry; any currently open entry is stopped first. Description, client, project, tags, and billable are all optional at start and editable later. Nothing is required to start a bare timer.
  2. Stop sets end = now on the open entry.
  3. Status reports the open entry and its derived elapsed, or that nothing is running.
  4. Resume starts a fresh entry copying the most recent entry's description, client, project, and billable flag.
  5. Manual add creates a completed entry from explicit from/to times (backfill). Backfilled entries are first-class and indistinguishable from tracked ones except by their timestamps.
  6. The running entry is editable like any other (description, client, project, tags, even start); its end is set only by stopping it.
  7. Default billable is true when a client is set, false when clientless; always overridable.
06 Editing

Split, merge, delete, overlap

  1. Edit any field of any entry. Delete any entry (CLI requires confirmation or --force).
  2. Split cuts one entry into two at a chosen instant within its span; both inherit the original's client/project/tags/billable; the user adjusts from there.
  3. Merge combines a contiguous selection into one entry spanning the earliest start to the latest end. Conflict rule: when merged entries disagree on client/project/billable, the GUI prompts which to keep; descriptions are concatenated and tags unioned. tt merge uses the first entry's attributes unless --client/--project override.
  4. Overlap is warned, not blocked. Any add/edit that overlaps an existing entry surfaces a clear warning (GUI banner; tt note on stderr) but is allowed. Overlapped spans are flagged in reports so the same time cannot silently bill twice.
07 Clients, projects, tags

Reference data

  1. Clients and projects can be created, renamed, archived, and listed from both GUI and CLI. A project must name a client. Referenced records are archived rather than hard-deleted to preserve history.
  2. Tags are flat, reusable labels, applied many-per-entry, used for filtering and grouping (e.g. all "meetings" across clients). Created on the fly when first applied.
08 Billable & non-billable

Tracking overhead without billing it

  1. An entry may have no client; clientless time defaults to non-billable internal time (admin, email, own tooling).
  2. The billable flag overrides either way: a client entry can be marked non-billable (goodwill), and internal time can rarely be flagged billable.
  3. Reports default to billable-only, with a toggle to show all or non-billable, so the user can see total worked vs. invoiceable.
09 Reports & export

The deliverable

  1. Date range: presets (today, this week, last week, this month, last month) plus arbitrary custom ranges.
  2. Grouping: default client → project → entries; also by day or by tag.
  3. Filters: client, project, tag, billable.
  4. Rounding: stored time is exact; rounding applies only at display/export, off by default. When on, grouped totals (the billable line, not each entry) round to a configurable increment — default nearest 15 min; 6 / 10 / 30 also available; nearest, not always-up.
  5. Flags surfaced: overlapped time and unreviewed sleep-flagged entries are marked in every report so they can't quietly reach an invoice.
  6. Export: CSV (one row per entry: client, project, tags, description, start, end, raw duration, excluded sleep, billable, overlapped) and JSON. The GUI also shows an on-screen summary. No PDF, no Markdown.
10 Running-timer hygiene

Two independent mechanisms

Sleep-span flagging suspend + resume

  1. While an entry is open, the app listens for the OS suspend and resume events (Electron powerMonitor) and timestamps each on receipt (source = event).
  2. Each suspend→resume cycle during the open entry is recorded as a SleepSpan and attached to the entry, which is flagged "slept through." Multiple cycles per timer are all captured.
  3. On launch, the app reconciles by comparing the wall-clock gap since its last recorded timestamp against the open entry: a gap large enough to imply the machine slept is recorded as a SleepSpan (source = gap), recovering a cycle missed while the app was not running. The wall-clock comparison captures that a sleep occurred and its approximate bounds.
  4. The timer keeps running through sleep (now − start stays honest as wall-clock); the flag and spans let the user decide about the dead time.
  5. Slept time handling: flagged, not auto-subtracted. A one-tap action (GUI) / flag (tt) adds the span(s) to excluded_seconds, trimming billable duration precisely and reversibly.
  6. Translation note: powerMonitor reports user-visible suspend/resume only, so one sleep is recorded as one span. The macOS-specific nuances of the original spec — kernel-timestamp reconciliation and Power Nap / dark-wake filtering — have no cross-platform analog and are deliberately dropped; v1 commits to wall-clock-gap reconciliation instead (see Appendix A).

Check-ins notification

  1. The first check-in notification fires at start + 60 min, then every 30 min by default.
  2. The schedule is autonomous: if the user ignores a notification, the next still fires on time.
  3. The notification shows light context — the description if present (and client if set), otherwise "your timer."
  4. It offers an inline Stop and a keep going action, plus a dropdown setting minutes until the next check-in (default 30). A custom pick applies to the next gap only, then reverts to the default.
  5. Cadence applies to every running timer, billable or not.
  6. The schedule is persisted/derivable so it survives relaunch: a check-in that came due while the app was closed fires once on relaunch, then resumes cadence. After a long sleep, cadence realigns from wake — no backlog of missed notifications.
Out of scopeOS-level user-idle detection (keyboard/mouse inactivity). It is fuzzy, OS-version-dependent, and a cross-application concern. Stint builds only on sleep/wake, which are clean and authoritative. See Appendix A.
11 CLI specification

tt — full parity

Human-readable tables by default; --json on every read command for scripting. Clean exit codes (0 success, non-zero on error). All writes go through @stint/core with BEGIN IMMEDIATE + busy timeout so they cooperate with the running app.

CommandDoes
tt start [desc]Stop any open entry, open a new one. Flags: --client --project --tag (repeatable) --no-bill --at <time>.
tt stopClose the open entry. --at <time> to backdate the stop.
tt statusShow the open entry + derived elapsed, or "nothing running." --json.
tt resumeStart a new entry from the last entry's description/client/project/billable.
tt add <desc>Backfill a completed entry. Requires --from --to; same attribute flags as start.
tt edit <id>Amend any field: --desc --from --to --client --project --tag/--untag --bill/--no-bill.
tt split <id> --at <time>Cut an entry into two at an instant within its span.
tt merge <id…>Merge a contiguous selection. --client/--project resolve conflicts.
tt rm <id>Delete an entry. Confirms unless --force.
tt listList entries. --today --week --range a b + filters + --json.
tt reportTotals. --week --month --range a b, --by client|project|day|tag, --round 15, --all|--non-billable, --csv|--json.
tt exportRaw entries for a range. --csv|--json, -o <file>.
tt client …add | rename | archive | ls.
tt project …add --client | rename | archive | ls.
tt sleep …Review sleep-flagged entries; tt sleep subtract <id> excludes slept time.

Time arguments accept absolute (14:30, 2026-06-24T14:30) and relative (-90m, -1h30m) forms. The database path resolves to TT_DB if set, else the default location (§13).

12 GUI specification

Menu bar + main window

  1. Tray (Electron Tray + popover BrowserWindow): the running timer, counting up; one click to stop, switch, or start; quick-pick recent client/project; opens the main window. This is the cross-platform analog of a native menu-bar item — the system tray on Windows/Linux, the menu bar on macOS — not pixel-identical OS chrome.
  2. Global hotkey (default Ctrl+Alt+T, registered via Electron globalShortcut as CommandOrControl+Alt+T) toggles the timer from anywhere — stops if running, resumes the last entry if idle.
  3. Main window: entries grouped by day with inline edit, split, merge, and delete; client/project/tag management; the report builder with on-screen summary and CSV/JSON export.
  4. Flags visible in context: overlap warnings and sleep-through flags appear on the affected entries with the one-tap subtract action.
  5. Empty states instruct rather than decorate (e.g. "No entries yet — press Ctrl+Alt+T or run tt start to begin").
13 Storage & schema

One file, overridable path

Default location: the OS's standard per-user app-data directory — the GUI resolves it via Electron's app.getPath('userData'), and tt resolves the documented per-OS equivalent (e.g. $XDG_DATA_HOME/~/.local/share on Linux, ~/Library/Application Support on macOS, %APPDATA% on Windows) — holding timetracker.sqlite (WAL mode). Both the app and tt resolve the same path, overridable by the TT_DB environment variable. Backup = copy the file; export = tt export. No network, ever.

-- schema sketch (illustrative)
CREATE TABLE client  (id INTEGER PK, name TEXT, archived INT DEFAULT 0);
CREATE TABLE project (id INTEGER PK, client_id INT REFERENCES client, name TEXT, archived INT DEFAULT 0);
CREATE TABLE entry (
  id INTEGER PK,
  client_id  INT REFERENCES client,        -- nullable
  project_id INT REFERENCES project,        -- nullable; implies client
  description TEXT,
  start_utc  TEXT NOT NULL,                 -- ISO-8601 UTC
  end_utc    TEXT,                          -- NULL = running
  billable   INT NOT NULL DEFAULT 1,
  excluded_seconds INT NOT NULL DEFAULT 0   -- subtracted sleep
);
CREATE TABLE tag (id INTEGER PK, name TEXT, archived INT DEFAULT 0);
CREATE TABLE entry_tag (entry_id INT, tag_id INT, PRIMARY KEY(entry_id, tag_id));
CREATE TABLE sleep_span (id INTEGER PK, entry_id INT REFERENCES entry,
  sleep_utc TEXT, wake_utc TEXT, source TEXT);
CREATE TABLE setting (key TEXT PRIMARY KEY, value TEXT);
-- invariant: at most one row with end_utc IS NULL, enforced in @stint/core transactions

15 Platform & tech

Cross-platform, shared-core

  • GUI: Electron — a Tray with a popover BrowserWindow for the running timer, and a standard BrowserWindow for lists and reports. The renderer is web tech (HTML/CSS, framework optional). Follows system light/dark; uses system typography; system accent applied only to the primary action and running state.
  • CLI: a Node executable, tt, built on commander with --json everywhere, importing the same @stint/core package.
  • Core: a TypeScript package, @stint/core, owning schema, transitions, invariants, reporting, and rounding. Both surfaces import it; no duplicated logic.
  • Persistence: SQLite via Node's built-in node:sqlite (Node 22+), WAL mode set by pragma, BEGIN IMMEDIATE writes, busy timeout for cross-process cooperation. No native build step.
  • System integration: fs.watch for file-watch; Electron powerMonitor suspend/resume events (with wall-clock-gap reconciliation on launch) for sleep flagging, and powerMonitor.getSystemIdleTime() available for idle; Electron globalShortcut for the hotkey; Electron Notification for check-ins.

17 Acceptance criteria

v1 is done when…

  1. A timer started in tt appears in the tray near-instantly, and vice versa; tt status and the tray never disagree.
  2. At most one entry is ever open, regardless of how rapidly start/stop is issued from either client.
  3. Closing the app, sleeping the machine, or crashing mid-timer never corrupts elapsed time.
  4. Stored entry times are never altered by rounding or sleep subtraction; both are reversible and applied only to derived/billable duration.
  5. A sleep through a running timer produces a flagged entry with second-accurate span(s) and a working one-tap subtract.
  6. The first check-in fires at 60 min and the cadence continues autonomously, surviving relaunch and realigning after sleep.
  7. Reports group, filter, and round per spec, flag overlaps and unreviewed sleep, and export valid CSV and JSON.
  8. Every GUI capability is reachable from tt.
  9. The app makes no network connections.
A Appendix — Power/idle APIs

What the process can actually observe

Research summary grounding §10, expressed against Electron's powerMonitor. Sleep/wake are event-backed (live suspend/resume), with a wall-clock-gap fallback on launch; idle is only pollable and fuzzy — which is why §10 builds on the former and excludes the latter. Note the honest loss versus the original macOS spec: there is no cross-platform kernel sleep/wake timestamp and no Power Nap / dark-wake distinction, so reconciliation is by wall-clock gap and a single sleep is one span.

SignalAvailable?How
Went to sleepYes, live eventpowerMonitor 'suspend' (stamp on receipt, source = event). If missed (app not running), inferred after the fact from a wall-clock gap, source = gap.
Woke from sleepYes, live eventpowerMonitor 'resume'. No cross-platform kernel wake timestamp; a wake missed while closed is recovered on launch by comparing the wall-clock gap against the open entry.
Became idlePoll only, fuzzyNo event. Poll powerMonitor.getSystemIdleTime() (seconds since last input); infer start as now − idle. The OS "idle" definition is OS-provided and version-variable. Excluded from v1 per §10.
Stopped being idleInferredIdle time resets to 0 on input; powerMonitor 'unlock-screen' as a coarse proxy.
Power Nap / dark wakeNot distinguishableNo cross-platform analog. powerMonitor surfaces user-visible suspend/resume only, so a maintenance wake is not separately observable and one sleep is recorded as one span. Accepted limitation versus the macOS-only original.
