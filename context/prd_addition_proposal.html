05 Timer & favorites — proposed additions

Pinning the resolution rules behind start & resume

  1. No end-of-day clamp on start / resume / favorite. new core Starting, resuming, or resuming-from-a-favorite always closes any currently open entry at now — there is no end-of-day / 23:59 clamp. The closing end equals the new entry's start (the same instant), so there is no gap, no overlap, and exactly one open entry.
    Why The audit found a resume-from-favorite path that closes a still-open entry at a 23:59 day boundary rather than at now — which contradicts the §04 R04 / §16 "close any open entry (end = now)" rule and must be ruled out explicitly. (The original draft also proposed a "switch instant & carry-forward default"; Switch is a redundant alias of Start being removed entirely — issue #34 — so that part is dropped, not adopted.)
    Status gap · spec-conflicting  Cites §04 R04 · §16 "Start while a timer runs"
  2. Favorite name uniqueness, explicit-attribute capture & not-found errors. new Favorite names are unique case-insensitively; creating a name that already exists (case-insensitively) is rejected with a defined error, never a silent overwrite. A favorite may be captured from a closed entry, the running entry, or explicit attributes. Operations naming an unknown favorite (rename, unpin, resume) fail with a defined not-found error and make no change. Whether uniqueness is DB- or store-enforced is stated (the §13 favorite sketch carries no UNIQUE, so it is store-enforced).
    Why §05 R09/R10 pin capture-from-entry/running-timer, resume-creates-a-fresh-entry, and favorite-never-mutated, but — unlike the report table — the §13 favorite name is not UNIQUE COLLATE NOCASE; the store guards uniqueness in code, resolves names case-insensitively, and rejects duplicates / unknown names with messages ("already exists", "no favorite named …") none of which the spec pins.
    Status partial partial  Cites §05 R09–R10 · §03 Favorite entity · §13 favorite/favorite_tag schema
  3. Resume-from-favorite override layering. new On tt start --fav / one-click resume, each unset flag inherits the template's value; --desc overrides description; --client/--project override the client/project pair together; --bill/--no-bill overrides billable; supplying one or more --tag replaces the template's tag set (not a merge); --at overrides the start time. The favorite row is never modified.
    Why §11 says resume "with explicit-flag overrides" without defining the per-field merge rules — in particular that tags replace rather than merge, that client/project are coupled, and that an unset flag inherits.
    Status partial partial  Cites §05 R10 · §11 tt fav start / tt start --fav
  4. Resume tie-break / "most recent." new "Most recent entry" (for plain resume) resolves to the entry with the latest start, with id as the descending tie-breaker.
    Why §05 R04 says "most recent entry" but never defines it; the store uses ORDER BY start_utc DESC, id DESC LIMIT 1, a contract callers and tests rely on.
    Status gap gap  Cites §05 R04
06 Editing — proposed additions

Boundary inequalities, merge composition & overlap shape

  1. Edit/add/split/stop boundary rules. new core Pin the boundary inequalities: a stop must be at or after start; a manual add requires to strictly after from; an edit's resulting end must be after its start; a split point must be strictly inside the span (an open entry's end treated as now). Violations are rejected with a clear error.
    Why §06 R02 says split happens "within its span" and §16 lists the edit cases, but the strict-vs-non-strict boundary at each operation (equal start/stop, to==from, split at exact start/end) is decided only in code.
    Status partial partial  Cites §05 R02/R05 · §06 R02 · §16
  2. Merge composition. new core A merge spans the earliest start to the latest end, but the merged entry is open iff any source is open — an open source's implicit "now" end never competes as the latest end; openness wins outright and the latest-end computation is skipped. Descriptions are concatenated in entry order with ' / ' and de-duplicated; tags are set-unioned; excluded_seconds are summed across sources; the first entry's client/project/billable win unless overridden (CLI uses the first entry's billable, with no override flag).
    Why §06 R03 specifies first-entry-wins, tag union, description concatenation and the override flags, but the audit found four unpinned, load-bearing specifics the code commits to: the open-iff-any-open result (which the "earliest start to latest end" prose actively contradicts for an open source), the ' / ' separator with de-dup, the summing of excluded_seconds, and CLI billable resolution.
    Status partial partial  Cites §06 R03 · §16 "Merge entries with differing attributes"
  3. Split attribute inheritance — excluded sleep not redistributed. new core A split copies client, project, billable, description, and tags onto both halves; excluded_seconds is left intact on the original (first) half and reset to 0 on the new (second) half — sleep exclusion is not divided across the cut.
    Why §06 R02 pins client/project/tags/billable inheritance but is silent on excluded_seconds; the billing consequence of splitting a sleep-subtracted entry (the first half keeps all the exclusion) is a deliberate but unstated choice.
    Status partial partial  Cites §06 R02
  4. Overlap semantics & warning shape. new core Overlap is computed on half-open intervals [start, end): two spans overlap iff aStart < bEnd and bStart < aEnd, so entries that merely touch at an endpoint do not overlap; an open entry's end is treated as now. Every entry write returns a non-blocking warning of kind 'overlap' carrying the count and ids of overlapped entries; the operation still succeeds (exit 0, normal stdout, the note on stderr). The per-entry banner reports only the single worst overlap (largest overlapping seconds; ties to the earliest-encountered neighbour), labelled previous when the neighbour starts strictly before and next otherwise; reported overlap seconds are symmetric and never exceed either entry's own duration.
    Why §06 R04 / §12 R09 pin warned-not-blocked and the "Overlap: Nm with previous/next entry" banner, but not the half-open boundary (touching ≠ overlap — a common back-to-back backfill case), the structured warning shape (kind:'overlap', overlappedEntryIds, open=now), the worst-neighbour selection + first-seen tie-break, the strict-before/at-or-after relation rule, or that the warning leaves exit code 0.
    Status partial partial  Cites §06 R04 · §12 R09 · §16 · glossary "Overlap"
  5. Overlap detection respects the billable filter. new Overlap detection considers all entries in the selected range, including those excluded by the active billable filter, so a billable entry overlapping a non-billable one is still flagged; the reported overlap set, however, lists only entries that survive the active filter.
    Why §06 R04 / §09 R05 establish overlap flagging and the billable-only default, but never address the interaction — that detection runs over the full set before the filter while reporting is restricted to the kept set.
    Status gap gap  Cites §06 R04 · §08 R03 · §09 R05
07 Reference data — proposed additions

Case-insensitive names, tag normalization & archive surface

  1. Case-insensitive name uniqueness & resolution. new Names of clients, projects, tags, favorites, and saved reports are unique and resolved case-insensitively; creating a name that already exists (case-insensitively) is an error, never a silent overwrite; renaming an entity to its own current name is a no-op. State which constraints are DB-enforced versus store-enforced (favorite uniqueness is store-enforced; report.name is DB-enforced UNIQUE COLLATE NOCASE).
    Why Only report.name is pinned UNIQUE COLLATE NOCASE (§13). For clients/projects/tags/favorites the case-insensitive uniqueness, case-insensitive lookup, and duplicate-create-errors rules live only in code.
    Status partial partial  Cites §07 R01 · §13 report schema
  2. Tag normalization, on-the-fly creation & idempotent re-apply. new Tag names are trimmed on input; empty or whitespace-only names are ignored; comma-separated input splits into discrete tags; tags are matched case-insensitively so a given name yields exactly one tag row (find-or-create on first apply); re-applying a tag already on an entry/favorite is idempotent (the join uses INSERT OR IGNORE). An entry's tags read back in a stable ascending (alphabetical) order.
    Why §07 R02 pins "created on the fly when first applied" and reusability, but trimming, empty-skip, comma-split, case-insensitive dedupe (one name → one row), idempotent re-apply, and the ascending read-back order (which export/grouping depend on) are code-only.
    Status partial partial  Cites §07 R02 · §03 "Flat, reusable, cross-cutting"
  3. Project–client authority at resolution time. new A supplied projectId's client_id is authoritative and overrides any supplied clientId. A project referenced by name with no client must already exist; if it does not, the operation errors and directs the user to name a client so the project can be created under one.
    Why The "project's client is authoritative" rule is pinned (§03 invariant), but the resolution-time behavior and error for a name-only, not-yet-existing project ("project \"X\" not found; name a client to create it under one") is an unstated implementation contract.
    Status partial partial  Cites §03 invariants · §07 R01 · glossary
  4. Archive list surface & rename propagation. new Listing clients/projects/tags returns only non-archived rows by default; an includeArchived option additionally returns archived rows marked archived. Archived rows are never deleted and remain usable in history. Renaming a client/project/tag is immediately reflected wherever that entity's name shows on existing entries.
    Why §07 / §03 pin archive-not-delete and "hidden from pickers, selectable in history," but not the default-excludes-archived list API, the includeArchived escape, or that a rename propagates to existing entry labels (denormalized vs joined-at-read is left open).
    Status partial partial  Cites §07 · §03 entity rows
09 Reports, search & export — proposed additions

Ordering, rounding reconciliation, range bounds, search & export rules

  1. Group and list ordering. new Define stable orderings: entries list by start ascending (id tie-breaker); clients/projects/tags/favorites/saved reports list ascending by name, case-insensitively (COLLATE NOCASE); sleep spans by sleep time. For grouped views, day groups render newest-day-first while client/project/tag groups render ascending by name; the empty buckets are labelled exactly (no client) / (no project) / (untagged) and sort first; group keys for day grouping are the local-date YYYY-MM-DD token.
    Why §09 R02 names the grouping dimensions but pins no ordering. The code commits to an asymmetric rule (days descending, names ascending), specific placeholder strings, empty-buckets-first, and a local-date day key — all user-visible contracts feature files assert via "groups exactly …" but the spec is silent on.
    Status gap gap (name/day ordering, labels) · partial on day-key locale  Cites §09 R02 · §11 list/report
  2. Tag-grouping fan-out & grand-total invariance. new When grouping by tag, a multi-tag entry appears under each of its tags (and once under (untagged) if it has none); per-tag line totals may therefore sum to more than the grand total, which remains the de-duplicated sum of distinct entries and is identical under every grouping.
    Why The §09 R02 note asserts grouping-invariance of the grand total, but the fan-out consequence (double-counting per-tag lines vs an invariant total) is implied by many-to-many tags yet never stated — exactly the surprising rule a re-implementer would get wrong.
    Status partial partial  Cites §09 R02 · §03 Tag many-to-many
  3. Rounding direction, ties & grand-total reconciliation. new Each grouped line rounds independently to the increment, to the nearest (not always-up), with exact half-way values rounding up (round-half-up). A parent/grand rounded total is the sum of its children's rounded values (round-then-sum), not the rounding of the aggregate exact total, so a parent's displayed total equals the sum of the lines shown under it. Stored time is never mutated.
    Why §09 R04 pins nearest-not-always-up, the {6,10,15,30} set, line-not-row, and display-only. Unpinned and money-affecting: the half-way tie rule, and whether grand_rounded_seconds is round(Σ) or Σ round — the report schema carries both grand and per-line rounded fields without constraining the relationship.
    Status partial partial  Cites §09 R04 · report.schema.json
  4. Unreviewed-sleep flag predicate & report ids. new A slept-through entry is "unreviewed" (and so flagged in reports) precisely when its excluded_seconds is strictly less than the entry's total slept seconds (the sum of its SleepSpans, each clamped ≥ 0); once excluded covers all slept time the entry is reviewed. The report's unreviewed_sleep_entry_ids is the set of such entries within range.
    Why §09 R05 / §17 surface "unreviewed sleep-flagged entries," but the exact predicate (slept-through AND excluded < total slept, strictly-less) and the report-schema array's membership rule are only implicit; a re-implementer could use excluded == 0 instead.
    Status partial partial  Cites §09 R05 · §17 R7 · glossary "subtract"
  5. Range preset resolution — local-anchored, half-open. new The five presets resolve against the user's local calendar as half-open [from, to) windows anchored at local midnight, then serialized to UTC: today = [midnight, +1 day); this week = [week-start-of-this-week, +7 days) honoring the Week-start setting; last week the preceding 7-day window; this month = [first-of-month, first-of-next-month); last month the preceding month. to is always exclusive; the resolved UTC bounds shift with the local offset and DST, and an entry's membership is decided by its local-calendar position.
    Why §09 R01 + the report-def schema enumerate the presets and §14 sets Week-start, but half-openness, local-midnight anchoring, the exact arithmetic, and the offset/DST consequence are code-only — and decide which entries fall in each preset.
    Status partial partial  Cites §09 R01 · §14 Week start · report-def.schema.json · §04 R06
  6. Local-zone day/week bucketing. new Day and week grouping (and all preset resolution) are computed in the user's local time zone — a "day" is a local calendar day, an entry is bucketed by the local date of its start — while stored timestamps and exported/custom range bounds remain UTC.
    Why §04 R06 pins UTC storage + local display + DST-safe durations, but never says which time zone defines a "day" for grouping. The code buckets by local date; near midnight this changes which calendar day an entry lands under.
    Status partial partial  Cites §04 R06 · §16 DST rule · §13
  7. Free-text search match rule. new Search is a case-insensitive substring match of the query against the entry's description, client name, project name, and any tag name (a hit in any field includes the entry, matched as equal fields); an empty or whitespace-only query applies no narrowing and matches all entries in the active range/filters; search composes (AND) with the active range and billable filter.
    Why §09 R07 pins the searched fields, case-insensitivity, and AND-composition, but not substring-vs-prefix/token, the blank-matches-all rule, or whether the "secondarily" wording denotes a real ranking (the code treats the fields equally). Project must be matched as a field, not only via a rendered "Client / Project" label, identically across the live view, the IPC path, and tt --search.
    Status partial partial  Cites §09 R07 · §12 R09 · parity-matrix
  8. Export is raw, billable=all, unaffected by report filters. new Export emits every entry in the range regardless of billable/client/project/tag report filters (each row carries its own billable flag); the exported overlapped flag is computed over exactly the exported set (open entry's end = now), so it depends on the export's own selection. CSV uses a fixed header matching the column list with ISO-8601 UTC timestamps; JSON is pretty-printed (2-space) with a trailing newline; the GUI export is byte-identical to tt export for the same range. Default range when none is given is the current week; the default filename is stint-export-<YYYY-MM-DD>.<ext> keyed off the range start.
    Why §09 R06 pins the column set and "byte-identical GUI vs tt" only in implementation notes; the deliberate raw / billable=all / no-narrowing rule, the export-set scoping of overlapped, CSV quoting + date encoding, JSON indent/newline, the default range, and the filename pattern are unstated.
    Status partial partial  Cites §09 R06 · §11 tt export · export-entry.schema.json · parity-matrix
  9. Saved-report reference, errors & export semantics. new A saved report may be referenced by name or numeric id; an unknown name produces a defined error ("no saved report named …"). Exporting from a saved report emits the raw entries for the resolved range (billable='all', no client/project/tag/search narrowing) — byte-identical to tt export over that window — whereas the grouped/human and report-result views honor the definition's filters and rounding. On report save with no --round, rounding is stored off (a saved definition is self-contained), unlike an ad-hoc report which inherits the settings rounding; report edit patches rounding only when --round/--no-round is supplied (--no-round exists only on edit).
    Why §09 R08–R09 establish saved reports and "export from a saved report," but name-or-id resolution, the unknown-name error, the surprising raw/unfiltered CSV export (vs the filtered human view), and the three divergent rounding-resolution paths (ad-hoc inherits / save off / edit untouched) are code-only.
    Status partial partial · the three rounding paths are gap  Cites §09 R08–R09 · §13 report schema · §11
10 Running-timer hygiene — proposed additions

Sleep reconciliation, subtract toggle & check-in cadence

  1. Gap-reconciliation threshold & clamp. new On-launch gap reconciliation takes (lastSeenUtc, nowUtc) and records a source='gap' SleepSpan on the open entry only when the wall-clock gap (last-seen → now) strictly exceeds 90 seconds AND an entry is open; otherwise it makes no change (no span, excluded_seconds untouched), returning a null/no-op for both the sub-threshold and nothing-open cases. The recorded span is clamped so it never starts before the open entry's start.
    Why §10 specifies the reconciliation mechanism and its gating on an open entry but deliberately hedges the magnitude ("a gap large enough to imply the machine slept"). The 90-second threshold, the clamp to the entry start, and the null-vs-empty-span contract are undocumented constants in code.
    Status partial partial · threshold value is gap  Cites §10 item 3 · glossary "Reconcile" · Appendix A
  2. Subtract-sleep is a value-equality toggle. new Subtract-sleep decides it is "already applied" by value equality — excluded_seconds exactly equals the current summed slept seconds (no separate applied flag). Re-running restores excluded_seconds to 0; if sleep spans change between presses the new total re-applies; an excluded_seconds set by another mechanism that happens to equal the slept sum will toggle off. Stored start/end are never altered. The spec should state whether this value-equality reversibility is the intended contract or whether an explicit applied-state should be tracked.
    Why §10 / §17 R04 pin "precisely and reversibly," but the mechanism (value equality against the slept sum, no recorded applied-state) and its two ambiguous edges (foreign equal value toggles off; baseline moves when spans change) are real correctness questions for a billing action the spec leaves to code.
    Status partial partial  Cites §10 item 5 · §17 R04 · glossary "Subtract"
  3. Check-in realignment & collapsed backlog. new On relaunch or wake the next check-in is the first interval-aligned instant strictly after now (so the next due time lies in (now, now+interval]), and exactly one check-in fires for any number of missed slots. The number of collapsed missed slots is reported as collapsedBacklog = the count of nominal-grid slots falling strictly between the previously-due slot and the fire instant (0 on-time, 1 for one ignored slot, N for a sleep spanning N slots). The nominal grid is enumerated on a half-open [from, to) window (a slot landing exactly on to is excluded).
    Why §10 pins "fires once, then realign from wake — no backlog," but not the realignment formula, the observable collapsedBacklog field/semantics (and its off-by-one), or the half-open grid-enumeration boundary.
    Status partial partial · the field name & grid bounds are gap  Cites §10b R06 · §16 "Check-in due while app closed"
  4. Custom-pick anchoring & backlog interaction. new A custom interval pick of M minutes at fire-instant T sets the next due time to exactly T+M (anchored at the fire instant, re-basing the grid), and the following default due is T+M+defaultInterval — it does not re-snap to the original nominal grid. When a single evaluation both collapses a backlog and receives a custom pick, the custom pick wins and re-bases the grid; the spec should pin whether collapsedBacklog is then the slots actually missed or zeroed.
    Why §10 pins "a custom pick applies to the next gap only, then reverts to the default," but not the fire-instant anchoring (T+M, then T+M+default) versus re-snapping to the original grid, nor the combined backlog-collapse-plus-custom-pick seam.
    Status partial partial · the combined seam is gap  Cites §10 "custom pick applies to the next gap only"
  5. Check-in scheduler totality. new The check-in interval must be a positive integer; a non-positive or unparseable stored interval is rejected/clamped to the default on read. The pure scheduler must be total: given any interval value (including zero or negative), evaluating the schedule terminates and fires at most once, never looping. Settings validation is the primary guard; the evaluator is a defensive backstop.
    Why The spec mandates the cadence defaults and treats the interval as a validated positive setting (§10b, §14), but is silent on the behavior of the pure evaluator handed a non-positive interval — termination / at-most-once-fire is a purely defensive code choice with no requirement behind it.
    Status gap gap  Cites §10b · §14
  6. app_state canonical keys & idle-clears-schedule. new core Fix the canonical app_state key names — checkin_state (the check-in schedule) and last_seen_utc (the heartbeat) — as core-owned constants no surface duplicates. Any transition to an idle DB (notably stop()) deletes the schedule row in the same transaction as the entry close; an open entry implies a schedule present and anchored at its start, an idle DB implies no schedule. A persisted schedule with no open entry is an invariant violation, holding across reopen.
    Why §20 R07 pins the same-transaction co-commit, and the §13 sketch describes the table's contents in prose, but not the literal key strings (a cross-module contract a typo breaks) nor the stronger idle ⇒ schedule-absent direction (R07's "co-committed" wording covers writing, not the deletion on stop).
    Status partial partial  Cites §20 R07 · §13 app_state · §10b R06
  7. Heartbeat cadence, reconciliation bound & clean-quit final write. new While a timer runs the last-seen heartbeat is persisted at most once per fixed cadence (e.g. 30s), bounding the maximum error in gap-reconciled sleep bounds to that cadence even on a hard kill. On a clean quit the app writes a final last-seen so the next launch's reconciliation has a tight bound; if that write or the store close fails, the bound falls back to the throttled cadence and the failure is logged, never silently dropping recoverable sleep.
    Why §04 / §12 R03 pin the 1-second display tick and §20 R07 pins last-seen durability, but the heartbeat write cadence (30s), the resulting reconciliation-precision bound, and the clean-quit-vs-kill distinction (final write on quit) are unstated tuning the precision rests on.
    Status partial partial · the clean-quit write is gap  Cites §04 · §12 R03 · §20 R07 · §10a item 3
11 CLI contract — proposed additions

Exit codes, env vars, output strings & flag resolution

  1. Exit-code taxonomy. new Pin the scripting contract: 0 for success (and for --help/--version display); a single non-zero code (2) for all anticipated failures — domain/store errors, usage errors, and time-parse errors (and a CommanderError without its own code); a distinct code (1) reserved for unexpected/uncaught errors. A refused destructive action (rm/backup restore without --force) exits non-zero.
    Why §11 pins only "0 success, non-zero on error" and that exit codes are GOLD-asserted. The fine partition (2 for domain/usage/parse vs 1 for unexpected) that a branching script relies on lives only in bin.ts.
    Status partial partial  Cites §11 · acceptance §08
  2. Environment surface. new Enumerate tt's env vars: TT_DB (DB path override, already specified); TT_NOW (pins the clock to a given ISO instant; ignored when blank/unparseable, falling back to the real clock); TT_BUSY_TIMEOUT_MS (overrides the SQLite write-lock busy timeout; default 5000ms; ignored unless a finite value ≥ 0).
    Why Only TT_DB is specified. TT_NOW (a clock seam for scripting/tests) and TT_BUSY_TIMEOUT_MS are absent from the corpus, as is the concrete 5000ms default — "a busy timeout" is described only qualitatively.
    Status partial partial · TT_NOW / TT_BUSY_TIMEOUT_MS are gap  Cites §13 TT_DB · §11/§15 "busy timeout"
  3. Stable output lines & refusal text. new Pin the human/machine output lines for add/split/merge ("added entry <id>", "into <a> and <b>", "merged into entry <id>") and the rm/backup restore refusal text (a clear message naming the target and instructing --force); domain warnings (e.g. overlap) are written to stderr with a stable warning: prefix while the command succeeds (exit 0) and emits its normal stdout. Human-table timestamps render as short ISO to second precision (millis stripped); a null/open timestamp renders as an em-dash —; the FLAGS column shows comma-joined overlap/slept tokens (overlap recomputed over the displayed set).
    Why §11 pins the confirm-gate and the warned-not-blocked policy, but the exact success/refusal/warning strings, the warning: prefix and its generalization to all domain warnings, the human-table timestamp format, the — null render, and the FLAGS tokens are code/golden-only contracts pipelines depend on.
    Status partial partial  Cites §11 · §06 R04 · §10 · §16
  4. stderr hygiene. new tt keeps stderr free of runtime noise that would corrupt the scripting/golden contract: the node:sqlite ExperimentalWarning (§15) is suppressed while all other Node warnings pass through unchanged (same text, formatting, once-per-warning dedup), without depending on warning-listener attachment order.
    Why Nothing in the corpus mentions the ExperimentalWarning suppression; golden-comparable stderr (acceptance §08) implies a clean stderr, but the specific suppress-exactly-the-SQLite-warning-pass-everything-else contract is unstated.
    Status gap gap  Cites §15 "stability-experimental module"
  5. Filter, range & grouping defaults per command. new State the no-flag defaults and their deliberate asymmetry: tt list with no range shows all entries (unbounded) and is ungrouped/flat; tt report/tt export with no range default to the current week, and report/report save default to group-by-client; tt report save with no range stores the week preset. A --client/--project/--tag filter naming a non-existent entity narrows to empty (zero entries), never erroring or silently dropping the filter. Billable-filter precedence is uniform across commands: --non-billable > --all > default billable-only, and the same flag set is exposed by ad-hoc report, report save, and report edit.
    Why §09/§11 list the flags and the billable-only default, but the per-command range/grouping default asymmetry, the unknown-name-narrows-to-empty rule, and a uniform billable-filter precedence (the audit found two opposite resolutions by subcommand and an extra flag only on report edit) are unspecified.
    Status partial partial · unknown-filter & per-command asymmetry are gap  Cites §08 R03 · §09 R01/R02 · §11
  6. Entity-reference resolution & aliases. new For manage subcommands a purely numeric <ref> is treated as the entity's id; any other string is matched against its name; a name with no match fails with a clear "no match" error and non-zero exit (note the rare consequence for entities named entirely with digits). Enumerate command aliases / default subcommands: ls as an alias for list; bare tt sleep and bare tt config default to their ls subcommand. Because tt report is both an ad-hoc query and a parent of saved-report subcommands sharing option names, option parsing binds flags to the correct level (report ls --json vs report --week --json).

Corruption taxonomy for the recovery gate. new core The startup check is PRAGMA quick_check specifically (faster than integrity_check, with the accepted trade-off that it skips the exhaustive UNIQUE/row-count pass). The DB is healthy only when quick_check returns exactly one row equal to the literal 'ok'; any other row, any non-ok message, or a throw while running the pragma is a failed check. Define what counts as corruption for recovery: a failed integrity check and any other open-time throw (e.g. an unreadable/clobbered header) are treated as corruption (recover); a pragma/configuration refusal (DbOpenError) is not corruption and propagates without quarantine. Recovery never falls back to a fresh empty DB.
    Why R03/R05 pin quick_check, no-write-on-failure, quarantine + restore, and never-empty-start, and the runbook commits to quick_check, but the OK oracle (exactly one 'ok' row; a pragma throw is failure), the trade-off note, and the taxonomy distinguishing arbitrary open-time throws (corruption) from pragma refusals (propagate) are unstated. R03's "quick_check / integrity_check" slash should be tightened to the runbook's commitment.

  3. Backup is conditional, self-contained & best-effort. new core "Changed since the last backup" is determined by a full-content fingerprint (a content hash, e.g. SHA-256) of the checkpointed main file compared against the latest backup — never file size or mtime (under WAL a same-size in-place rewrite must still trigger a backup). Equal fingerprints skip the copy (backupNow/launch return null). A backup is a single self-contained file: the WAL is checkpointed into the main file before the copy and no WAL/SHM companion is written beside the .bak, so any one backup file is independently restorable; a failed/empty checkpoint does not abort the copy. Launch backups are best-effort — a failure is logged but never blocks opening the DB — and a no-op for an unchanged DB, an in-memory DB (:memory:), or a dbPath whose file does not yet exist (neither throwing nor creating a stray file).
    Why R04 pins "if the DB changed, write a copy; keep last N; checkpoint-then-copy," but the change-detection metric (content hash, not size/mtime), the single-self-contained-file restorability property, the best-effort (failure-never-blocks-open) rule, the null-when-unchanged/in-memory/missing-file returns, and the no-stray-file guards are unstated.

  4. Backup naming & name-order authority. new core Backups are named <dbFilename>.bak-<YYYYMMDDTHHMMSSZ> where the stamp is the UTC instant in fixed-width zero-padded form (no separators, no millis). Because the stamp is fixed width, lexicographic name order equals chronological order; "latest backup" selection and retention pruning order backups by this in-name timestamp (not file mtime, which serves only as a same-second tiebreaker — most-recently-written wins, or state that same-second collisions are impossible by construction). The Settings/CLI list is newest-first.
    Why The filename grammar and "newest-first" are pinned (backup.schema.json, runbook), but the load-bearing invariant — ordering is computed from the in-name stamp (lexicographic == chronological), not mtime, and the fixed-width padding is what guarantees it (the schema's separate created_utc could be misread as the sort key) — plus the same-second tiebreaker are the genuine gap.

Quarantine sibling naming & companion grouping. new core The corruption path quarantines the bad file as <dbPath>.corrupted-<YYYYMMDDTHHMMSSZ> and a user-initiated restore displaces the live file as <dbPath>.replaced-<YYYYMMDDTHHMMSSZ> — the UTC timestamp suffix avoiding collisions across repeated events so no prior quarantine is overwritten. In both paths the WAL/SHM siblings are moved alongside the main file under the same grouping (or removed if unmovable), and original bytes are never destroyed. (Reconcile the §16/§20 prose, which says bare .corrupted, with this timestamped form.)
    Why The criteria layer already pins both .corrupted-<ts> and .replaced-<ts> with the suffix (contradicting the PRD's bare .corrupted), but that the WAL/SHM companions travel with the main file under the same suffix (or are removed) is unspecified everywhere, and the PRD prose needs reconciling to the timestamped form.

Recovery edges: no-backup, refused-open immutability & named-restore errors. new core If no good backup exists, recovery fails loudly (a RecoveryError) and leaves the corrupt file untouched — never an empty start. On a refused integrity-failed open with no recovery available, the corrupt file is left byte-for-byte unchanged (no data write, truncate, journal creation, or metadata touch), observably both "open refused" and "file untouched." A user-initiated restore naming a backup that does not exist beside the DB fails with a RecoveryError, performs no quarantine and no copy, and leaves the live database byte-for-byte unchanged; the error names the missing backup, identically on both surfaces.
    Why R05 specifies quarantine + restore-from-latest + never-silently-lose-data, but the no-backup-then-RecoveryError-and-leave-the-file behavior is only in impl prose; the stronger byte-for-byte immutability of a refused open (vs merely "no write") and the entire manual restore-of-an-unknown-name failure path (the spec describes restore success only) are gaps.
