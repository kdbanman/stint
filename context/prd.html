01 Overview & goals

What Stint is

Stint is a single-user desktop application for tracking billable time on macOS and Linux, in the spirit of early Toggl: a live start/stop timer, with manual backfill, organized by client. It runs locally, stores all data in one SQLite file, and is fully offline. Its output is trustworthy time totals per client and date range, exported as CSV or JSON, which the user turns into invoices in a separate tool. It installs in one step — the app in Applications / the app launcher and tt on PATH — and updates itself from GitHub Releases.

Primary goals

  • Accurate billable hours. The stored total is always the exact truth; nothing is silently rounded, added, or removed.
  • Durable, recoverable data. Committed time survives crashes and power loss; automatic backups and corruption recovery mean a single bad file never loses the record (§20).
  • Equal CLI and GUI. Anything the tray app can do, tt can do, over the same database, with no privileged surface — including favorites and saved reports.
  • Low-friction tracking. Starting a timer takes one keystroke or one click; forgetting to start one is always recoverable by hand; common timers are one click away as pinned favorites.
  • Quiet, minimal, native. A monochrome desktop app that respects system light/dark and stays out of the way, installed and updated like any other macOS/Linux app.

Primary user

A freelancer, consultant, or contractor working solo at their own computer, tracking a handful of clients, comfortable in a terminal, who invoices from the exported numbers using other software.

03 Concepts & domain model

Entities and the core invariant

The conceptual keystone: a running timer is the single time entry whose end is null. "Running" is a row state, not a separate object. Elapsed time is always derived (now − start), never stored or incremented.

EntityKey fieldsNotes
Clientid, name, archivedTop of the hierarchy. No rate. Archivable, not deleted, once referenced.
Projectid, client_id, name, archivedOptional refinement. A project belongs to exactly one client; selecting a project implies its client.
Entryid, client_id?, project_id?, description?, start, end?, billable, excluded_secondsThe core record. end null ⇒ running. Times stored UTC. client_id nullable (clientless = internal time).
Tagid, name, archivedFlat, reusable, cross-cutting. Many-to-many with entries via a join.
Favoriteid, name, description?, client_id?, project_id?, billable, tagsnew A named, pinned timer template. One-click resume starts a fresh timer from it (§05 R09–R10).
Reportid, name, range_spec, group_by, filters, roundingnew A saved, named report definition (§09 R08–R09). range_spec is relative (e.g. "this-week") or an absolute range.
SleepSpanid, entry_id, sleep_at, wake_at, sourceA sleep→wake cycle that occurred during a running entry. Multiple per entry. source = event | gap | unknown.
Settingkey, valueKey-value app config (rounding, week start, check-in cadence, etc.).

Invariants

  • At most one open entry (end IS NULL) exists at any time. Enforced transactionally in the shared core and at the DB level by a partial unique index (§20 R02).
  • Billable duration of an entry = (end ?? now) − start − excluded_seconds. Stored truth (start, end) is never mutated by rounding or sleep subtraction.
  • A project's client is authoritative: an entry with a project takes that project's client.
Core requirement A requirement is core iff it does one of: (a) ensures data integrity — atomicity, invariants, immutability of stored truth; (b) protects against data loss — crash-safety, durability, backups, the export escape hatch, destructive-action confirmation; or (c) enables core data entry — creating entries, starting timers, stopping timers, manual backfill. Core requirements carry a core badge and get a screen recording in QA evidence. Sleep accuracy (§10), the offline guarantee (§17 R09), and cross-surface sync (§17 R01) were considered and ruled not core — they concern accuracy, privacy, and consistency, not integrity-or-loss. The new durability requirements of §20 are all core.
04 Architecture

One core, one file, two thin shells

The requirements below are normative. For the consolidated picture — topology, write-path and IPC diagrams, and the file-level @stint/core module map — see architecture.html, which gathers this section together with §13, §15, §19, and §20.

  1. Shared core. A TypeScript package (@stint/core) owns the schema, every state transition, and all invariants. The tray app and the tt CLI both import it; there is no duplicated logic and no privileged surface. implemented
  2. Single source of truth. core One SQLite file in WAL mode (concurrent readers, single writer). All reads and writes go through @stint/core. Open-time invariants and durability pragmas are asserted on every open (§20 R01). implemented
  3. Derived elapsed. core No process holds or increments a timer. Elapsed is computed now − start on demand; the tray's per-second update is display only. Duration arithmetic tolerates wall-clock skew (§20 R06). implemented
  4. Atomic transitions. core Starting a timer is one transaction (BEGIN IMMEDIATE): close any open entry (end = now), insert a new open entry. This makes "start while running" safe regardless of which surface triggers it. The CLI uses a busy timeout so it cooperates with the running app. implemented
  5. Change propagation by file-watch. Each surface watches the SQLite file (and its -wal) via fs.watch and refreshes on change, so a write from one surfaces in the other near-instantly. (The tray's 1-second display tick is independent of this.) This is a consistency/UX affordance and is not a core requirement. implemented
  6. UTC storage, local display. core All timestamps stored UTC; rendered in the system locale and time zone. Durations are timezone-independent and DST-safe. implemented
  7. Renderer reaches data only through the core. core The GUI's renderer holds no direct database, filesystem, or network access; it reaches stored data exclusively through the shared core across a controlled bridge, so every read and write passes through the same invariants. implemented

The integrity and durability guarantees these requirements rest on are hardened in §20 — open-time pragma assertions, a DB-level at-most-one-open index, integrity checks, backups, and corruption recovery — and the write path that ties them together is diagrammed in architecture.html §04.

10 Running-timer hygiene

Two independent mechanisms

Sleep-span flagging suspend + resume

  1. While an entry is open, the app listens for the OS suspend and resume events (Electron powerMonitor) and timestamps each on receipt (source = event). implemented
  2. Each suspend→resume cycle during the open entry is recorded as a SleepSpan and attached to the entry, which is flagged "slept through." Multiple cycles per timer are all captured. implemented
  3. On launch, the app reconciles by comparing the wall-clock gap since its last recorded timestamp against the open entry: a gap large enough to imply the machine slept is recorded as a SleepSpan (source = gap), recovering a cycle missed while the app was not running. The wall-clock comparison captures that a sleep occurred and its approximate bounds. implemented
  4. The timer keeps running through sleep (now − start stays honest as wall-clock); the flag and spans let the user decide about the dead time. implemented
  5. Slept time handling: flagged, not auto-subtracted. A one-tap action (GUI) / flag (tt) adds the span(s) to excluded_seconds, trimming billable duration precisely and reversibly. implemented
  6. powerMonitor reports user-visible suspend/resume, so each sleep is recorded as one span. A sleep that occurs while the app is closed is recovered by the wall-clock-gap reconciliation in item 3 (see Appendix A). implemented

Not coreSleep accuracy improves correctness, but it is an accuracy concern, not data integrity or loss; it is deliberately not classified core (§03).

Check-ins notification

  1. The first check-in notification fires at start + 60 min, then every 30 min by default. implemented
  2. The schedule is autonomous: if the user ignores a notification, the next still fires on time. implemented
  3. The notification shows light context — the description if present (and client if set), otherwise "your timer." implemented
  4. It offers an inline Stop and a keep going action, plus a dropdown setting minutes until the next check-in (default 30). A custom pick applies to the next gap only, then reverts to the default. implemented each fired notification offers Stop, Keep going, and an inline set of interval choices (+15m/+30m/+60m/+120m — the OS-notification form of the dropdown); a pick reschedules only the next check-in, then cadence reverts to the configured default without touching the persisted setting (PROP-proven next-gap-only).
  5. Cadence applies to every running timer, billable or not. implemented
  6. The schedule is persisted/derivable so it survives relaunch: a check-in that came due while the app was closed fires once on relaunch, then resumes cadence. After a long sleep, cadence realigns from wake — no backlog of missed notifications. The schedule state is written in the same transaction as the change that sets it (§20 R07). implemented
Out of scopeOS-level user-idle detection (keyboard/mouse inactivity). It is fuzzy, OS-version-dependent, and a cross-application concern. Stint builds only on sleep/wake, which are clean and authoritative. See Appendix A.

13 Storage & schema

One file, overridable path core

Default location: the OS's standard per-user app-data directory — the GUI resolves it via Electron's app.getPath('userData'), and tt resolves the documented per-OS equivalent ($XDG_DATA_HOME / ~/.local/share/stint/… on Linux, ~/Library/Application Support/… on macOS) — holding timetracker.sqlite (WAL mode). Both the app and tt resolve the same path, overridable by the TT_DB environment variable. Backup = the automatic timestamped copies of §20 R04 (and copying the file by hand); export = tt export. No network, ever.

-- schema sketch (illustrative)
CREATE TABLE client  (id INTEGER PK, name TEXT, archived INT DEFAULT 0);
CREATE TABLE project (id INTEGER PK, client_id INT REFERENCES client, name TEXT, archived INT DEFAULT 0);
CREATE TABLE entry (
  id INTEGER PK,
  client_id  INT REFERENCES client,        -- nullable
  project_id INT REFERENCES project,        -- nullable; implies client
  description TEXT,
  start_utc  TEXT NOT NULL,                 -- ISO-8601 UTC
  end_utc    TEXT,                          -- NULL = running
  billable   INT NOT NULL DEFAULT 1,
  excluded_seconds INT NOT NULL DEFAULT 0   -- subtracted sleep
);
CREATE TABLE tag (id INTEGER PK, name TEXT, archived INT DEFAULT 0);
CREATE TABLE entry_tag (entry_id INT, tag_id INT, PRIMARY KEY(entry_id, tag_id));
CREATE TABLE favorite (                    -- NEW: pinned timer template (§05 R09)
  id INTEGER PK, name TEXT NOT NULL,
  description TEXT,
  client_id  INT REFERENCES client,        -- nullable
  project_id INT REFERENCES project,        -- nullable
  billable   INT NOT NULL DEFAULT 1);
CREATE TABLE favorite_tag (favorite_id INT, tag_id INT, PRIMARY KEY(favorite_id, tag_id));
CREATE TABLE report (                      -- NEW: saved report definition (§09 R08)
  id INTEGER PK, name TEXT NOT NULL UNIQUE COLLATE NOCASE,
  range_kind TEXT NOT NULL,                -- 'preset' | 'absolute'
  range_preset TEXT,                        -- today|week|last-week|month|last-month (NULL when absolute)
  range_from_utc TEXT, range_to_utc TEXT,   -- absolute bounds (NULL when preset)
  group_by   TEXT NOT NULL,                 -- client|project|day|tag
  billable_filter TEXT NOT NULL,            -- billable|all|non-billable
  client_id INT REFERENCES client, project_id INT REFERENCES project,
  tag TEXT, search TEXT,                     -- optional narrowing filters
  rounding INT NOT NULL DEFAULT 0,          -- 0=off, 1=on (display-only, §09 R04)
  rounding_increment_min INT NOT NULL DEFAULT 0,
  created_utc TEXT NOT NULL);
CREATE TABLE sleep_span (id INTEGER PK, entry_id INT REFERENCES entry,
  sleep_utc TEXT, wake_utc TEXT, source TEXT);
CREATE TABLE setting (key TEXT PRIMARY KEY, value TEXT);
CREATE TABLE app_state (key TEXT PRIMARY KEY, value TEXT);  -- heartbeat / last-seen & check-in schedule (§10, §20 R07)
-- at-most-one-open invariant: enforced in @stint/core transactions AND by a
-- partial unique index on the constant (1) WHERE end_utc IS NULL (§20 R02).

On every open, the core asserts the durability pragmas (journal_mode=WAL, foreign_keys=ON, a busy timeout, synchronous per §20 R01) and runs an integrity check before writing (§20 R03). See §20 for the WAL assertion, FK enforcement, integrity check, the at-most-one-open index, and the backup / recovery story.

15 Platform & tech

macOS + Linux, shared-core

The stack at a glance; architecture.html §09 lays it out as a layer table and carries the node:sqlite runtime-constraint detail.

  • GUI: Electron — a Tray with a popover BrowserWindow for the running timer (single-click popover, no dropdown menu — §12 R01), and a standard BrowserWindow for the sidebar shell and its views. The renderer is web tech (HTML/CSS, framework optional). Follows system light/dark; uses system typography; system accent applied only to the primary action and running state.
  • CLI: a Node executable, tt, built on commander with --json everywhere, importing the same @stint/core package.
  • Core: a TypeScript package, @stint/core, owning schema, transitions, invariants, reporting, and rounding. Both surfaces import it; no duplicated logic.
  • Persistence: SQLite via Node's built-in node:sqlite (Node 22.5+), WAL mode set by pragma, BEGIN IMMEDIATE writes, busy timeout for cross-process cooperation. No native build step. node:sqlite must load in both runtimes — the system Node behind tt and Electron's bundled Node — which is the stack's sharpest version dependency; the full constraint is in architecture.html §09.
  • System integration: fs.watch for file-watch; Electron powerMonitor suspend/resume events (with wall-clock-gap reconciliation on launch) for sleep flagging, and powerMonitor.getSystemIdleTime() available for idle; Electron globalShortcut for the hotkey; Electron Notification for check-ins.

16 Edge cases & rules

Decided behavior

SituationBehavior
Start while a timer runsAtomically stop the open entry, then open the new one.
App closed for an hour, timer openOn reopen, elapsed is correct (derived now − start); nothing to recover.
Crash with a timer openEntry remains correctly open; it is simply still running on relaunch.
Manual edit overlaps another entryWarn, allow, flag in reports.
Merge entries with differing attributesGUI prompts which client/project/billable to keep; descriptions concatenated, tags unioned. CLI uses first entry unless overridden.
Check-in due while app closedFire once on relaunch, then resume cadence.
Long sleep mid-timerRecord sleep span(s), flag entry, realign check-in cadence from wake (no backlog).
Entry spans a DST changeDuration correct (UTC math); displayed in local time.
Wall clock jumps (NTP / manual)Duration arithmetic tolerates the skew; now < start never yields negative/garbage elapsed (§20 R06).
Update arrives mid-timerThe in-app update never touches the database; the running entry is untouched and still running after the app is replaced (§19 R04).
Fresh launchIf the DB changed since the last backup, write a timestamped backup beside it before any write (§20 R04).
Corruption detected on openDo not write; quarantine the corrupt file (.corrupted) and restore from the latest good backup, informing the user (§20 R03/R05).

20 Data durability & integrity hardening

Committed time survives — every requirement here is core

This section hardens the integrity and loss-protection guarantees the rest of the PRD rests on. Every requirement is core and new.

  1. DB open invariants. new core On every open, assert/enforce journal_mode=WAL, foreign_keys=ON, a busy timeout, and synchronous=FULL (or NORMAL under WAL with documented rationale) so committed writes survive power loss. Done when an open with the wrong pragmas is corrected or refused before any write. implemented @stint/core openDb calls assertOpenPragmas on EVERY open — it SETS the pragmas in dependency order (busy_timeout, then journal_mode=WAL, then foreign_keys=ON, then synchronous=FULL) and then READS THEM BACK before migrate() runs: if any pragma is not at its required value it retries the full set once, and still failing throws a DbOpenError so the open is refused before any write. synchronous=FULL is chosen over NORMAL so a committed write survives power loss (an extra fsync per commit, irrelevant for this single-user low-write-rate tracker); :memory: skips WAL/synchronous (no journal/durability concept) but still sets busy_timeout + foreign_keys. Pinned by GOLD (core/test/gold/contracts.test.ts "DB open durability pragmas (§20 R01)" — on-disk read-back journal_mode='wal'/foreign_keys=1/busy_timeout>0/synchronous=2 (FULL), and a :memory: open with FK on + busy_timeout>0) and PROP (core/test/prop/open-invariants.test.ts — the invariant holds for any generated busyTimeoutMs).
  2. At-most-one-open enforced at the DB level. new core A partial unique index on end_utc IS NULL guarantees the single-open invariant at the storage layer, in addition to core logic (defense in depth). Done when a second open row is rejected by the index even if core logic were bypassed. implemented The v3 schema adds CREATE UNIQUE INDEX one_open_entry_idx ON entry((1)) WHERE end_utc IS NULL in @stint/core db.ts. It indexes the CONSTANT (1) — not end_utc — deliberately: every open row has end_utc = NULL and SQLite treats NULLs as DISTINCT in a unique index, so an entry(end_utc) index would permit unlimited open rows; collapsing all open rows onto the value 1 makes a second open row collide with UNIQUE constraint failed: index 'one_open_entry_idx' while closed rows (end_utc NOT NULL) stay excluded from the partial index (unlimited closed rows, reopen-after-close free). It is defense-in-depth alongside the human-readable one_open_entry_insert/update triggers: PROP core/test/prop/invariants.test.ts ("a raw second open INSERT is rejected by the DB index (§20 R02)") drops the triggers and proves the index ALONE rejects a RAW second open INSERT — bypassing the transactional core path — for arbitrary generated attributes, while unlimited closed rows never collide; GOLD core/test/gold/contracts.test.ts "schema shape (§13)" pins the index name, the UNIQUE + partial WHERE end_utc IS NULL clause, and the constant-(1) shape. The v2→v3 bump is additive (the triggers already held v2 DBs to one open row, so the index never fails on existing data).
  3. Integrity check on open. new core Run PRAGMA quick_check / integrity_check at startup; on failure, do not write — trigger recovery (R05). Done when a corrupt file is detected before any write occurs. implemented @stint/core openDb runs checkIntegrity (quick_check) before migrate on every file-backed open; on failure it does not write and triggers recovery via the §20 R05 seam (quarantineAndRecover), so a corrupt DB never falls through to normal operation. The bare detect-and-refuse half (corruption detected before any write; the corrupt file's bytes left unchanged) is proven over core + tt by features/integrity_check.feature, and the full detect→recover round-trip by features/backup_recovery.feature; the on-install behavior is the CHECK INTEGRITY-ON-OPEN runbook.
  4. Automatic backups + retention. new core On launch, if the DB changed since the last backup, write a timestamped copy beside the DB; keep the last N (default 5; §14); expose restore in Settings. Done when a fresh launch leaves a recoverable timestamped backup and retention prunes to N. implemented Store.open calls backupDb (checkpoint-then-copy to <db>.bak-<UTC>, no-op when unchanged, prune to the backup_retention setting); listBackups/backupNow/restoreFromBackup back both surfaces (tt backup ls|now|restore, GUI Settings → Backups). Proven by features/backup_recovery.feature + GOLD backup.schema.json.
  5. Corruption recovery. new core On integrity-check failure, quarantine the corrupt file (.corrupted) and restore from the latest good backup, informing the user; never silently lose data. Done when a corrupted DB is recovered from the latest backup, the bad file is preserved as .corrupted, and the user is told. implemented quarantineAndRecover renames the corrupt main file (and WAL/SHM) to a .corrupted-<ts> sibling and copies the latest backup into place (a RecoveryError when none exists — never an empty start); Store.lastRecovery() surfaces it and the GUI shows a recovery dialog. Proven over core + tt by features/backup_recovery.feature (zero data loss, quarantine sibling present).
  6. Monotonic-time guard. new core Duration arithmetic tolerates wall-clock skew (NTP / manual clock changes); now < start never yields negative or garbage elapsed. Done when a backward clock jump during a running timer yields a sane, non-negative elapsed. implemented core/src/time.ts adds elapsedSeconds(start, now) = max(0, round((now − start) / 1000)), returning 0 when now < start or either parse is NaN — the asymmetric never-negative guard for live/derived elapsed, distinct from the signed secondsBetween raw-span primitive kept for stored start/end span math (§10a). Store.toView routes the open entry's elapsed through it; closed entries keep the stored-end span, still max(0,…)-clamped. Both surfaces render derived elapsed via core, so the guard flows through automatically. PROP prop/monotonic.test.ts.
  7. app_state durability. new core State needed for reconciliation / scheduling (last-seen timestamp, check-in schedule) is persisted in the same transaction as the write that changes it, so it can never drift from the entry it describes. Done when a crash between an entry write and its schedule update is impossible — they commit together. implemented The canonical keys (checkin_state, last_seen_utc) live in @stint/core (checkin.ts CHECKIN_STATE_KEY/LAST_SEEN_KEY) so no surface duplicates the literal. Store.start() seeds the check-in schedule (anchored at the new open entry's start, via initCheckinState) and stamps last-seen INSIDE the SAME BEGIN IMMEDIATE as the entry insert; Store.stop() clears the schedule and stamps last-seen INSIDE the same transaction as the close — so a crash mid-transition rolls back to a consistent state and the schedule can never point at a stale entry. A private writeAppStateTx is the in-transaction primitive; the typed checkinState() reader, setCheckinState(state, now) (schedule advance + heartbeat as one durable unit, the GUI tick's fire path), and recordLastSeen(now) (the standalone heartbeat, its own short tx) own the GUI-facing writes; the GUI no longer holds the raw keys or its own load/save helpers. Backfill (add) creates a CLOSED entry and deliberately does NOT touch the schedule. Proven by PROP core/test/prop/appstate.test.ts (over random start/stop/add/resume/gap sequences on a REAL on-disk DB reopened between ops: after every committed op the persisted schedule agrees with the entry table — open ⇒ present & anchored at start, idle ⇒ absent; a transition forced to throw mid-tx leaves app_state byte-identical to before, proving schedule+entry commit-or-rollback as one unit; a reopened Store yields the committed checkinState(), proving durability across connections) and BDD features/app_state_durability.feature (run TWICE over core + tt: starting durably persists the schedule anchored at start and it survives a reopen; stopping clears it atomically with the close — both true together after a reopen).
  8. Additive, idempotent migrations. new core The database carries a schema version. Opening a database already at or beyond the current version makes no change; opening an older one applies only additive changes and stamps the version forward, never rewriting or dropping existing rows. There are no down-migrations. Done when opening an up-to-date database is a no-op and opening an older one preserves every existing row byte-for-byte while adding the new structures. implemented
A Appendix — Power/idle APIs

What the process can actually observe

Reference for §10, expressed against Electron's powerMonitor. Sleep/wake are event-backed (live suspend/resume), with a wall-clock-gap fallback on launch; idle is only pollable and fuzzy — which is why §10 builds on the former and excludes the latter. powerMonitor surfaces user-visible suspend/resume, so each sleep is one span and a missed sleep is reconstructed from the wall-clock gap.

Went to sleepYes, live eventpowerMonitor 'suspend' (stamp on receipt, source = event). If missed (app not running), inferred after the fact from a wall-clock gap, source = gap.
Woke from sleepYes, live eventpowerMonitor 'resume'. A wake missed while the app was closed is recovered on launch by comparing the wall-clock gap against the open entry.
Became idlePoll only, fuzzyNo event. Poll powerMonitor.getSystemIdleTime() (seconds since last input); infer start as now − idle. The OS "idle" definition is OS-provided and version-variable. Excluded from v1 per §10.
Stopped being idleInferredIdle time resets to 0 on input; powerMonitor 'unlock-screen' as a coarse proxy.
