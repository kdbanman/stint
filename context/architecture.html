01 The shape

One core, one file, two thin shells

There are three kinds of part: the Core, the Database, and a surface. A surface — the Electron app or the tt CLI — imports @stint/core and reads/writes the SQLite file; neither is privileged. Surfaces hold presentation, OS integration, and argument parsing; the Core holds the schema, every transition, every invariant, reporting, rounding, and durability. Logic is never duplicated across surfaces.

  ┌─────────────────────────┐        ┌─────────────────────────┐
  │   Electron GUI (gui)    │        │      tt  CLI  (cli)     │
  │  tray · window · IPC ·  │        │  commander · --json ·   │
  │  powerMonitor · hotkey  │        │  human tables           │
  └───────────┬─────────────┘        └────────────┬────────────┘
              └───────────────┬───────────────────┘  imports
                  ┌─────────────▼─────────────┐
                  │      @stint/core          │  the only place logic lives
                  └─────────────┬─────────────┘
                       │  node:sqlite · single writer
                  ┌─────────────▼─────────────┐
                  │   timetracker.sqlite      │  one file · WAL · source of truth
                  └───────────────────────────┘
        ▲ both surfaces watch the file (fs.watch) and refresh on change

Figure 1 — process & dependency topology

Dependencies point inward only. @stint/core depends on no surface and no UI or CLI framework — Node built-ins alone. The CLI adds commander; the GUI adds Electron. That direction is why parity is structural, not a promise (§04 R01, §17 R08).

02 The keystone

A running timer is an open row

A running timer is the single entry row whose end_utc is NULL — a row state, not a process (concept §02). Three consequences shape the rest:

  • Elapsed is derived, never stored. Whoever looks computes now − start; the count-up is display only (§05).
  • Surfaces don't synchronize. They are two views of one row; state lives in the file, not in either process (§06).
  • One invariant carries the weight: at most one open entry. Enforce it and start-while-running, cross-surface stop, and crash-mid-timer are all safe (§04).
03 Core module map

What @stint/core owns

One package, two layers: the Store — the transactional façade and the only writer — and a set of pure helpers the surfaces reuse so no rule (rounding, grouping, label-joining, time math) is re-derived in a renderer or formatter. Responsibilities by module:

04 Storage & the write path

One file, one writer, atomic transitions

The Database is a single SQLite file in WAL mode — concurrent readers, one writer — at the OS app-data path, overridable by TT_DB. Schema and pragmas are §13; the hardening rules are §20. Two things are architectural: the write path, and the invariant it protects.

Atomic stop-then-start

Starting a timer while one runs is one transaction, so it is safe regardless of which surface triggers it or how fast start/stop arrive (§04 R04).

  start(...)  // switch is the same transaction, surfaced as one action
    BEGIN IMMEDIATE                            // take the write lock; busy-timeout lets surfaces cooperate
      close any open entry  (end_utc = now)
      insert the new open entry
      advance check-in schedule + last-seen  // app_state, same tx (§20 R07)
    COMMIT                                     // all-or-nothing; a crash mid-transition rolls back clean

Figure 2 — the write path. Schedule state commits with the entry it describes.

At most one open entry

The core invariant is enforced twice — in the Store's transaction and, independently, by a partial unique index at the storage layer — so a second open row is rejected even if the core path were bypassed. The index construction and its NULL-distinctness rationale are in §20 R02.

05 Derived elapsed

Time is computed, not kept

No process holds or increments a timer (§04 R03). Open-entry elapsed is now − start, clamped to non-negative so a backward wall-clock jump never yields garbage (§20 R06). Stored start_utc/end_utc are never mutated by rounding or sleep subtraction — both apply to derived duration only, and both reverse (§17 R04). Times are stored UTC, rendered local, so durations are DST-safe (§04 R06).

06 Change propagation

The file is the bus

There is no IPC between surfaces. Each watches the file (and its -wal) with fs.watch and refreshes its read model on change, so a write from one appears in the other within a heartbeat (§04 R05). Correctness does not depend on it — the next read is authoritative because state lives only in the file — so it is explicitly not a core requirement. The GUI's internal renderer⇄main IPC (§08) is within one surface and is a separate mechanism.

07 Durability & integrity

One anchor: DB open

Every durability rule is core and specified in §20. They share one architectural property: all run at open, before any write, so a misconfigured or corrupt file never reaches normal operation.

  open
    1. assert pragmas     WAL · foreign_keys · busy_timeout · synchronous=FULL   (§20 R01)
    2. integrity check    on failure → quarantine .corrupted + restore backup    (§20 R03/R05)
    3. backup-on-launch   if changed since last, timestamped copy + prune        (§20 R04)
    4. migrate           to current schema version (additive)
       → only now are writes allowed

Figure 3 — the open sequence

Recovery never loses data: the corrupt file is preserved as a .corrupted sibling and the latest good backup is copied into place. Backup/restore is reachable from both surfaces. Schedule state in app_state commits in the same transaction as the entry that changes it (§20 R07), so the two can't drift.

08 The two surfaces

Equal front doors, parity enforced

Both surfaces are thin shells over the Core (§11 CLI, §12 GUI), and parity is the rule: every capability is reachable from both (§17 R08).

The CLI is a commander program; each command calls a Store method, with human tables by default and --json on reads for scripting. The GUI is two Electron processes: the main process owns the Store, the OS integration (tray, hotkey, powerMonitor sleep detection, check-in notifications, file-watch refresh), and the IPC handlers; the renderer is sandboxed web tech that reaches the main process only through a context bridge.

  renderer  →  window.stint.<channel>(payload)   — its only door out of the sandbox
  preload   contextBridge  →  ipcRenderer.invoke(channel, …)
  main      ipcMain.handle(channel)  →  Store method / pure helper
            └ broadcasts 'changed' → renderer refresh (fs.watch does the same cross-surface)

Figure 4 — GUI internal IPC

The renderer-reachable channels form the parity matrix: each maps to an equivalent tt path in acceptance/criteria/parity-matrix.json, asserted in CI. Three capabilities have no CLI twin and sit outside the matrix — the tray, the global hotkey, and Software Update — so update rides a separate update:* IPC namespace rather than the parity channels.

09 Platform & stack

macOS + Linux, no native build

The normative list is §15; the layers:

LayerBuilt on
CoreTypeScript; persistence via Node's built-in node:sqlite — no native build step.
CLINode executable tt on commander, importing the Core.
GUIElectron — tray + popover and a main window; web-tech renderer; system theme and accent.
OSfs.watch, powerMonitor, globalShortcut, Notification (see Appendix A).

node:sqlite is the stack's sharpest dependency: it exists only on Node 22.5+ and must load in both runtimes — the system Node behind tt and Electron's bundled Node — and is stability-experimental, so it is pinned and confirmed against both.

10 Packaging & update

Self-updating from GitHub Releases

Specified in §19. CI builds a macOS bundle and a Linux artifact — no Windows target — and every merge to main publishes a GitHub Release, so the public repo is the distribution backend. One installer per platform lands both the GUI and tt in one step; both report one shared YYYY.M.D[.N] version.

The in-app updater makes the app's only outbound request — a user-initiated query to the GitHub Releases API over Electron's net, sending no data (§17 R09) — then guides the install. An update never touches the Database, which is why it lives entirely outside the Core and the parity matrix (§16).
