How Stint is shaped: one shared core package, one SQLite file, and two equal surfaces over it. This is the explanatory companion to the spec — it gathers the architecture that prd.html spreads across §04, §13, §15, §19, and §20 into one place, adds the diagrams and the module map, and follows from the one idea in concept.html.

What this doc is — and isn't This is a non-normative architecture overview. The requirements and their live implementation status stay in prd.html (§04 Architecture, §13 Storage & schema, §15 Platform & tech, §19 Packaging, §20 Durability) — that is still the single source of truth for what must hold. This doc explains how the pieces fit and is safe to read first. Where a section here states a rule, it links to the PRD requirement that owns it. The vocabulary (surface, Core, Database) is defined in glossary.html §06. The build / CI / verification architecture is a separate concern — see process.html and issue #19.

01 The shape

One core, one file, two thin shells

Everything in Stint is one of three things: the Core, the Database, or a surface. A surface is a way in — the Electron tray app or the tt CLI. Both import the same @stint/core package and speak to the same SQLite file; neither is privileged, and there is no duplicated logic. The surfaces are deliberately thin: presentation, OS integration, and argument parsing only. Every rule that matters — the schema, every state transition, every invariant, reporting, rounding, durability — lives in the Core.

  │  tray · window · IPC    │        │  commander · --json     │

Figure 1 — process & dependency topology. Two surfaces, one core, one file.

The dependency arrows only ever point inward. @stint/core depends on no surface and on no UI or CLI framework — only Node built-ins (node:sqlite, node:fs, node:os, node:path, node:crypto). The CLI adds commander; the GUI adds Electron. That direction is what makes parity structural rather than a promise (prd §04 R01, §17 R08).

02 The keystone

A running timer is just an open row

The architecture rests on one reframing, stated in full in concept.html §02: there is no timer object. A running timer is simply the single entry row whose end_utc is NULL. "Running" is a row state, not a process.

Three consequences shape everything downstream:

  • Elapsed is derived, never stored. Whoever looks computes now − start. The tray's per-second count-up is pure display. Close the lid for an hour and the number is simply correct on read, because it was never being kept anywhere to drift (§05).
  • The surfaces can't disagree. They are two views of one row, not two systems that must be synchronized. State lives in the file, not in either process (§06).
  • One invariant carries the weight: at most one entry is open at any time. Enforce that one rule and "start while running", "stop from the other surface", and "crash mid-timer" all become safe (§04).
03 The shared core

What @stint/core owns, module by module

The Core is a TypeScript package whose public API is a barrel export (index.ts): the Store class (the transactional façade over the DB) plus a set of pure helpers (time math, reporting, list-building, export, label-joining) that the surfaces reuse so no rule is re-implemented in a renderer or a CLI formatter. The PRD describes the Core as a whole (§04, §15); this is the file-level map that the PRD only hints at in its status notes.

ModuleOwns
db.tsOpens the SQLite DB via node:sqlite; defines the v3 schema; asserts durability pragmas and the one_open_entry_idx partial unique index + triggers on every open; runs migrations. openDb, SCHEMA_VERSION.
store.tsThe Store class — the only writer. Atomic transitions (start/stop/resume/switch/add/edit/split/merge/remove) under BEGIN IMMEDIATE; favorites & saved-report CRUD; reporting and list reads; backup/recovery delegation. The façade both surfaces call.
types.tsDomain types — Entry/EntryView (endUtc null ⇒ running), Client, Project, Tag, Favorite, SleepSpan, Status, WriteResult, Warning.
time.tsTime parsing/formatting. parseTime (relative -90m, clock 14:30, ISO), toUtc, signed secondsBetween (stored spans), and elapsedSeconds = max(0, …) — the monotonic guard for live elapsed (§20 R06).
report.tsbuildReport (group by client/project/day/tag, billable filter), roundSeconds (nearest, display-only), and the one overlap rule — spansOverlap/detectOverlaps/describeOverlaps — shared by write-time checks and report scans.

Why the pure helpers matterThe split between the Store (stateful, owns the DB) and the pure helpers (time, report, entrylist, export, label) is deliberate: the helpers are what let the GUI renderer and the CLI formatter present the same numbers without re-deriving them. A renderer that re-rounded, re-grouped, or re-joined would be a parity bug waiting to happen; instead it paints what the Core returns.
04 Storage & the write path

One file, one writer, atomic transitions

The Database is a single SQLite file (timetracker.sqlite) in WAL mode — concurrent readers, a single writer — at the OS app-data path, overridable by TT_DB. The full schema and pragmas are in prd §13; the hardening rules are §20. The architectural point is the write path and the one invariant it protects.

Atomic stop-then-start

The trickiest transition — starting a timer while one runs — is one transaction, so it is safe no matter which surface triggers it and no matter how fast start/stop are issued (§04 R04, §16).

  start(...)            // also: switch = the same, surfaced as one action
    BEGIN IMMEDIATE     // take the write lock now; busy-timeout lets surfaces cooperate
      UPDATE entry SET end_utc = :now WHERE end_utc IS NULL   -- close any open row
      INSERT INTO entry (start_utc, …) VALUES (:now, …)       -- open the new row
      UPDATE app_state … (check-in schedule + last-seen)      -- §20 R07, same tx
    COMMIT               // all-or-nothing: a crash mid-transition rolls back clean

Figure 2 — the write path. Schedule state commits in the same transaction as the entry it describes.

At most one open entry — defense in depth

The core invariant is enforced twice: in the Store's transactional logic, and at the storage layer by a partial unique index, so a second open row is rejected even if the core path were bypassed (§20 R02).

CREATE UNIQUE INDEX one_open_entry_idx ON entry((1)) WHERE end_utc IS NULL;

The index is over the constant (1), not over end_utc, on purpose: every open row has end_utc = NULL, and SQLite treats NULLs as distinct in a unique index — so an entry(end_utc) index would permit unlimited open rows. Collapsing all open rows onto the value 1 makes a second one collide, while closed rows (end_utc NOT NULL) stay out of the partial index entirely.

05 Derived elapsed

Time is computed, not kept

No process holds or increments a timer (§04 R03). Elapsed for the open entry is now − start, computed on demand by core/time.ts elapsedSeconds, which clamps to max(0, …) so a backward wall-clock jump (NTP, manual change) never yields a negative or garbage value (§20 R06). Stored truth — start_utc, end_utc — is never mutated by rounding or sleep subtraction; both apply only to derived billable duration and are reversible (§17 R04). Times are stored UTC and rendered in the local zone, so durations are DST-safe (§04 R06).

06 Change propagation

The file is the message bus

There is no IPC between surfaces. Each surface watches the SQLite file (and its -wal) with fs.watch and refreshes its read model when the file changes, so a write from one appears in the other within a heartbeat (§04 R05). The tray's 1-second display tick is independent of this — it just re-derives now − start. This is a consistency/UX affordance, explicitly not a core requirement: the data is correct on the next read regardless, because state lives only in the file.

NoteThe GUI's own internal IPC (renderer ⇄ main, §08) is a different thing entirely — that is within one surface. Cross-surface coordination is purely the shared file.

07 Durability & integrity

Committed time survives

Every durability rule is core and specified in full in prd §20. Architecturally they form one layered defense, all anchored at DB open in core (db.ts + backup.ts), before any write:

  openDb / Store.open
    1. assert pragmas      WAL · foreign_keys=ON · busy_timeout · synchronous=FULL   (§20 R01)
    2. integrity check     PRAGMA quick_check                                       (§20 R03)
         └─ on failure →  quarantine .corrupted + restore latest backup           (§20 R05)
    3. backup-on-launch    if changed since last backup, timestamped copy + prune    (§20 R04)
    4. migrate            bring schema to SCHEMA_VERSION (additive)
       then, and only then, writes are allowed

Figure 3 — the open sequence. A corrupt file is detected and recovered before a single write occurs.

  • Pragmas asserted, not assumed — set in dependency order, read back, retried once, else the open is refused (a corrupt-or-misconfigured DB never falls through to normal operation).
  • Backups + retention — timestamped sibling copies, pruned to backup_retention (default 5); restore is reachable from both surfaces (tt backup … ↔ Settings → Backups).
  • Recovery never loses data — the corrupt file is preserved as a .corrupted-<ts> sibling and the latest good backup is copied into place; the user is told.
  • Schedule state is transactional — app_state (check-in schedule, last-seen) commits in the same transaction as the entry write it describes (§20 R07), so it can never drift from that entry.
08 The two surfaces

Equal front doors, parity enforced

Both surfaces are thin shells over the Core (prd §11 for the CLI, §12 for the GUI). Parity is the rule: every GUI capability is reachable from tt, and every tt capability is reachable from the GUI (§17 R08).

The tt CLI

A Node executable built on commander. program.ts declares the whole command surface; bin.ts opens a Store and maps errors to clean exit codes; serialize.ts emits the --json scripting contract (validated against the JSON schemas in acceptance/); format.ts renders human tables. No business logic — every command calls a Store method.

The Electron GUI

Two processes. The main process owns the Store, the OS integration (tray + popover, global hotkey, powerMonitor sleep detection, check-in Notifications, file-watch refresh), and the IPC handlers. The renderer is plain web tech and never touches Node, the filesystem, or the DB — it reaches the main process only through a context-bridge.

  renderer (app.js, editor.js, reports.js, …)
        │  window.stint.<channel>(payload)        — only door out of the sandbox
        ▼
  preload.ts  contextBridge  → ipcRenderer.invoke(channel, payload)
        │
        ▼
  main.ts  ipcMain.handle(channel, …)  → Store method / pure helper
        │                                          (uistate, start, toggle, …)
        ▼   broadcasts 'changed' → renderer refreshes; fs.watch does the same cross-surface
  @stint/core  ·  SQLite

Figure 4 — GUI internal IPC. The renderer is sandboxed; window.stint.* is its only reach.

The channel set is the parity matrix: each renderer-reachable channel (ipc.ts CHANNELS) is mapped to an equivalent tt path in acceptance/criteria/parity-matrix.json, asserted in CI. The GUI keeps its main-process logic in pure, Electron-free helper modules (uistate, start, toggle, timerview, liveview, favorites, reportview, backupview, tags, confirm, checkin-actions) so they unit-test without an Electron host.

The deliberate exceptionA few GUI capabilities have no tt twin and are intentionally outside the parity matrix: the tray, the global hotkey, and Software Update (update.ts, §10). Update rides a separate update:* IPC namespace (window.stint.update.*) so the parity assertion stays clean — a CLI install is updated by its package manager, not by the app.
09 Platform & tech stack

macOS + Linux, shared-core, no native build

The normative list is prd §15; this is the shape of it.

LayerBuilt on
CoreTypeScript package @stint/core; persistence via Node's built-in node:sqlite — no native build step.
CLINode executable tt on commander, --json everywhere, importing the Core.
GUIElectron — Tray + popover BrowserWindow, a main BrowserWindow for the shell; web-tech renderer; system light/dark, system type, accent on the primary action only.
OS integrationfs.watch (file-watch), powerMonitor suspend/resume + wall-clock-gap reconciliation (sleep), globalShortcut (hotkey), Notification (check-ins). See Appendix A.
The node:sqlite runtime constraintnode:sqlite exists only on Node 22.5+ and runs in two distinct runtimes — the system Node behind tt and Electron's own bundled Node. v1 therefore requires system Node ≥ 22.5 and an Electron whose bundled Node is ≥ 22.5 with node:sqlite loadable in the packaged main process (it is a stability-experimental module — confirm against the pinned versions). This is the single sharpest platform dependency in the stack.
10 Packaging & update

Installed like a real app, self-updating from GitHub Releases

Specified in full in prd §19. Architecturally: CI builds a macOS bundle and a Linux artifact (AppImage / .deb) — no Windows target — and every merge to main publishes a GitHub Release, so the public repo is the distribution backend. One installer per platform lands both the GUI (Applications / app launcher) and tt (on PATH) in one step. The version is a single shared APP_VERSION (core/version.ts) stamped YYYY.M.D[.N], reported identically by the app and tt.

The in-app updater (gui/update.ts) makes the app's only outbound request — a user-initiated query to the GitHub Releases API over Electron's net (never node:https/fetch, never sending data, §17 R09) — then downloads the artifact and guides the install. Crucially, an update never touches the Database: a mid-timer update leaves the open entry intact (§16). That isolation is why update can live entirely outside the Core and outside the parity matrix.

11 Build & process

A separate kind of architecture

How Stint is built, verified, and released — the CI gates, the acceptance apparatus, the guards that keep green meaningful — is a distinct concern from the runtime architecture above. It has its own (seed) home in process.html, with the full SDLC specification tracked in issue #19. The acceptance machinery that proves the runtime architecture holds lives under acceptance/ (criteria + evidence) and features/ (Gherkin run against both surfaces); the PRD→proof map is acceptance/criteria/COVERAGE.md.
