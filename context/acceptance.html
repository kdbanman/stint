Acceptance Criteria Strategy · v1 · method set + coverage map + worked samples

No single notation can verify the whole PRD well. This proposes a small set of five complementary acceptance-criteria methods, routes every PRD section to the right one, and shows one worked sample per method in the project's ubiquitous language.

Read with concept.html (the why) and prd.html (the what); this is the how we'll know it's true. It is a strategy plus samples, not the full AC catalogue — the samples fix the house style for each method so the bulk can be authored against it. The criteria bind to @stint/core, tt, and the GUI. Targets are macOS and Linux only; Windows is out of scope across code, build, docs, and runbooks.

The five methods gate the build; recordings do notExecutable acceptance (BDD · PROP · GOLD · JUDGE · MANUAL) is the system that decides whether a requirement is met. The screen recordings described in §12 are QA evidence gathered at the tail of the workflow, not a sixth method — they demonstrate, they do not gate. The two reviews (§13) are how the AC apparatus is enforced, distinct from authoring AC.

01 Why a set, not one method

The PRD has five kinds of claim

The requirements in prd.html are not uniform, and the temptation to pick one notation (all-Gherkin, or all-unit-tests) breaks on that variety. Sorting the PRD by what kind of thing is being asserted yields five distinct shapes, each with a notation that fits and several that don't:

  • Pure, deterministic logic — the one-open-entry invariant, derived elapsed, billable-duration arithmetic, rounding maths, split/merge, overlap detection, UTC/DST math, monotonic-time skew guard (§20 R06). Truths over all inputs; best caught by properties and example tables, not prose scenarios.
  • Observable behavioural flows — start/stop/resume/backfill, resume from a favorite, save and run a report, "start while open stops the open one", overlap warns-not-blocks, backup-on-launch and corruption recovery. Given/When/Then in domain language reads as both spec and test: Cucumber's home turf.
  • A machine contract — tt's exit codes, table output, and --json shape for entries, favorites, and saved reports; CSV columns; the export schema; the favorite and report table shapes. Verified by golden/snapshot comparison and schema validation, where the artefact is the criterion.
  • Subjective & emergent qualities — "feels at home on the desktop," monochrome, accent on the primary action only, one clickability convention (§15), the visual time-range picker's drag affordances, empty states that instruct, flags visible in context. No assertion captures these; a rubric judged by an LLM (or human) over driven-UI screenshots can.
  • Physical / OS reality — real sleep/wake via powerMonitor, wall-clock-gap reconciliation on launch, notifications firing on a wall-clock schedule, "makes no network connections," the single installer placing app + tt, the in-app updater completing a guided install, a real corrupt DB detected and recovered on open. These resist CI; they need scripted-but-supervised checks on real hardware.

So the method set is chosen to cover those five shapes with minimal overlap — each method earns its place by being the best (often the only) fit for one shape, while a couple double as secondary nets for others.

The toolkit, and what each is for

MANUAL Scripted manual / supervised checks. A numbered runbook on real hardware (macOS / Linux), using the OS's sleep controls (systemctl suspend / rtcwake on Linux, pmset on macOS), a network monitor, and a real install of the artifacts; an operator (human or agent) confirms each step and records evidence. Physical/OS reality: real sleep/wake + wall-clock-gap reconcile, notification cadence across relaunch, the no-network guarantee, the tray's on-screen count-up and global-hotkey toggle, the single installer (app + tt on PATH), the in-app guided update, and real corruption detection + backup recovery on open. Regression speed and scale (slow, can't run on every commit); reserved for what genuinely can't be faked in CI.
Cucumber, specificallyGherkin is adopted for the behavioural layer only — it is not the whole strategy. Using it where input spaces explode (rounding, time parsing) or where output must match byte-for-byte (CLI/JSON) would be a misuse; those go to PROP and GOLD. One feature can drive either @stint/core directly or the tt executable, which is how the "full parity" claim (§17 R8, R14) gets tested without writing every scenario twice — now including parity for favorites and saved reports.
03 How to route a requirement

A decision rule for any new criterion

So the set stays disciplined as the catalogue grows, route each requirement by the first rule that matches:

  1. Does it touch money-affecting maths or a stored invariant (durations, rounding, exclusion, one-open-entry, immutability, DB-open invariants, monotonic-time)? → PROP for the law, GOLD/example table for specific values. Never let JUDGE be the only gate here.
  2. Is the deliverable exact text or a data shape (tt output, exit code, --json, CSV columns, the favorite/report schema, the version stamp)? → GOLD + schema.
  3. Is it a user-observable flow or decided edge case a stakeholder should be able to read (resume-from-favorite, save/run report, backup & recovery)? → BDD, stepped on @stint/core or tt.
  4. Is it visual, subjective, or emergent in the GUI (sidebar shell, popover, Timer view, time-range picker, clickability convention)? → JUDGE over agentic exercising, with plain Playwright for any deterministic sub-fact in the renderer, and electronApp.evaluate() for main-process facts (e.g. single-click opens the popover only; no dropdown context menu).
  5. Does it depend on real OS/hardware behaviour over wall-clock time, or on a real install / update / on-disk file? → MANUAL runbook, with as much scripted setup as possible.

Two requirements often split across methods: §17 R12 (backups & recovery) becomes PROP/GOLD for the backup-file shape and retention algebra, BDD for the detect-then-restore flow against a planted corrupt fixture, and MANUAL for a real corrupt DB recovered on open. That decomposition is the point of having a set.

06 Sample — Gherkin / BDD

BDD behavioural flows in ubiquitous language

One .feature can be stepped onto @stint/core directly or onto the tt executable. Note the deliberate vocabulary: open entry, start, stop, overlap, warned-not-blocked — all straight from the glossary.

Feature: One open entry across surfaces
  # PRD §05 R1, §16 "start while running", §17 R2 — the keystone invariant,
  # observed as behaviour. Steps bind to @stint/core; @cli re-runs them through tt.

  Background:
    Given an empty database
    And a client "Client A" with project "API"

  Scenario: Starting while an entry is open closes the first one
    Given I start an entry "auth refactor" for "Client A" / "API" at 09:00
    When  I start an entry "code review" at 10:30
    Then  exactly one entry is open
    And   the entry "auth refactor" is closed with end 10:30
    And   the open entry is "code review"

  @cli
  Scenario: A timer started in tt is visible to status with no disagreement
    When  I run `tt start "auth refactor" --client "Client A" --project API`
    Then  the exit code is 0
    And   `tt status` reports an open entry "auth refactor" for "Client A / API"

  Scenario: Backfill that overlaps an existing entry is warned, not blocked
    # PRD §06 R4 — overlap is allowed if meant, and flagged downstream
    Given a closed entry from 09:00 to 11:00
    When  I backfill an entry from 10:00 to 10:30
    Then  the backfill succeeds
    And   a non-blocking overlap warning is surfaced
    And   both entries are flagged "overlapped" in a report covering the day

The new favorites and saved-report flows (§05 R09–R10, §09 R08–R09) are authored the same way; one is worked through in §11.

07 Sample — Property-based

PROP invariants over generated inputs

Where Gherkin pins one path, properties assert a law over thousands of generated ones. These are the criteria guarding the numbers that reach an invoice — and now the core durability hardening of §20.

// PRD §03 invariant + §17 R2 + §20 R02 — at most one open entry, under ANY interleaving
test.prop([fc.array(timerOp())])("at most one open entry after any op sequence", (ops) => {
    const core = openCore({ inMemory: true });
    ops.forEach((op) => core.apply(op));       // start/stop/resume/backfill, any order
    expect(core.openEntries().length).toBeLessThanOrEqual(1);
    // and the partial unique index would reject a second open at the DB level
});

// PRD §03 + §17 R4 — stored truth is immutable under rounding & subtract
test.prop([entryArb(), roundIncArb(), secondsArb()])("rounding and subtract never mutate start/end", (e, inc, sub) => {
    const before = [e.startUtc, e.endUtc];
    report(e, { rounding: inc });              // display lens
    subtractSlept(e, sub);                     // excluded_seconds only
    expect([e.startUtc, e.endUtc]).toEqual(before);
});

// PRD §20 R06 — monotonic-time guard: clock skew never yields negative/garbage elapsed
test.prop([entryArb(), skewMillisArb()])("elapsed is clamped >= 0 under wall-clock skew", (e, skew) => {
    const nowSkewed = e.startUtc + skew;        // skew may be negative (NTP/manual change)
    expect(elapsedAt(e, nowSkewed)).toBeGreaterThanOrEqual(0);
});

// PRD §20 R04 — backup retention keeps exactly the last N (default 5), newest-first
test.prop([fc.array(backupStampArb()), fc.integer({ min: 1, max: 20 })])("retention keeps the newest N", (stamps, n) => {
    const kept = applyRetention(stamps, n);
    expect(kept.length).toBe(Math.min(n, new Set(stamps).size));
    expect(kept).toEqual([...kept].sort((a, b) => b - a));   // newest-first, no older survivor
});

The rapid-start/stop concurrency of §17 R2 also gets a stress variant: many concurrent processes issue start/stop against one WAL database with the busy timeout set, asserting the open-entry count stays ≤ 1 throughout — the property above, proven across processes rather than in one, and now backstopped by the §20 R02 DB-level index rejecting a second open row outright.

09 Sample — LLM-judged + agentic

JUDGE the GUI qualities that resist assertion

An agent drives the GUI through a scripted route (Playwright over the Electron app via Chromium), captures screenshots and the accessibility tree at each step, and an LLM scores the evidence against an explicit rubric. Deterministic facts in the renderer (a Stop button exists and is hittable) are plain Playwright assertions; main-process facts (single-click opens the popover only, no dropdown context menu) are exercised via electronApp.evaluate(). The renderer windows run headless under xvfb-run; the tray icon's own title text has no host there, so its live count-up is confirmed under MANUAL.

Agentic route driver script

  1. Launch with an empty database. Capture the main window — assert the fixed-width sidebar is present and the current view is highlighted. → shell + empty state (§12 R03)
  2. Open the Timer view; create/start a timer with details; wait 3s and re-capture. → live count-up advanced (§12 R14)
  3. Pin the running timer as a favorite; capture the favorites rail; one-click resume from it. → resume-from-favorite (§05 R09–R10)
  4. Open a closed entry's edit form; click the time field to open the time-range picker; drag the rectangle body, then its bottom edge; capture overlapping neighbours. → move/resize, 5-min snap, gray others / yellow overlap (§12 R15)
  5. Open the Reports view; build & save a definition; run it; capture totals + flags. → saved reports in-sidebar (§12 R08)

Judge rubric pass / fail per item

# Each item scored PASS/FAIL with a one-line justification citing the screenshot.
# Any FAIL fails the suite; a sample of PASSes is spot-checked by a human.

SIDEBAR_SHELL    The fixed-width sidebar is present in EVERY captured view and does not
                 reflow on window resize; the current view is highlighted. (§12 R03)

POPOVER_ONLY     A single tray left-click opens the compact popover ONLY — no 3-item
                 dropdown context menu appears. (§12 R01)

FAVORITE_RESUME  The favorites rail shows the pinned template; one click starts a fresh
                 open entry carrying its attributes (new id, not a re-open). (§05 R09-R10, §12 R14)

PICKER_AFFORDANCE  The time-range picker shows a draggable entry rectangle; other entries
                 render gray and overlaps render yellow (warn-only); text entry stays
                 present and authoritative. (§12 R15)

CLICKABILITY     Every clickable text affordance carries the neutral button background;
                 inert text never does; the accent appears on this view's single primary
                 action ONLY. (§15 R-clickability)
GuardrailJUDGE is non-deterministic, so it is never the sole gate on anything that changes a billable number. It owns presentation and discoverability — the sidebar shell, the popover, the Timer view, the picker, the one clickability convention — while the arithmetic underneath is nailed by PROP and GOLD, and the saved-report results are also GOLD-checked. Rubric items are crisp PASS/FAIL claims, not vibes.
10 Sample — Manual / supervised checklist

MANUAL the physical & OS-level guarantees

Real sleep/wake, the no-network promise, a real install/update, and a real corrupt-DB recovery can't be faked in CI. Setup is scripted as far as possible; an operator (human or agent with shell + GUI access on macOS or Linux) confirms each step and attaches evidence.

CHECK SLEEP-SPAN — second-accurate spans + working subtract   (§10a, §17 R5)
  1. tt start "deep work" --client "Client A"
  2. Record start. Sleep the machine:  systemctl suspend  (Linux) / pmset sleepnow (macOS)
  3. Wake after ~120s. Note the suspend/resume times from the system log.
  ✓ The open entry is flagged slept-through; a SleepSpan (source=event) is attached;
    (resume − suspend) matches the powerMonitor delta within 1s.
  ✓ `tt sleep subtract <id>` moves those seconds into excluded_seconds; reversible.

CHECK BACKUP-ON-LAUNCH + CORRUPTION RECOVERY                  (§17 R12, §20 R03–R05)
  1. Make a change, then relaunch.
  ✓ A timestamped backup is written beside the DB; retention keeps the last 5.
  2. Quit. Overwrite the DB file with garbage bytes (planted corruption). Relaunch.
  ✓ The integrity check on open FAILS; the app does NOT write to the corrupt file.
  ✓ The corrupt file is quarantined as `.corrupted`; the latest good backup is restored;
    the user is informed. No entries are lost vs. the pre-corruption state.

CHECK INSTALL + IN-APP UPDATE                               (§17 R13, §19 R02–R04)
  1. Run the single installer (macOS .pkg / Linux install script) on a clean machine.
  ✓ The GUI appears in Applications / the app launcher AND `tt` is on PATH — one step.
  2. With an older version installed and a timer running, Settings → Software Update →
     Check for updates.
  ✓ A newer GitHub release is detected; download + guided install completes (one-time
    Gatekeeper approval on macOS); the database is untouched; the running timer survives.

CHECK NO NETWORK                                             (§17 R9)
  1. Run the app + tt through a full session under a network monitor (lsof -i / ss).
  ✓ Zero outbound connections EXCEPT the explicit, user-initiated update check
    (Settings → Software Update); no telemetry/analytics path exists.

The no-network check keeps a cheap GOLD backstop in CI — scanning the build's dependency tree and source for any networking import or outbound-request code path outside the single, user-initiated update-check module — so a regression is caught fast even though live-traffic confirmation stays manual.

    And a closed entry "API deep work" for "Client A" / "API", billable

    When  I run `tt fav add deep-api --from-entry 1`
    And   I run `tt fav start deep-api`
    Then  the exit code is 0
    And   `tt status` reports an open entry "API deep work" for "Client A / API"

  Scenario: Rename and unpin

And the saved-report definition (§09 R08) carries a GOLD schema note: a saved report stores a relative range_spec (e.g. this-week) or an absolute {from_utc, to_utc}, plus group_by, filters, and rounding. Storing the spec, not the resolved range, is the criterion — tt report run must re-resolve this-week against current data each time. The schema in §08 is the approved fixture; a BDD scenario then proves a definition saved on Monday yields a different week's totals when run the following Monday, confirming the relative spec is resolved at run time and not frozen at save time.

12 Screen-recording QA evidence

Demonstrations on the PR — not one of the five AC methods

Not executable acceptance · does not gate the build

Screen recordings are QA evidence, gathered at the tail of the workflow — after plan → implement → verify (the executable AC above) → the two reviews (§13) are complete. They are not a sixth acceptance method: they demonstrate requirements being exercised for a human reviewer on the PR, but they do not decide pass/fail and do not gate the build the way BDD/PROP/GOLD/JUDGE/MANUAL do. A requirement is "met" because its mapped executable AC passes with real evidence — the recording is a courtesy demonstration layered on top, never the proof.

This distinction is deliberate and load-bearing: the five methods are the contract; recordings are the show-and-tell. A green recording with a failing or stubbed AC does not pass; a passing AC with no recording still passes the gate (though the workflow requires the recording before merge as PR hygiene).

Recording scope

  • All core-flow GUI requirements — every GUI requirement marked core (●): §05 R01/R02/R05 (start / stop / add via the GUI), §12 R05/R07/R08-export/R13/R14.
  • All changed / new GUI requirements — every Rec ▶ row in the mapping: §12 R01, R03, R04, R08, R14, R15, and §05 R09/R10 (favorites).
  • All code-change-adjacent requirements — any requirement whose implementation touched code in this transition and is demonstrable in the GUI, including the §19 update flow (R03/R04) and §20 backup/restore (R04/R05).

Where they live

Recordings are attached to / linked from the PR body as QA evidence under acceptance/evidence/recordings/, indexed by requirement id. The binaries themselves are git-ignored and linked from the PR; only the index is tracked. They are captured on a real macOS or Linux desktop session — which is also why several of the same requirements have a MANUAL runbook: the runbook is the executable gate, the recording is its visible artefact.

13 The two review stages

How the AC apparatus is enforced — distinct from authoring it

Authoring AC (everything above) says how a requirement would be proven. The two reviews, run after implementation and AC verification, are how that proof is enforced before the PR can merge. Each produces feedback that loops back into an improvement pass, and only then are recordings gathered.

AC-evidence-sufficiency review adversarial completeness critic

A per-requirement critic that defaults to insufficient and only flips to sufficient when the requirement is, all three: implemented, covered by its mapped executable AC (BDD/PROP/GOLD/JUDGE/MANUAL as the coverage map dictates), and reflected in regenerated evidence. It treats a stub, a skipped test, or a placeholder as a fail. Its question is not "is there a test?" but "is there real, passing evidence for the mapped method, today?"

Code-quality & architecture review Pocock lineage

Adapted from Matt Pocock's improve-codebase-architecture method. It hunts:

  • Shallow modules — interfaces nearly as complex as their implementation; prefer deep modules (simple surface over substantial behaviour).
  • Leaky seams — abstractions that leak their internals across the boundary (e.g. a favorite or saved-report API forcing callers to know the table shape).
  • Poor locality / cognitive bounce — logic for one change scattered so a reader must hop between files to follow it.
  • The deletion test — could this code, abstraction, or option be deleted with the requirement still met? If yes, it should be.

Findings are rated Strong / Worth exploring / Speculative, with one top recommendation. The review feeds an improvement pass and must not regress AC — a cleaner architecture that breaks a passing acceptance criterion is rejected. Together these two reviews enforce the apparatus: the first guards that every requirement is genuinely proven, the second guards that it was built in a way worth keeping. The human gate remains the PR merge.

Residual risk we accept

  • Sleep reconciliation is wall-clock-approximate. A sleep missed while the app was closed is reconstructed from the wall-clock gap (source=gap), which bounds the dead time but can't tell true sleep from app-closed time — so gap-sourced spans are flagged for review and never auto-applied. Verified on real hardware before release, not per commit.
  • Install & update are MANUAL by nature. The single installer, the in-app guided update, and publish-on-merge run on real macOS/Linux machines and against the live GitHub Releases backend; the only CI-fast piece is the GOLD version-stamp pattern. A real Gatekeeper-approval flow is supervised, not scripted end-to-end.
  • Corruption recovery is sampled, not exhaustive. Backup/retention algebra is PROP-covered and the detect-then-restore flow is BDD-covered against planted fixtures, but the space of real corruption modes is open-ended; a genuinely corrupt file is recovered under MANUAL for the modes we can produce.
  • Tray and global hotkey aren't exercised end-to-end in CI. Headless xvfb-run has no system-tray host and can't deliver an OS-global hotkey, so JUDGE covers the renderer windows and main-process state (electronApp.evaluate()) while the tray's live count-up and a real hotkey press are confirmed under MANUAL.
  • JUDGE is advisory on look-and-feel. "Feels like a quiet desktop app," the clickability convention, and the picker's feel are scored by rubric and spot-checked by a human; they gate presentation but never a billable number.
  • Recordings demonstrate, they don't prove. The screen recordings (§12) are PR evidence, not a gate; the gate is always the executable AC. A recording that looks right over a stubbed or skipped AC does not pass the sufficiency review.

Net: the money-affecting and data-loss core (invariants, durations, rounding, exclusion, exports, DB-open hardening, backups) reaches near-complete deterministic coverage; the irreducibly physical (sleep, install, update, real corruption) and the irreducibly subjective (GUI feel, clickability, the picker) are covered by the right weaker instrument and named here as known residue rather than papered over.
