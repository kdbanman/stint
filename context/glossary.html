Domain Language · Ubiquitous Glossary · v1 · macOS + Linux · single user

The words the codebase, the CLI, the UI copy, and these documents must all use the same way. One term per concept; the rejected synonyms are listed so they can be caught in review. Pair this with concept.html (the why) and prd.html (the what).

This is the project's ubiquitous language: a glossary of the non-obvious, Stint-specific terms plus the key relationships already decided. It is deliberately thin — it skips generic programming words and records only vocabulary where drift would cost us. When a term here is sharpened or a new one earns its place, this file is where it lands.

01 Core entities

The nouns of the model

The persisted records. These are the entity names used in schema, Core, and conversation alike.

TermDefinitionAliases to avoid
Entry The core record: one block of tracked or backfilled time, with an optional client, project, description, and tags. An entry with no end is open (the live timer); otherwise it is closed. Times are stored UTC, to the second. This is the only word for a tracked block. stint, session, span, spell, log, block
Stint The product name only — the cross-platform desktop app and the idea behind it. Never an entity. A tracked block of time is an Entry, not "a stint." The CLI's two-letter shorthand is tt. stint = a tracked Entry
Client A customer the user bills. Top of the reference hierarchy. Carries no rate (money lives downstream). Archivable, not deleted, once referenced. See §09 — do not use "client" for the app or CLI. customer, account
Project An optional refinement under exactly one client. Selecting a project implies its client — the project's client is authoritative. Archivable. task, job, matter
Tag A flat, reusable, cross-cutting label applied many-per-entry, used to filter and group across clients (e.g. all "meetings"). Created on the fly when first applied. Identity is case-insensitive and whitespace-trimmed: a given name resolves to exactly one tag row (the first-seen spelling is preserved), empty or whitespace-only names are dropped, and comma-separated input splits into discrete tags. Editing an entry's tags is a minimal diff under that identity — unchanged tags are left untouched and re-applying one is idempotent. label, category
Favorite A named, pinned timer template capturing description, client, project, tags, and billable flag — created ("pinned") from any entry or the running timer. Stored in its own favorite table; it is not an entry and holds no times. Resuming a favorite (§02) starts a fresh entry from its template. The canonical pair: a favorite is "pinned," its instantiation is "resume from favorite." preset, quick-start, shortcut, saved timer
Saved report A named, persistent report definition: a preset of {range-spec, group-by, filters, rounding}. Stored in its own report table; the range is held as a relative spec (e.g. this-week) or an absolute range. The definition, not its output — running it (§03 "report run") resolves it against current data. See §09 — "report" now carries two senses. report (unqualified — see §09), view, filter set, query
Range spec The stored form of a Saved report's date range, discriminated by kind: either a relative range preset (re-resolved on every run) or an absolute fixed UTC window passed straight through. The durable description of "which days," distinct from the concrete bounds a report run resolves it into. date filter, period, timeframe
Range preset One of the five relative windows — today, week (this week), last-week, month (this month), last-month — each resolved against the user's local calendar (honoring the Week-start setting) into a half-open [from, to) UTC window at run time. The relative arm of a range spec. quick range, shortcut, date range (unqualified)
Setting A key/value pair of app configuration (rounding, week start, check-in cadence, hotkey, …). Shared by both surfaces through the same database. preference, config flag
SleepSpan A single sleep→wake cycle that occurred while an entry was open, stored in its own sleep_span table against that entry (many per entry; sleep_utc/wake_utc). Each carries a source: event — authoritative, captured live from the OS power monitor; gap — a wall-clock-gap suspicion recovered on launch (see Reconcile, §05); unknown — provenance unrecorded. The raw evidence behind Slept-through, and the only v1 origin of excluded time once subtracted. sleep (unqualified), downtime, idle span, away time
02 Entry lifecycle & state

States the entry is in, and the moves between them

The keystone: a running timer is just the one entry whose end is null. "Running" is a row state, not a separate object — so the canonical word for that state is open.

TermDefinitionAliases to avoid
Open The state of the single entry whose end IS NULL — the live timer. The core invariant is stated in these terms: at most one open entry exists at any time. The tray may display this as a "running timer," but in the model, code, and these docs the state is open. running, live, active, ticking (as the data state)
Closed An entry that has an end. Synonymous with completed. Backfilled entries are born closed. finished, done, ended
Start Open a new entry. Atomically stops any currently open entry first (one transaction), preserving the one-open-entry invariant. Nothing is required to start a bare timer. begin, new timer
Stop Set end = now on the open entry, closing it. --at may backdate the stop. end (as a verb), pause, finish
Resume Start a fresh entry that copies the most recent entry's description, client, project, and billable flag. A new entry — never a re-opening of the old one. Contrast resume from favorite below, which copies a named template rather than the last entry. restart, continue, reopen
Resume from favorite Start a fresh entry from a Favorite (§01) — one action that instantiates the named template's description/client/project/tags/billable. CLI: tt fav start <name> or tt start --fav <name>. Distinct from plain Resume (copies the last entry) and never re-opens an existing entry. resume (unqualified, when a favorite is meant), launch favorite, run favorite
Pin Create a Favorite from an existing entry or the running timer, capturing its attributes as a reusable template. The inverse is unpin (delete the favorite). A favorite may also be renamed. save as favorite, bookmark, star
Backfill Create a closed entry from explicit from/to times, after the fact (tt add). Backfilled entries are first-class — indistinguishable from tracked ones except by their timestamps. manual entry (the act is "backfill"), import
Split Cut one entry into two at a chosen instant within its span. Both halves inherit the original's client/project/tags/billable; the user adjusts from there. divide, cut
Merge Combine a contiguous selection of entries into one spanning earliest start to latest end. On attribute conflict: GUI prompts; tt uses the first entry unless overridden. Descriptions concatenate, tags union. combine, join
Archive Soft-retire reference data (client, project, tag) once it's been referenced, preserving history. Distinct from delete, which removes an entry outright (CLI needs confirmation or --force). delete a client/project, hide, deactivate
03 Time & duration

The derived numbers — and the stored truth they come from

Stint stores only start and end. Every duration below is derived on read; none is stored or incremented. Keeping these three names distinct is how we avoid quietly billing the wrong number.

TermDefinitionAliases to avoid
Stored truth The exact start and end timestamps, UTC, to the second. Never mutated by rounding or sleep subtraction. The cardinal rule of a billing tool: store the truth, never silently change it. The hardening and recovery mechanisms (§05) exist to protect this truth, never to alter it. actual time, real time
Elapsed now − start for an open entry. The count-up shown live in the tray — pure display, never persisted, always correct on read because it is never kept anywhere to drift. Guarded against wall-clock skew: now < start never yields a negative or garbage value. running total, current duration
Raw duration (end ?? now) − start. The full wall-clock span of an entry, before any exclusion. This is the "raw duration" column in CSV export. total time, gross time, duration (unqualified)
Billable duration (end ?? now) − start − excluded_seconds. The trustworthy number that can reach an invoice. The one figure reports total and (optionally) round. billable time, net time, invoiced time
Excluded time Seconds carved out of raw duration to reach billable duration, held in excluded_seconds. In v1 its only source is subtracted slept time (see §05). Reversible and precise; never auto-applied. deducted time, idle time, break
Rounding A display/export lens, not an edit. Off by default. When on, it rounds the grouped billable line (not each entry) to a configurable increment — default nearest 15 min; nearest, not always-up. Stored truth is untouched. rounding up, adjusting, snapping
Report run One resolution of a saved report (§01) against current data: the range-spec resolves to concrete dates, filters and group-by apply, totals render, optionally rounded — optionally piped to a CSV/JSON export. A run is ephemeral output; the definition is the durable noun. CLI: tt report run <name>. See §09 — this is the second sense of "report." report (the definition), generate, render (unqualified)
Group key The value a report run's rows are bucketed under when grouping is on: a local-date token YYYY-MM-DD (by day), or a client / project / tag name. An entry lacking a value for the grouped dimension falls under a canonical empty-dimension bucket — exactly (no client), (no project), or (untagged) — which sorts first. Grouping by tag fans out: a multi-tag entry appears under each of its tags, so per-tag lines may sum to more than the grand total — which stays the de-duplicated sum of distinct entries and is identical under every grouping. bucket (reserve that for the empty-dimension label), segment, category
04 Billing semantics

What counts, and for whom

TermDefinitionAliases to avoid
Billable A per-entry flag. Default true when a client is set, false when clientless — always overridable (client work marked non-billable for goodwill; internal work rarely flagged billable). Reports default to billable-only. chargeable, invoiceable (that's the total, not the flag)
Clientless time An entry with no client (client_id IS NULL) — admin, email, own tooling. Defaults to non-billable. The canonical name describes the condition; it is what "internal/overhead time" refers to. internal time, overhead, non-client time
Non-billable Any entry whose billable flag is false — clientless by default, or client work explicitly excused from the invoice. Counts toward "total worked" but not "invoiceable." free, unbilled
05 Hygiene & integrity

Keeping the hours honest

The mechanisms that catch a number before it lands, wrong, on an invoice — and, newly, the mechanisms that keep the file itself intact. All surface the problem; none silently fixes the hours.

TermDefinitionAliases to avoid
Core requirement The classification (PRD §C) for a requirement that does one of: (a) ensures data integrity (atomicity, invariants, immutability of stored truth); (b) protects against data loss (crash-safety, durability, backups, the export escape hatch, destructive-action confirmation); (c) enables core data entry (start, stop, manual backfill). Carries a core badge across the PRD. Explicitly not core: sleep accuracy, the offline/no-network guarantee, and cross-surface sync — judged accuracy / privacy / consistency, not integrity-or-loss. critical, P0, essential, must-have
Slept-through The flag on an entry that was open while the computer slept. Set whenever one or more SleepSpans attach to the entry. The timer keeps running through sleep (wall-clock stays honest); the flag prompts a decision about the dead time. asleep, paused, away
Subtract The reviewed action — one-tap in GUI, tt sleep subtract <id> in CLI — that moves a SleepSpan's seconds into the entry's excluded_seconds, trimming billable duration precisely and reversibly. The only v1 path to excluded time. deduct, remove, cut
Reconcile On launch, compare the wall-clock gap between the app's last Heartbeat (last_seen_utc) and now against the open entry; when the gap exceeds the reconciliation threshold and an entry is open, record the missed sleep cycle as a SleepSpan with source = gap. Recovers a sleep that happened while the app wasn't running. sync, catch up, recover
Overlap Two entries covering the same instant, usually from backfill. Warned, not blocked (GUI banner, tt stderr note); allowed if meant. Overlapped spans are flagged in reports so the same time can't silently bill twice. conflict, collision, double entry
Check-in The autonomous notification guarding a left-open timer: first at start + 60 min, then every 30 min. Fires on schedule whether or not the user responds; survives relaunch (fires once if due while closed) and realigns from wake after a long sleep. nudge, reminder, alert, ping
Cadence The recurring schedule of check-ins for an open timer (the 60-then-30 rhythm, configurable). A custom interval picked from a notification applies to the next gap only, then reverts to default. frequency, schedule (unqualified)
Heartbeat The last_seen_utc timestamp the running app refreshes periodically (and on clean quit), kept in App state (§06). On next launch the gap between the heartbeat and now is what Reconcile measures to recover a missed sleep cycle. A liveness marker, never billable time. last ping, keepalive, last active
Check-in schedule The persisted next-check-in due time (checkin_state, in App state §06) that survives relaunch and wake — so a due check-in fires once even if the app was closed when it came due, and the cadence realigns from wake after a long sleep. Present iff a timer is open; cleared the moment the timer stops. timer, alarm, next reminder
Backup A timestamped copy of the SQLite file, written beside the database on launch when the DB has changed since the last one; the last N are retained (default 5). The durability net for the stored truth — a copy, never a mutation. Source for restore. snapshot, dump, archive (that's reference data), export (that's CSV/JSON data-out)
Restore Replace the live database with the latest good backup. Invoked on demand — which first sets the current file aside as a replaced file — or automatically by corruption recovery, which moves the corrupt file aside as a quarantined file and restores the newest good backup, informing the user. Data is never silently lost. rollback, revert, recover (use "restore" for the act)
Integrity check The PRAGMA quick_check/integrity_check run at startup. On failure, Core does not write — it triggers restore from the latest backup. One of the §20 hardening guarantees, alongside WAL/FK/busy-timeout assertions on open and the partial unique index enforcing at-most-one-open at the DB level. health check, validation, db check
Quarantined file The corrupt database moved aside to a .corrupted-<timestamp> sibling (with its -wal/-shm companions) when the integrity check fails on open and corruption recovery runs. Kept, never deleted — the bad file is preserved for inspection while a restore brings the live database back from the latest backup. corrupt backup, trash, bad file
Replaced file The previously-live database displaced to a .replaced-<timestamp> sibling during a user-initiated restore, so restoring never destroys the current data outright — it is set aside first, then the chosen backup becomes live. The restore-time counterpart of the auto-recovery quarantined file. old db, overwritten file, backup (that's the copy restored from)
06 System & surfaces

The parts that move the data

Domain-relevant architecture terms. Note especially surface — the deliberate replacement for "client" when we mean the app or the CLI. How these parts fit together is laid out in architecture.html.

TermDefinitionAliases to avoid
Surface One of the two equal ways in — the tray app or the tt CLI. Both import the same Core (@stint/core) and speak to the same database; neither is privileged. Use "surface" (or "front door") — never "client" — for these, to keep "Client" meaning the customer. Each runs on macOS and Linux (Windows is not supported). client (the app), frontend, UI (as a synonym)
Core The shared TypeScript package (@stint/core) that owns the schema, every state transition, all invariants, reporting, and rounding. The single home of logic — no duplicated rules across surfaces. engine, backend, shared lib
tt The CLI executable — Stint's terminal surface, at full parity with the GUI. Human-readable tables by default, --json for scripting. Favorites and saved reports are fully reachable here (tt fav …, tt report …). the CLI tool (just "tt" or "the CLI"), stint-cli
The database The single local SQLite file (WAL mode) that is the one source of truth. Default in the OS's standard app-data directory — macOS ~/Library/Application Support/stint/…, Linux ~/.local/share/stint/… (resolved via Electron's app.getPath('userData'), and the per-OS equivalent for tt), overridable via TT_DB. Offline, always. the store, the DB file, local cache
Invariant A rule Core enforces transactionally — chiefly at most one open entry (also defended by a DB-level partial unique index), and stored truth is never mutated by rounding or exclusion. When we say "the invariant," this is what we mean. constraint (unqualified), rule
Release A date/build-versioned artifact published to GitHub Releases — the public repo is the distribution backend. Every merge to main builds and publishes one. Versioned YYYY.M.D with a build suffix for same-day merges (e.g. 2026.6.27.2). Synonymous with published version. build (unqualified), distribution, package (overloaded), tag
Installer The single artifact that, in one step, places the GUI in Applications / the app launcher and puts tt on PATH. A macOS .pkg; a Linux install script or package. No Apple Developer ID / notarization dependency — the user clears Gatekeeper once. setup, bundle (that's the app bundle), package (overloaded)
Update The in-app, guided download + install of a newer release. Settings → Software Update checks the GitHub Releases API, reports up-to-date or newer, then downloads the artifact and walks the user through replacing the app. An update never touches the database. upgrade, patch, auto-update (it is guided, not silent)
App state The durable app_state key/value table holding the running app's own runtime bookkeeping — currently the check-in schedule (checkin_state) and the heartbeat (last_seen_utc). Distinct from Setting (user configuration) and from Entry data: it is what the app remembers about itself between runs, never tracked time. session state, cache, settings (that's user config)
tt launcher The small, dependency-free POSIX-sh shim symlinked onto PATH (/usr/local/bin/tt, falling back to ~/.local/bin/tt) by the single installer. It makes tt work with no separate Node install: it locates the CLI entrypoint inside the installed GUI app bundle and exec's the Electron-bundled Node against it (a system Node ≥ 22.5 is the fallback). The mechanism behind "one install puts tt on PATH." wrapper, tt binary (it's a shim, not a compiled binary)
Offline guarantee The standing invariant that no shipped code makes a network connection — the database is local, always — with the single exception of the user-initiated update check. Backstopped in CI by a scan that greps all shipped source for networking imports / request APIs and asserts production dependencies stay within a minimal allowlist (the live-traffic confirmation is manual, under a network monitor). privacy mode, airplane mode, local-only (informal gloss)
Version stamp The single shared APP_VERSION both surfaces report, written from the date-based release grammar (YYYY.M.D[.N]) at build time and overridable at runtime via STINT_VERSION. One source of truth for "which version is this," distinct from the release artifact it labels. build number, git sha, tag (that's the release tag)
Time-range picker The GUI's visual start+stop selection surface: a single-day calendar column with hour lines on which the entry is a draggable rectangle (drag body moves start, drag bottom resizes end; 5-min snap; other entries gray, overlaps yellow, warn-only). Opens on click of a time text field or its calendar icon. Used in add-entry, edit-closed-entry, and edit-running-start (not split); overnight spans use text. Text entry remains everywhere and stays authoritative — the picker is a convenience over it, never a replacement. calendar picker, timeline, scheduler, date picker
07 Relationships & cardinality

How the entities connect

RelationshipCardinalityRule
Client → Projectone-to-manyA project belongs to exactly one client. A client may have zero projects.
Client → Entryone-to-many (optional)entry.client_id is nullable; a clientless entry is internal time.
Project → Entryone-to-many (optional)An entry's project, if set, determines its client — the project's client is authoritative.
Entry ↔ Tagmany-to-manyVia the entry_tag join. Tags are flat and cross-cutting.

08 Language in use

The vocabulary in natural sentences

Short exchanges showing the terms used the way we want them used — and quietly not used the way we don't.

Dev"When the user runs tt start while a timer's going, do we error?"

Owner"No — start stops the open entry first and opens a new one, in one transaction. There's never more than one open entry, so there's nothing to conflict with."

Dev"The report total looks high for a day she was away at lunch."

Owner"Lunch isn't excluded time unless the computer slept and she subtracted the SleepSpan. Raw duration includes it. In v1 the only thing that touches excluded_seconds is subtracted sleep."

Dev"Should the tray and the CLI both be called clients of the Core?"

Owner"Call them surfaces. Client is the customer you bill — if we overload it we'll confuse every report query."

Dev"She clicked her 'Acme standup' tile and it picked up last night's description."

Owner"Then it wasn't a favorite — that's plain resume, which copies the last entry. Resume from favorite instantiates the named template. Two different starts."

Dev"'Open the weekly report' — do you mean the saved one or run it?"

Owner"The saved report is the definition — range, group-by, filters, rounding. Running it resolves that against today's data. Edit the definition; read a run. Never say just 'report' when which one matters."

Dev"The DB looked corrupt on launch — did we lose her morning?"

Owner"No. The integrity check failed, so Core didn't write — it quarantined the file and ran a restore from the latest backup. A backup is a copy of the stored truth; we never mutate it to fix it."

Dev"Reconcile added a two-hour SleepSpan on launch — do we trust it?"

Owner"That one's source = gap — a suspicion from the heartbeat gap, not a live power event. Flag it for review; don't subtract it automatically. Only the user's subtract moves seconds into excluded_seconds."

09 Flagged ambiguities

Resolved here, watch for drift

Terminology forks the source documents left open. Each is now decided; this section records the decision and the reasoning so it isn't relitigated.

AmbiguityResolution
"Report" means two things — the saved definition (range/group-by/filters/rounding), and a run/export of it against current data. Saved report (or "report definition") = the persistent, named preset, the durable noun in the report table. Report run = one resolution of it into totals, optionally exported. When the distinction matters, never say bare "report." "Run" and "export" are verbs on a definition; "edit"/"rename"/"delete" act on the definition itself.
"Resume" — last entry or a favorite? Both start a fresh entry from copied attributes. Resume (unqualified) copies the most recent entry. Resume from favorite instantiates a named, pinned template (the favorite table). A favorite is not an entry and holds no times. Don't say "resume" when a favorite is meant.
Visual range picker vs text entry — which is the source of truth for an entry's start/stop? Text entry is authoritative and remains everywhere. The time-range picker is a visual convenience layered over it (and the only path for some interactions, but never the canonical store). Overnight spans use text; the picker covers same-day add/edit-closed/edit-running-start, not split.
"Backup" vs "export" vs "archive" — three ways something is "saved." Backup = timestamped copy of the SQLite file for durability/recovery. Export = CSV/JSON data-out for invoicing. Archive = soft-retiring reference data. Three distinct concepts; none is a synonym for another.
"Client" means two things — the billed customer, and a consumer of the database (the app / the CLI). Client = the customer, always. The app and CLI are surfaces (or "front doors"). The concept doc's "two thin shells / two front doors" framing wins; "client" is never used for software here.
"Running" vs "open" vs "live" for the timer state. Open is canonical for the state (end IS NULL), matching the invariant and the "running is just an open row" insight. "Running timer" is allowed only as user-facing tray display copy, never as the model/code term.
"Entry" vs "stint/session/span" for a tracked block. Entry is the entity, everywhere. Stint is the product name only. The prose synonyms (span, session, spell) are retired from technical language.
Scope of excluded_seconds — sleep-only, or a general "subtract dead time" carve-out? Sleep-only in v1. The sole source is a subtracted SleepSpan, so "excluded time" and "subtracted slept time" are the same concept for now. If v2 adds manual exclusion (e.g. a mid-entry break), "excluded time" broadens and SleepSpan becomes one source among several — revisit this entry then.
Clientless time — "internal," "overhead," or literal? Clientless time (literal). "Internal time" / "overhead" stay as informal gloss in prose but are not the canonical term, since the defining fact is the absent client, not a category judgment.
What counts as "core"? — every must-ship requirement, or a narrower set? Core requirement is a precise classification (§C): data integrity, data-loss protection, or core data entry — nothing else. Sleep accuracy, the offline guarantee, and cross-surface sync are deliberately not core. "Core" is not a priority synonym; it gates the QA-evidence regime (recordings, the AC-sufficiency review).
How a SleepSpan was found — is "the computer slept" a fact or a guess? A SleepSpan records its source: event is an authoritative live capture from the OS power monitor; gap is a suspicion inferred on launch from a wall-clock heartbeat gap (see Reconcile); unknown is unrecorded provenance. Treat a gap span as a review-worthy guess, never ground truth; never silently bill or un-bill on a gap alone.
Quarantined vs replaced — two set-aside database files during recovery. Quarantined file (.corrupted-<ts>) is the corrupt file moved aside by automatic corruption recovery. Replaced file (.replaced-<ts>) is the previously-good live file moved aside by a user-initiated restore. Both exist so data is never destroyed outright; neither is a backup (the copy restored from).
Range preset vs range spec — the menu choice vs what a saved report stores. Range preset = one of the five named relative windows. Range spec = the durable range a saved report holds — either a preset (re-resolved each run) or an absolute UTC window. The preset is a value; the spec is the stored definition that may contain one.
ProcessThis glossary follows the grill-with-docs approach: fuzzy language gets sharpened through interview, then formalized here. New terms and refinements land in this file as the design tree is walked. It is the thinnest useful documentation layer — a glossary of non-obvious terms plus decisions already made — meant to be read alongside, and to stay consistent with, the concept and the PRD. Most recently extended via the issue #27 grill, which canonicalized domain that emerged during implementation: SleepSpan (with its event/gap source), App state / Heartbeat / Check-in schedule, the quarantined/replaced recovery files, group key & empty-dimension buckets, range preset/spec, and the packaging terms (tt launcher, offline guarantee, version stamp). "Switch" was deliberately not added and has since been removed (issue #34) — it was a redundant alias of Start, which already stops any open entry first as one atomic stop-then-start. Switching tasks is starting; attribute carry-forward is what Resume does.
