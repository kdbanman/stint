Process & Verification Requirements · v1 (seed) · properties the apparatus must satisfy

The application requirements in prd.html say what Stint does for its user. This says what the process that proves Stint must do — the properties the build, CI, and acceptance apparatus must satisfy. These are SDLC requirements, a different kind of claim from the application requirements, and until now they had no home.

Read with acceptance.html (the AC strategy — which notation proves each requirement) and prd.html (the what). Where acceptance.html routes a requirement to a verification method, this governs how the apparatus running those methods must behave. This is a seed: a first capture of principles that were institutional knowledge scattered across test comments, CI YAML, and the .claude/ machinery. Completing it — a full process glossary, the automatic-vs-manual inventory, release/branch governance, and the requirements-change process itself — is tracked in issue #19.

These are requirements on the apparatus, not the applicationNothing here changes what Stint does for a user. Each principle constrains how we prove and protect the software; a violation is a process bug — a false guarantee, a blind spot in the gate, a guard that can rot unseen — not a product bug. They are orthogonal to the five AC methods: a GOLD test can be perfectly correct and still violate one of these (run too late, or be unprotected).

01 Why a separate doc

Application requirements and process requirements are different kinds of claim

The four canonical docs — concept.html, prd.html, glossary.html, acceptance.html — are all about the application: the why, the what, the words, and how we'll know it's true. None of them is a home for requirements on the engineering process itself. So principles like "every packaging guard self-heals" had nowhere to live and ended up as comments in test files and workflow YAML. This doc is that home.

02 The principles

The verification apparatus is itself a system with requirements

Each principle below is stated, given its reason, and tied to the concrete episode that surfaced it (mostly the release-packaging break: the pack job failed on every merge with spawn … app-builder ENOENT while build/test CI stayed green, because nothing on the PR path exercised packaging). Together they cover the apparatus's own properties: coverage, timing, completeness of mirror, resilience, self-protection, and honesty.

  1. Green must be meaningful. A passing check must actually cover what its presence implies; a blind spot in the gate is a false guarantee, worse than no check because it reassures. Seen in: packaging was broken for weeks while CI was green — the signal silently excluded packaging.
  2. Verify before the gate, not after it. A check that runs only after the merge it is meant to protect cannot prevent a regression; it only discovers it later. Every concern that gates release earns a check on the path that gates it — the PR. Seen in: the real pack ran only post-merge (release.yml), so the break landed on main repeatedly unseen; the fix added a PR-time pack-smoke job.
  3. Every manual check earns an automated mirror (a "safety valve"). A human-run runbook step should have a cheap automated guard over its invariant, so a regression fails CI rather than waiting for the next manual pass. The mirror need not be the full check — a static config assertion or a fast smoke is enough to catch the common regression. Seen in: build-matrix.test.ts mirrors the "no Windows" runbook invariant; verify:packaging + pack-smoke mirror "the artifacts really build."
  4. Guards self-heal where the failure is known and recoverable; otherwise they fail loud and early. A guard facing a known-recoverable fault (a poisoned cache) should recover it; one that cannot should fail at the boundary with a specific, actionable message — not a cryptic failure deep inside a downstream tool. Seen in: the packaging guard reinstalls when the native helper is missing, and otherwise reports exactly which binary is absent instead of electron-builder's bare ENOENT.
  5. Protect the guard (meta-verification). A guard is code: it can be deleted, disabled, or quietly rot. Assert that it stays wired, so removing the guard is itself a caught regression. Seen in: build-matrix.test.ts asserts the pack-smoke job and the verify:packaging guard remain present in CI.
  6. Fix robustly under diagnostic uncertainty, and record the residue. When a failure's exact mechanism can't be reproduced, prefer a fix robust to the cause over one that bets on a single mechanism — and write down what stays uncertain rather than implying certainty. Seen in: the packaging fix dropped the suspect cache and added self-heal without claiming to have proven the npm mechanism, and the PR recorded the residual uncertainty. This is the build-system echo of acceptance.html's "honest residue / residual risk we accept."
The threadCoverage (1), timing (2), completeness of the manual→automated mirror (3), resilience (4), self-protection (5), and honesty (6) are the same demand applied to the apparatus that we already apply to the product: that a guarantee mean what it says.
03 Relation to the AC methods

The methods prove the product; these principles govern the apparatus

acceptance.html defines five methods — BDD · PROP · GOLD · JUDGE · MANUAL — and a decision rule routing each application requirement to the method that fits. That rule answers which notation proves this requirement. The principles here answer a different question: once a guard exists, how must it behave — when it runs (2), whether its green is meaningful (1), whether a manual check has an automated twin (3), how it fails or heals (4), whether it is itself protected (5), and how honestly its limits are recorded (6). The two compose: route a requirement to a method via acceptance.html §03, then make the resulting guard satisfy this doc.

04 Honest residue

This is a seed, not the complete SDLC spec

This first capture is deliberately narrow — the verification principles above and the map below. The rest of the project's SDLC is still a combination of institutional knowledge, scattered comments, and GitHub systems. Consolidating it is tracked in issue #19; explicitly not yet specified here, and pending that pass:

  • A full process glossary — the SDLC twin of glossary.html (gate, guard, safety valve, backstop, evidence, surface, parity, runbook, smoke, self-heal…), one canonical term per concept.
  • The automatic-vs-manual inventory — exactly what CI enforces vs what a human must run, and where the boundary sits.
  • Branch / PR / release governance — naming, ready-not-draft PRs, publish-on-merge (§19 R05), versioning (§19 R06), the merge gate.
  • The requirements-change process itself — the change-requirements skill + requirements-transition workflow and the old→new swap (§Z), and how this doc is changed vs prd.html.
  • The meta-decision of what must be specified vs what can stay convention, recorded so the apparatus is neither under- nor over-specified.
