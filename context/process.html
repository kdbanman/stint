Process & Verification Requirements · v1 (seed)

prd.html specifies the product; this specifies the apparatus that proves it. These are SDLC requirements — properties the build, CI, and acceptance system must hold — a different kind of claim from the application requirements, and until now without a home.

Read with acceptance.html, which routes each requirement to the method that proves it. That decides which notation; this constrains how the resulting guard must behave. A violation is a false green, not a user-visible bug, and is orthogonal to the five AC methods: a correct GOLD test can still violate these by running too late or being unprotected. This is a seed — the full SDLC spec (process glossary, automatic-vs-manual inventory, release governance, the requirements-change process) is tracked in issue #19.

02 Requirements

The principles

Each is normative. The reference names the concern that motivated it; the first surfaced from the release-packaging failure (the pack job failed on every merge with spawn … app-builder ENOENT while CI stayed green), and the later ones generalize the same discipline to the offline guarantee, evidence integrity, build-time source transforms, and the source census that measures the codebase.

  1. Green is meaningful. A passing check must cover every concern its presence implies; a gate with a blind spot is a false guarantee, worse than none because it reassures. (§19 R01: packaging was broken while CI was green — no check exercised it.)
  2. Verify before the gate. A check that runs only after the merge it protects cannot prevent a regression, only discover it. Every release-gating concern has a check on the PR path. (§19 R01: the real pack ran post-merge only.)
  3. Mirror every manual check. Each MANUAL runbook step has a cheap automated guard over its invariant — a static assertion or a smoke run — so the common regression fails CI, not the next manual pass. (build-matrix.test.ts; the pack-smoke job.)
  4. Self-heal or fail loud. A guard recovers a known-recoverable fault; otherwise it fails at the boundary with a specific, actionable message, never a cryptic downstream error. (packaging guard reinstalls a missing native helper, else names the absent binary.)
  5. Protect the guard. A guard is code: it can be deleted, disabled, or rot. Assert that it stays wired, so removing it is itself a caught regression. (build-matrix.test.ts asserts pack-smoke + verify:packaging remain.)
  6. Fix to the cause; record the residue. When a failure's mechanism is unproven, prefer a fix robust to the cause over one betting on a single mechanism, and document what stays uncertain rather than implying certainty.
  7. The offline guarantee has a backstop. The product makes no network connection except the user-initiated update check (§17 R09, §19 R03); an automated check asserts this across every shipped surface — including assets that ship outside packages/*/src — and asserts production dependencies stay within a reviewed minimal allowlist. The check's own coverage is asserted, so a newly shipped surface cannot silently escape it. (§17 R09: the offline promise is load-bearing and otherwise only manually confirmable.)
  8. Regenerated evidence must not drift. Any check that regenerates checked-in acceptance evidence asserts the result is identical to what is committed and fails on any difference. Evidence that is regenerated but never compared is a false green. (Generated proof is only trustworthy if a divergence between code and committed evidence breaks the build.)
  9. Build-time source transforms are idempotent and fail loud. A step that rewrites source as part of the build — such as stamping the version (§19 R06) — must be safe to re-run without changing the result, and must abort with an actionable error if the target it expects is absent, never silently leave the source untransformed. (A silent no-op ships an unstamped or stale build.)
  10. The codebase census is exhaustive and reconciles. Any tool that measures the source — SLOC, documentation, per-category breakdowns — classifies every tracked file with no catch-all bucket, and asserts its arithmetic reconciles: each file's lines sum to its parts, the parts sum to the whole, and files counted equals files tracked. A new directory or unforeseen file matches nothing and fails the run rather than vanishing from the tally; a total that does not add up fails rather than misleading. The report is generated, never committed, so it cannot drift from the tree it describes. (scripts/sloc-report.mjs, gated by --check in CI. A measurement that quietly drops files or fudges its sums is a false green of a subtler kind — it reassures with a number that is wrong.)

04 Map

Where requirements & context live

The current index, pending #19. CLAUDE.md's Files table is the front door.

Under acceptance/ the two halves of the AC apparatus are kept apart: criteria/ holds the acceptance criteria (what must hold — the rubric, schemas, parity matrix, runbook, and coverage map), and evidence/ holds the acceptance-criteria satisfaction evidence (the generated proof those criteria hold). Criteria are hand-authored and reviewed; evidence is regenerated by the suites.
