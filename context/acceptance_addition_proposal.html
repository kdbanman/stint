Acceptance Criteria — Proposed Additions · adversarial implementation-vs-spec audit · NOT yet adopted
Proposal · additions only · not adopted

This is a proposal of additions to acceptance.html, surfaced by the adversarial implementation-vs-spec audit of issue #27. Each item below is an implicit requirement found in the shipped implementation that the acceptance corpus does not (fully) pin down — an external file contract, an output string, an IPC shape, or an evidence-determinism convention that downstream tools, tests, or reviewers already depend on, but that lives only in code and golden fixtures.

It is not a replacement for acceptance.html and is not yet adopted. Nothing here changes the build gate until it is reviewed and merged into the canonical doc. Items are numbered P01… to keep them distinct from the adopted Rxx requirements; adoption would re-home each into the appropriate section of acceptance.html (or a sibling acceptance/criteria artifact, where noted).

The acceptance strategy pins behaviour, invariants, and the broad shape of the CLI/export contract. This audit found a residue of byte-level and contract-level details — exact column names, output strings, ack shapes, pinned clocks — that the implementation fixes but the corpus leaves to code. These are precisely the things a golden contract should own.

Each proposed addition carries: the proposed requirement text; a short Why (the implicit behaviour plus concrete code/test evidence); an audit status — gap (unspecified) or partial (shape pinned, encoding/contract detail not) — and, where partial, the existing citation it builds on. Where the gap is purely an AC-coverage matter with no new prose requirement, it is filed under §04 as an AC addition only.

01 CSV export — byte-level contract

The export file is consumed by other tools — pin its encoding

The acceptance corpus and PRD §09 R06 already enumerate the CSV column set and order, and §08 shows one golden header line. What no doc pins is the encoding: the exact machine column names, the tag delimiter and ordering, boolean rendering, RFC-4180 quoting, and the trailing newline. There is no csv.schema.json / golden-CSV fixture artifact (only export-entry.schema.json for the JSON sibling). The two audit findings for this area (core-domain-29, core-csv-01) are merged here.

  1. CSV export is a frozen golden contract: a single header row with exactly the columns client, project, tags, description, start_utc, end_utc, raw_duration_s, excluded_s, billable, overlapped in that order; tags is the entry's tags sorted and joined with ;; billable and overlapped render as true/false; any cell containing a comma, double-quote, CR, or LF is quoted per RFC 4180 with embedded quotes doubled; the file ends with a trailing newline.

    WhyThe exporter emits these exact headers, a ;-joined sorted tag list, true/false booleans, and RFC-4180-quoted cells with a terminating newline; downstream invoicing/spreadsheet tooling parses against this byte layout. The corpus fixes the column set/order but is silent on header spelling (start_utc vs start, raw_duration_s vs raw duration), the ; delimiter, tag sorting, boolean form, quoting, and the trailing newline — these live only in code, with no CSV schema/golden artifact.

    Statuspartial  Citesprd.html §09 R06 (CSV column set/order); acceptance/criteria/schemas/export-entry.schema.json (JSON sibling, same field set/order). No CSV schema or golden-CSV fixture exists.

    ACGOLD a golden CSV fixture — the header line plus a fixture row with a multi-tag entry (deep;meeting) and a comma-containing, quoted description — byte-compared against exporter output.

Why this belongs in the corpus, not the PRD proseThe column set is already a PRD requirement; what is missing is the encoding, which is exactly the "artefact is the criterion" case the acceptance doc reserves for GOLD. The cleanest home is a checked-in golden CSV beside the existing JSON schemas.
02 CLI human output & tables

The human surface is a contract too — declare its goldens canonical

The corpus pins the --json shape hard, and §08 establishes the house style for human output with a single golden line. But the full per-command human-string contract and the precise table-layout rules exist only as snapshots in the CLI golden suite, not as declared, canonical criteria. Findings cli-humanstrings-19 and cli-table-format-20.

  1. Each tt command's human (non-JSON) stdout and its empty-state copy are a frozen golden contract. The corpus declares them canonical and enumerates (or points at the golden fixtures for) the running-status line ▸ running <dur> · <desc>… and its nothing running form, the stopped <dur> · <label> and added entry <id> · <dur> confirmations, the per-collection empty states (no entries / no clients / no projects / no tags / no favorites / no saved reports / no backups / sleep-flagged), and the unchanged — no new backup needed message.

    WhyEach command emits a fixed human line in a ·-separated house style; these strings are asserted in the CLI golden suite but never declared canonical in the corpus. Only fragments appear in the docs (nothing running in §11; one golden line and one illustrative empty state in acceptance.html §08 / prd.html §12) — the full family of strings, the ▸ glyph, and unchanged — no new backup needed are unspecified prose.

    Statuspartial  Citesprd.html §11 (nothing running); prd.html §12 (one illustrative empty state); acceptance.html §08 (one golden human line establishing the · style).

    ACGOLD an approved fixture per command capturing its human stdout and empty-state line, byte-compared (most already exist; the addition is to declare them canonical and have PRD §11 reference that they are golden-pinned).

  2. The human table layout is a golden contract: uppercase column headers; left-aligned columns each padded to max(header, cell) width and joined by a 2-space gutter; trailing whitespace trimmed per line; no ANSI / colour (monochrome). The tt list column set/order is ID, START, END, DUR, CLIENT/PROJECT, DESCRIPTION, BILL, FLAGS. For grouped (--by) output, each block carries a <group key> (<hours>h) header, blocks are separated by a blank line, and groups appear in ascending key order.

    WhyThe renderer pads left-aligned columns with a 2-space gutter, trims trailing whitespace, emits no colour, and renders --by groups as <key> (<Xh>) headers in ascending order separated by blank lines. The corpus only states "human-readable tables by default" and demonstrates uppercase headers via §08 snapshots; the precise layout rules, list column set/order, and group-header format are implied by snapshots, never stated.

    Statuspartial  Citesprd.html §11 ("Human-readable tables by default"); prd.html §09 (grouping by client→project, day, tag); acceptance.html §08 (golden fav ls table demonstrating uppercase headers and column layout).

    ACGOLD byte-compared fixtures for tt list and tt report --by day covering header casing, the 2-space gutter, per-line trim, the column set/order, and ascending group order.

03 GUI IPC & renderer contracts

Internal write contracts the banner behaviour rides on

The PRD pins the behaviour that overlapping writes warn-not-block and surface a banner. The audit found the uniform write-acknowledgement shape that carries those warnings — and its special-case return shapes — is an internal IPC contract no doc specifies. Finding gui-ipc-14. (Two further renderer findings, gui-renderer-07 and gui-renderer-18, are AC-only and filed in §04; the live-total definitional gap gui-renderer-28 is in §04 as well as it pins meaning rather than a shape.)

  1. The GUI write-acknowledgement contract is schema-pinned: every write IPC returns a uniform WriteAck of the form { warnings: [{ kind, message, overlapsWith[] }] } carrying the overlap warnings (if any) the GUI renders as a banner; split always returns warnings: [] (it cannot introduce new overlap); and exportEntries returns either { canceled: true } or { written, path }. Best authored as an acceptance/criteria schema rather than PRD prose.

    WhyEvery write IPC returns the uniform WriteAck shape; the renderer's overlap banner reads warnings[].overlapsWith; split returns empty warnings; exportEntries returns canceled-vs-written-path. The non-blocking-warning behaviour is specified (§06 R04, §12 R07/R09), but the ack shape and the split/export special cases are implementation contracts the spec is silent on.

    Statuspartial  Citesprd.html §06 R04 (overlap warned-not-blocked, GUI banner); prd.html §12 R07/R09 (at-write-time and durable overlap banners); prd.html §09 R06 (export as the data-out path).

    ACGOLD a JSON-schema fixture for the WriteAck shape and for the export-result shape, asserted against representative IPC responses.

04 AC-only additions (no new prose)

Where the spec already says enough — only the AC needs strengthening

For three findings the prose requirement already exists; the audit's concern is purely that the mapped acceptance criteria do not yet pin the implementation contract. These add or extend AC without proposing new requirement text.

Check-in interval picker decode gui-renderer-07 · AC only

The action set and ordering are already specified — PRD §10b R04 lists Stop, Keep going, and the inline +15m/+30m/+60m/+120m choices with next-gap-only revert (echoed in glossary line 329). What is unpinned is the numeric button-index decode in checkin-actions.js (index 0 → stop, 1 → keep-default, out-of-range → keep-default), which is an implementation detail, not spec prose.

Statuspartial  Citesprd.html §10b R04; glossary.html line 329.  Additionnone (no new requirement).

ACGOLD pin the check-in action list and its decode (button index → { stop | keep-default | +Nm }) so a notification rebuild matches the spec's set and ordering.

Split picker default & in-span rejection gui-renderer-18 · AC only

The midpoint default and the strictly-interior rule are both already pinned — PRD §06 R02 ("an instant within its span") and the SPLIT_AFFORDANCE rubric (picker defaults to the span's midpoint; core enforces the strictly-in-span rule). The only unspecified item is the renderer's client-side rejection copy (Pick an instant inside the entry.); the spec deliberately leaves message wording open, so this is an AC/manual demonstration, not new prose.

Statuspartial  Citesprd.html §06 R02; acceptance/criteria/judge-rubric.md SPLIT_AFFORDANCE.  Additionnone (no new requirement).

ACJUDGE/MANUAL the Split control seeds the midpoint and an instant on/outside the boundary is refused with a clear inline message before any core call.

Entries-view on-screen totals — forbid an undefined third total gui-renderer-28 · definitional

This one is definitional, so it does propose a tightening. §17 R11 pins that list and report totals update live; LIVE_FILTER pins the report total (#week-total) as billable-only equal to tt report; ENTRY_LIST_SEARCH pins per-group headers as "summed billable hours". The implementation, however, computes an on-screen Entries total by summing core-owned billableSeconds across all visible rows regardless of the billable flag — a third quantity that is neither raw-all nor billable-only and that the spec never defines.

  1. Every on-screen Entries-view total is one of the two already-defined quantities: the prominent total equals the billable-only report total (= tt report for the same selection), and per-group header totals are summed billable hours. No additional "list total" may exist that sums billableSeconds while ignoring the billable flag; an undefined third total is forbidden.

    WhyThe live list currently shows a total that sums billableSeconds over all visible rows ignoring the billable flag, so it agrees with neither the billable-only report total nor a raw-all total. The corpus pins the report total and per-group totals precisely but is silent on this third on-screen number — a real definitional gap that lets a non-billable row inflate an on-screen total.

    Statuspartial  Citesprd.html §17 R11 (live list and report totals); judge-rubric.md LIVE_FILTER (#week-total = billable-only = tt report) and ENTRY_LIST_SEARCH (group header = summed billable hours).

    ACJUDGE extend LIVE_FILTER with a machine-scored assertion whose fixture includes a non-billable row, verifying the on-screen Entries total counts only billable rows (matching tt report for the same selection) — catching the current "billableSeconds summed regardless of flag" behaviour.

05 Evidence determinism & judge report

Pin how evidence is generated, and what a pass means

acceptance.html §13 and process.html §04 require that evidence be regenerated by the suites and committed. The audit found two unpinned mechanisms underneath that principle: the determinism mechanism (a pinned clock against a fresh DB) and the judge-report's own pass tri-state convention. Findings evidence-clock-01 and evidence-json-01.

  1. Generated evidence (the CLI transcript, the judge report) is deterministic and regenerable: produced against a fresh, throwaway database with the wall clock pinned via a TT_NOW injection. If the pinned timestamps are normative, they are recorded — the CLI transcript at 2026-06-24T17:30:00Z and the JUDGE fixtureClock at 2026-06-24T23:00:00Z — with any intentional difference between the two clocks explained.

    Whynpm run evidence regenerates a checked-in transcript against a fresh DB with TT_NOW pinning the clock; two different pinned timestamps appear (transcript 17:30Z, judge fixtureClock 23:00Z). The corpus pins the principle (evidence is regenerated and committed) but names neither TT_NOW, the fresh-DB requirement, nor the specific timestamps, and does not address whether the two differing clocks are intentional.

    Statuspartial  Citesacceptance.html §12–§13 (evidence regenerated & committed); process.html §04 (evidence regenerated by the suites); runbook "Executed evidence" notes.

    ACGOLD assert npm run evidence is idempotent — regenerating the transcript twice yields byte-identical output, proving the clock is pinned rather than wall-time.

  2. The judge-report pass tri-state is defined: pass: true/false are deterministic, machine-checked facts; pass: null means captured-but-not-yet-scored and MUST be routed to human/LLM rubric review — never counted as a pass. A null at gate time blocks green.

    WhyThe generated judge-report.json uses pass: true|false|null, where null marks items captured for rubric review rather than machine-asserted. The corpus pins that JUDGE mixes machine facts with rubric-judged items, that any FAIL fails the suite, and that JUDGE is never the sole gate — but the explicit tri-state and the rule that null routes to review and never auto-passes appear only in the report-format's own note, not in authored judge-rubric.md or acceptance.html.

    Statuspartial  Citesacceptance.html §02 (JUDGE never the sole gate) and §09 (PASS/FAIL scoring, FAIL fails the suite, PASS spot-checked); acceptance/criteria/judge-rubric.md.

    ACGOLD a schema for judge-report.json pinning pass as {true|false|null}, plus a harness assertion that no pass: null item is treated as passing when computing the suite verdict.
