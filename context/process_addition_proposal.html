Process & Verification Requirements · Proposed Additions · issue #27
Proposal — not adopted

This page proposes additions to process.html. It is the synthesizer output of issue #27's adversarial implementation-vs-spec audit: each item below is an implicit requirement the implementation already relies on (with concrete code/test evidence) that the process spec does not — or does not fully — pin down.

It is not a replacement for process.html and is not yet adopted. Nothing here is normative until reviewed and merged into the canonical doc. Numbering uses a provisional P-prefix to keep it visibly distinct from the live R-requirements.

The audit surfaced eleven process-level gaps — concerns where a guard's own source is the sole definition of a release-gating invariant, and a spec reader cannot derive what is being enforced. They group into four areas: the no-network backstop, packaging & CI composition, versioning, and the install launcher.

Each proposed addition is given in this doc's voice: the requirement text, a short Why (the implicit behavior plus the code/test evidence that proves the implementation depends on it), the audit status (gap unspecified vs partial stated-but-underpinned), the existing citation where partial, and the proposed AC. Status semantics match the principles in process.html §02: an underpinned guard is a candidate false green — "green is meaningful" only if the spec, not just the scanner, says what green covers.

01 Proposed additions

The no-network backstop

The application requirement — "no network connections, beyond the explicit, user-initiated update check/download" (prd.html §17 R09, §02 non-goals, §13 "No network, ever.") — is well specified. What process.html does not pin is the verification contract: that the guarantee is enforced by an automated scan, what that scan covers, and what closed sets it asserts against. Today scripts/check-no-network.mjs is the sole authority for all four. That is exactly the blind-spot risk "green is meaningful" forbids: an unscanned directory passes silently.

  1. An automated no-network scan gates every shipped source file. A CI-enforced scan MUST assert that no shipped source — including renderer assets that ship outside packages/*/src (e.g. renderer/app.js, renderer/popover.js, renderer/util.js) — imports a networking module or issues an outbound API call. The single user-initiated update-check/download path is the only permitted exception. The scan's file set MUST be derivable from the workspace, so a new shipped directory cannot escape coverage.
    • Why The renderer files ship but live outside src; only an explicit scan of that directory keeps them honest. The offline guarantee is repeatedly specified but the scan enforcing it, and its coverage of the out-of-src renderer, is a process detail absent from the seed.
    • Status partial — guarantee specified, scan + its renderer scope not.
    • Citation prd.html §17 R09; §02 non-goals; §13; process.html (verification apparatus).
    • Proposed AC GOLD the scan's file set explicitly includes the renderer files outside src and fails if any shipped file references a network primitive.
  2. The renderer ships as offline-only DOM scripts. Renderer code MUST contain no browser request API — fetch, XMLHttpRequest, WebSocket, EventSource, navigator.sendBeacon — and MUST load as classic file:// scripts with no network-module imports (no ES-module fetch over the wire), so §17 R09's offline guarantee is enforceable per-commit rather than by inspection.
    • Why "Renderer makes no network calls" follows from §17 R09, but the exact forbidden-API enumeration and the classic-script/no-ES-module loading model are renderer constraints the static guard imposes — not stated in the spec.
    • Status partial — offline intent specified, forbidden-API set + loading model not.
    • Citation prd.html §17 R09; §02 non-goals; §19 R03 ("Electron's built-in net, never node:https / fetch").
    • Proposed AC GOLD a per-commit static guard greps renderer sources for the forbidden request APIs and module imports and fails on any hit.
  3. The shipped production-dependency allowlist is a closed no-network invariant. The shipped production dependencies MUST be exactly @stint/core, commander, and electron. Introducing any other production dependency is an offline/no-network violation requiring explicit review, because each added prod dep is unaudited outbound-request surface.
    • Why The single-allowed-outbound carve-out is specified, and commander / electron are mentioned, but nowhere does the spec state the closed three-dep set or that any other prod dep is a violation. That set lives only in check-no-network.mjs.
    • Status partial — carve-out specified, the closed allowlist not.
    • Citation prd.html §17 R09, §19 R03/R04; COVERAGE.md §19 R03/R04 rows; §13; runbook "CHECK NO NETWORK".
    • Proposed AC GOLD no-network.test.ts asserts the prod-dep set equals the spec-stated allowlist, failing on any addition.
  4. The no-network scan's surface and forbidden set are specified, not scanner-defined. process.html MUST state the scan contract that operationalizes §17 R09: the directories scanned (all packages/*/src plus the GUI renderer dir, since it ships); the file extensions (.ts, .js, .mjs, .cjs, .html, excluding *.test.ts); and the forbidden enumeration — networking modules (node:http(s), node:http2, node:net, node:dgram, node:tls, node:dns; axios, node-fetch, undici, got, ws, socket.io, request), tokens (XMLHttpRequest, WebSocket, EventSource, navigator.sendBeacon), and bare global fetch(). An omitted directory or a new networking API must itself be a caught regression.
    • Why The spec states the goal and that a backstop scans for "any networking import," but never enumerates the concrete surface (which dirs — notably the out-of-src renderer — which extensions, the *.test.ts exclusion) nor the forbidden module/token list or bare-fetch detection. A spec reader cannot derive what "no network" concretely forbids; the scanner is the sole definition.
    • Status gap — scan surface & forbidden set unspecified.
    • Citation none enumerating it (§17 R09 states only "makes no network connections"; runbook "CHECK NO NETWORK" says it scans "the shipped source and production dependency tree for any networking import" without enumeration).
    • Proposed AC GOLD a meta-assertion that check-no-network.mjs's scanned-dir list includes every shipped source dir (compared against the workspace package list), so a new package or renderer dir cannot silently escape the scan.
02 Proposed additions

Packaging & CI composition

process.html's principles (§02) were motivated by the app-builder ENOENT packaging incident, and pin that a release-gating concern must have a PR-path check that stays wired. What remain unpinned are the operational specifics of the packaging guard, the shape of the CI pipeline it sits in, and — most sharply — whether the CI evidence-regeneration step actually asserts anything. The first two are acknowledged residue (§03, deferred to #19); the third is a principle-violating blind spot.

  1. The packaging-toolchain guard proves the helper runs, then self-heals or fails loud. Before packaging, CI MUST verify that electron-builder's app-builder helper exists, is executable, and actually runs (invoke --version under a bounded timeout, e.g. 20s) — a spawn-and-run check, not a stat. On a known-recoverable failure it MUST perform a defined reinstall (clear the npm cache, remove node_modules, npm ci) and re-verify, else fail at the boundary naming the absent binary. The chosen packaging fix npmRebuild: false MUST be recorded with its rationale.
    • Why The principles behind the guard are specified; the concrete operational values — the 20s timeout, the spawn-and-run (vs stat-only) policy, the cache-clean / rm -rf node_modules / npm ci repair recipe, and npmRebuild:false — live only in scripts/check-packaging.mjs. process.html is a seed, so these are acknowledged residue.
    • Status partial — principles specified, operational contract not.
    • Citation process.html §02 ("Self-heal or fail loud", "Verify before the gate", "Green is meaningful"); COVERAGE.md §19 R01 row; runbook "CHECK BUILD MATRIX".
    • Proposed AC GOLD extend build-matrix.test.ts to assert check-packaging.mjs spawns the helper (not only stats it) and that the repair path exists.
  2. The CI gate set and ordering are specified. The PR-path verify job MUST run, in order: npm ci → version stamp → build → lint → typecheck → full test → no-network backstop → packaging guard → regenerate acceptance evidence, on a pinned runner (node 22, ubuntu-latest). The pipeline MUST also include a separate pack-smoke job (Linux electron-builder --dir, asserting the packaged payload — app.asar or resources/app/package.json) and a headless judge job (Playwright/Chromium under xvfb). A dropped gate must be a caught regression.
    • Why Individual gates are pinned (packaging guard + pack-smoke; version:stamp before build; the no-network backstop; the headless judge), but the full ordered step list, that evidence is regenerated in CI, the node-22 / ubuntu-latest pin, and the pack-smoke payload assertion are not. process.html §03 explicitly defers the automatic-vs-manual inventory to #19.
    • Status partial — individual gates pinned, composition not.
    • Citation process.html §02 & §03 residue; COVERAGE.md §19 R01 row; §17 R09 GOLD verify:no-network; acceptance.html §02/§09 (judge under xvfb).
    • Proposed AC GOLD extend build-matrix.test.ts (or a new ci-shape test) to assert ci.yml's verify job contains the required ordered steps and that the judge + pack-smoke jobs exist with their headless/xvfb configuration.
  3. CI evidence regeneration must assert no drift. Any CI step that regenerates checked-in acceptance evidence MUST assert the regenerated output is byte-identical to the committed evidence (e.g. git diff --exit-code) and fail on drift. A regeneration that does not compare is a false-green no-op gate ("Green is meaningful"). This in turn requires the evidence generators to be deterministic (pinned clock and inputs).
    • Why CI regenerates evidence on every run but never asserts it is unchanged, so the regeneration proves nothing — a blind spot the "Green is meaningful" and "Protect the guard" principles squarely forbid. The concrete no-drift requirement is written nowhere; this sits in the acknowledged §03 residue but has a clear principled basis.
    • Status gap — no doc states a CI evidence drift-gate.
    • Citation process.html §02 ("Green is meaningful", "Protect the guard"); §03; acceptance criteria describe evidence as "regenerated by the suites" but pin no no-drift check.
    • Proposed AC GOLD a meta-check asserting npm run evidence is followed by a non-empty drift assertion, plus a deliberately stale-evidence fixture that must fail CI, proving the gate bites.
03 Proposed addition

Version source-of-truth stamping

The single-source-of-truth design (prd.html §19 R06) and the STINT_VERSION runtime override are specified. What is not pinned is the rewrite contract the stamper depends on — the exact source-line shape, that it rewrites only the fallback literal, fails loudly on a missing pattern, preserves the override, and is idempotent.

  1. The version-stamping rewrite contract is pinned. APP_VERSION in packages/core/src/version.ts MUST be process.env.STINT_VERSION ?? <stamped literal>. scripts/stamp-version.mjs MUST rewrite only the fallback literal, MUST throw (fail loudly) if the expected APP_VERSION assignment pattern is absent, MUST preserve the STINT_VERSION env override intact, and MUST be idempotent (a second run is a no-op). STINT_VERSION is a supported runtime override.
    • Why The spec pins behavior (both surfaces read one stamped version, override honored) but not the implementation mechanics: the precise ?? <literal> source shape the stamper keys off, that it rewrites only the literal, throws on a missing pattern, and is idempotent. Those are stamp-version.mjs internals.
    • Status partial — design + override specified, rewrite contract not.
    • Citation prd.html §19 R06; COVERAGE.md §19 R06 row ("APP_VERSION equals the STINT_VERSION override when set…"); runbook ("overridable at runtime via STINT_VERSION").
    • Proposed AC GOLD a test that runs stamp-version.mjs twice and asserts the second run is a no-op (idempotent), and that it throws when version.ts lacks the expected APP_VERSION assignment.
04 Proposed additions

The tt launcher contract

The Node ≥ 22.5 floor (required for node:sqlite) and the bundled-Node happy path are specified, and process.html's "Self-heal or fail loud" principle is normative. What is not written for the launcher is its failure contract: that a system-node fallback must vouch for the floor before exec, and that each failure mode emits a distinct, actionable message with a defined exit code. Exec'ing a Node 20 that crashes deep inside node:sqlite is precisely the "cryptic downstream error" the principle forbids.

  1. The launcher's system-node fallback enforces the version floor. When falling back to a system node, the tt launcher MUST verify that node satisfies the Node ≥ 22.5 floor before exec'ing it, and otherwise refuse with the actionable not-found message. Exec'ing an older node that then fails inside node:sqlite is a "fail loud" violation. The launcher must never exec a runtime it cannot vouch for.
    • Why The floor and the fail-loud principle exist; the specific obligation that the fallback verify the floor (rather than exec blindly) is derivable from a normative principle but not written for the launcher. A system-node fallback is itself outside the spec'd bundled-Node happy path.
    • Status partial — floor + principle specified, fallback obligation not.
    • Citation prd.html §15 (bundled Node ≥ 22.5 for node:sqlite); runbook "CHECK INSTALL"; process.html §02 "Self-heal or fail loud".
    • Proposed AC MANUAL runbook step under CHECK INSTALL: with only a Node < 22.5 on PATH and the bundled runtime removed, run tt and assert it refuses with the actionable "≥ 22.5" message rather than crashing inside node:sqlite. Optionally a GOLD shell-lint asserting the fallback contains a version gate.
  2. The launcher's two failure modes emit distinct actionable messages. The tt launcher's two failure modes — CLI entrypoint not found in any bundle, and no eligible runtime found — MUST each exit non-zero with a specific, actionable stderr message naming the missing thing and its remedy (reinstall / install Node ≥ 22.5). The spec MUST state whether the two modes use distinct exit codes for diagnosability or may share 127.
    • Why The success contract is pinned, but neither failure mode's exit code (currently 127 for both) nor its stderr wording is specified anywhere in context/ or criteria/. Sharing 127 across both modes — and using it for "runtime found but bin missing" — is a deliberate-but-undocumented decision; a tester cannot assert the failure contract from the spec.
    • Status gap — failure exit codes & messages unspecified.
    • Citation none (no context/ or criteria/ doc names exit 127 or either message; runbook "CHECK INSTALL" asserts only the success path).
    • Proposed AC GOLD shell-lint or MANUAL step asserting each failure path emits its distinct actionable message and the agreed exit code(s), so a reworded/swapped message or changed code is a caught regression.
