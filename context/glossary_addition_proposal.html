This page proposes additions to glossary.html surfaced by an adversarial implementation-vs-spec audit (issue #27). Each item is an implicit requirement found in the implementation that the canonical spec does not (or does not fully) pin down.

It is a sibling page, not a replacement: nothing here amends the glossary until reviewed and merged in. Definitions below are written in the glossary's voice so they can drop straight into their target sections once adopted. Each carries an audit status — gap (unspecified) or partial (intent specified, exact contract not pinned) — plus the concrete code/test evidence and any existing citation.

Four proposed entries: empty-dimension group labels and the tag fan-out rule, the canonical stored values for settings defaults, and the tag-identity normalization contract that both surfaces must share.

P1 Grouping & group keys

The labels reports group under — and how tags fan out

Targets §03 Time & duration / the reporting vocabulary. Grouping dimensions are specified in prd.html §09 R2 (client → project → entries; also by day or by tag) and clientless time is a documented concept (§03), but the literal placeholder labels and the tag fan-out counting rule are fixed only in code. Both are user-visible (the strings) and affect totals (the fan-out).

  • G R01 Empty-dimension group labels New Gap · unspecified

    When a report groups by a dimension an entry lacks, the entry falls under a canonical placeholder group whose label is exactly (no client), (no project), or (untagged) — parenthesized, lower-case, the literal strings above. Both surfaces render the same label.

    WhyReports bucket clientless / projectless / untagged entries under fixed strings, but the exact labels live only in the implementation; they are user-facing and must match across the CLI and GUI.

    StatusGap — unspecified. The dimensions and clientless time are documented; the placeholder strings are nowhere in the corpus.

    Citationprd.html §09 R2 (grouping: default client → project → entries; also by day or by tag); §03 (clientless time).

    ACGOLD: placeholder strings match exactly for a clientless, projectless, and untagged entry under each group-by.

  • G R02 Tag fan-out & grouping-invariant grand total New Gap · unspecified

    When grouping by tag, an entry appears once under each of its tags; an entry with no tags appears once under (untagged). This fan-out means the same entry is counted in multiple tag groups, so per-tag-group subtotals may sum to more than the worked time — yet the report's grand total stays grouping-invariant (identical across every group-by, fan-out notwithstanding).

    WhyTag grouping fans a multi-tag entry into every matching group, while the grand total is computed over distinct entries. The counting rule is user-visible (subtotals vs. grand total) and is currently code-only.

    StatusGap — unspecified. The tag dimension is specified; the fan-out counting rule and the invariance of the grand total are not.

    Citationprd.html §09 R2 (grouping by tag).

    ACGOLD: a multi-tag entry contributes to each of its tag groups; an untagged entry lands once in (untagged); the grand total is unchanged across group-by.

Proposed glossary insertionAdd a Group key entry to §03 (or a short "Grouping" subsection): the labels above are canonical, and the tag fan-out rule is part of the term's definition so neither surface invents its own strings or double-counts the grand total.
P2 Tag identity & normalization

When two spellings are the same tag

Targets the §01 Tag entry (and Entry ↔ Tag in §07). The glossary describes tags as flat, reusable, created-on-the-fly labels and the JUDGE TAG_CHIPS rubric pins that tag edits send minimal addTags/removeTags — but the deeper identity normalization contract (the rule both surfaces must apply through Core) is nowhere defined. Two proposed clauses, one defining identity and one defining the diff that follows from it.

  • T R01 Tag identity is case-insensitive and trimmed New Gap · unspecified

    A tag's identity is its name matched case-insensitively and whitespace-trimmed. The first-seen spelling is preserved; empty/whitespace-only names are dropped. Applying an existing tag to an entry in a different case or with surrounding whitespace is the same tag — a no-op (no recase, no duplicate). Both surfaces normalize identically because they go through Core.

    WhyTag matching folds case and trims whitespace, dedupes keeping the first spelling, and treats a recase as a no-op; re-typing meeting over Meeting must not fork the tag or rename it. This identity rule is the data-model contract the GUI diff mirrors and is undefined in the spec.

    StatusGap — unspecified. The glossary says tags are flat/reusable/created-on-the-fly but states no case/trim/dedupe identity rule.

    CitationNone (genuine gap). Related: glossary.html §01 Tag; §07 Entry ↔ Tag many-to-many.

    ACPROP/GOLD: applying Meeting then meeting  yields one tag keeping the original spelling, across Core and both surfaces.

  • T R02 Minimal tag diff follows from identity New Partial · pin the rule

    Editing an entry's tags diffs original against edited under the T R01 identity: empties dropped, duplicates collapsed to one (first-seen spelling), comparison case-insensitive and trimmed, a tag present on both sides (in any case) left untouched — producing the minimal addTags/removeTags the edit IPC carries.

    WhyThat the edit sends minimal add/remove is already specified; what is unpinned is the normalization the diff rests on (case-fold, trim, drop-empty, dedupe-first-spelling, both-sides-untouched-regardless-of-case). Both surfaces must apply the same normalization through Core.

    StatusPartial — the minimal-diff outcome is specified; the normalization contract underneath it is not.

    Citationprd.html §07 R02 (tags flat, reusable, created on the fly); glossary.html §07 Entry ↔ Tag; acceptance/criteria/judge-rubric.md TAG_CHIPS ("minimal addTags/removeTags").

    ACPROP/GOLD: the tag-diff over an original and an edited set yields minimal add/remove and is case-insensitive, trimmed, and dedupe-first-spelling.

Proposed glossary insertionExtend the §01 Tag definition with the identity sentence (T R01) and add a short cross-reference so the §07 Entry ↔ Tag row and any edit-flow copy point at the same normalization rule.
P3 Settings defaults — stored values

The persisted form of each default, not just its intent

Targets §14 Settings (or a settings glossary). The intent of every default is already pinned in the §14 table and §12 R02. What is partial is only the encoding of the stored value — the exact enum/literal the snapshot persists — so the on-disk representation is unambiguous and the DEFAULT_SETTINGS snapshot has something canonical to assert against.

  • S R01 Canonical stored default values New Partial · pin stored form

    Each setting's persisted default value is fixed alongside its human-facing default:

    SettingStored defaultHuman-facing default (§14)
    hotkeyCommandOrControl+Alt+TGlobal hotkey Ctrl+Alt+T (Electron accelerator form)
    accentsystemSystem accent, primary action only
    date_formatsystemSystem locale
    week_startmonday (lower-case token)Week starts Monday
    roundingfalse / 0Rounding off

    WhyThe defaults snapshot persists exact literals (e.g. week_start='monday', not 'Monday'; accent='system'; rounding=false). The spec states the human-facing default but not the stored token, leaving the persisted representation underspecified.

    StatusPartial — substantive defaults are specified; only the stored enum/literal encoding is unpinned.

    Citationprd.html §14 Settings table (rounding off; week start Monday; global hotkey Ctrl+Alt+T; accent "system accent, primary action only"; date/number "system locale"); §12 R02 (registered as CommandOrControl+Alt+T); §15.

    ACGOLD: the DEFAULT_SETTINGS snapshot asserts each stored default value matches the §14 table.

Proposed glossary insertionThis is fundamentally a §14 (PRD) pin; it is recorded here so the glossary's Setting entry (§01) can reference the canonical stored tokens. The adopting edit may live in the PRD §14 table with the glossary Setting entry cross-linking it.
